# Memory benchmark methodology

This is the auditable protocol behind Lore's memory benchmark (issue #961) and the
"The compaction tax" blog post. Everything here is reproducible from this directory;
the scripts, task fixtures, and scorer are all versioned alongside it. Raw per-run
results and the aggregated scorecards live in `RESULTS.md`.

The goal is to measure two things a coding agent's memory layer is supposed to do,
and to do it without grading on our own curve:

1. **Within a single long session**, keep the facts established early from being lost
   when the context window fills and the client compacts.
2. **Across sessions**, carry facts stated in one session into a later, separate one.

## Design principles

- **Real agents, not replay.** Every run drives a real OpenCode agent end to end
  (`opencode run`), turn by turn, against a real model over the network. Nothing is
  mocked or replayed. The arms differ only in the memory layer attached.
- **Code-invisible facts.** We probe on four arbitrary values that are stated once,
  in passing, and cannot be re-derived by reading the codebase: an order `status`
  (`SUBMITTED`), a `channel` (`WHOLESALE`), a `region` (`EMEA`), and a `warehouse`
  (`WH-07`). A model with no memory of the original statement has no way to guess
  them; if they appear in the final file, the memory layer carried them.
- **Signal-only scoring, no LLM judge.** The scorer (`score.mjs`) is a mechanical
  check for those four values in the produced code. Structural "freebies" that a
  competent model would write anyway (a frozen dataclass, a custom exception, integer
  cents) are detected but **gated OFF** so they cannot inflate a no-memory arm. There
  is no model in the loop judging correctness, so there is no self-grading confound.
- **Generous to the alternatives.** The pull-memory competitors get an `AGENTS.md`
  telling the agent it has a persistent memory tool and should use it. Lore gets no
  such note, because it needs none. The default note is deliberately casual (what a
  real user writes), with an over-coached "mandatory workflow" variant available via
  `--mcp-agents strong` for the competitor's best case.

## Scenarios

Both scenarios share one project seed (`seed-min`, a small stdlib-only Python
"orderkit") and pipe a large reference blob into most turns to drive the context
window up. Blobs are generated by `gen-blob.mjs` and git-ignored.

- **single-long** (`task-pref-long.json`): one session, 16 turns, a ~40K-token blob
  per turn. The four facts are buried as casual asides in turn 1. Later turns keep
  building small pure functions until native compaction has fired several times, then
  the final turn asks for the file that needs all four values. This measures erosion
  across repeated within-session compaction.
- **cross-session** (`task-pref-combined.json`): two separate sessions of 5 turns
  each, a ~64K-token blob per turn (forces two native compactions per no-memory
  session). The facts are stated in session 1; the probe is the final turn of
  session 2. Session 2 starts with no conversation carryover, so only a memory layer
  can bridge the gap.

## Arms

Each arm is one value of `--arm`:

- `lore`: an isolated Lore gateway in the request path (automatic distillation +
  injection). No memory tool, no `AGENTS.md`.
- `nolore`: the bare agent. Memory is whatever OpenCode's native compaction preserves.
- `mcp-mnemonic`: mnemonic (local SQLite + FTS5 + vector, the closest local-DB
  analogue to Lore), driven by the agent over MCP. Needs a `GEMINI_API_KEY`.
- `mcp-mem0`: mem0 cloud, driven over MCP. Needs a `MEM0_API_KEY`.
- Also wired: `mcp-kg` (official knowledge-graph server) and `mcp-basicmem`.

## Models and the worker

- Answering models: **MiniMax-M3** (a cheaper, everyday model, capable but not a frontier
  model) and **Claude Sonnet-5** (a frontier flagship). The cost/capability split is the
  whole point: it is where the memory approaches diverge.
- **Worker = the same model as the answering model.** Lore's background distillation
  runs on the same model the session runs on (Sonnet distills with Sonnet, M3 with
  M3). We do not use a smaller worker; distillation quality drops when we do. This
  keeps Lore's cost fully on the model you already run, with no third-party service.
- The worker API key is derived from the worker model's provider (`workerKeyFor`).
  Hardcoding one provider's key silently auth-fails the worker for every other
  provider, which zeroes out distillation and makes Lore fall back to temporal-only
  recall. This bug confounded an early Sonnet run (see "Known fixes").

## Fairness controls

- **Isolation.** Every arm runs in its own workspace: a private `XDG_*` tree, a fresh
  git-seeded project copy, and (for Lore) its own gateway process on a free port with
  its own SQLite DB. The eval never touches a real user gateway or DB.
- **Context cap.** Some models advertise a 1M window that the endpoint will not
  actually serve, so OpenCode (trusting 1M) never compacts and errors instead.
  `--cap-context` sets a realistic limit so compaction fires cleanly, and it is
  applied **identically to both arms** (OpenCode's compaction runs under the Lore
  plugin too), so no arm is advantaged by the cap.

## Scoring and metrics

- **Retention**: the signal probe score, out of 4, from `score.mjs`. This is the only
  correctness number.
- **Efficiency**, parsed from OpenCode's `--format json` stream and its session DB:
  steps, `tokensIn/Out`, `cacheRead`, `cacheWrite`, total tokens, tool calls, peak
  context, wall time, and native `compactions`.
- **Cost**: the answering-model cost OpenCode reports, plus, for Lore, the gateway's
  own `daily_costs` buckets (`conversation` and `worker`) so the background
  distillation spend is counted, not hidden.

## Known fixes and gotchas (audit trail)

- **Worker key by provider.** Fixed a hardcoded minimax worker key that auth-failed
  the Sonnet worker (0 distillations, Lore dipped to 8/12 on temporal recall alone).
  With the fix, Sonnet distillation works and Lore is a clean 12/12.
- **Embedding model.** A truncated local ONNX model download (`protobuf parsing
  failed`) forced FTS-only recall. Fixed by refreshing the native ONNX backend and
  purging the corrupt model so it re-downloaded clean; runs since have `vec=on`.
- **DeepSeek/Zen not evaluable for Lore.** The anonymous Zen endpoint has no keyed
  worker path, so Lore cannot distill there. A keyed provider is required to include
  it. M3 answering with an M3 worker is the clean cheaper-model config.

## Independent check: contextrot

This benchmark was inspired by [contextrot](https://github.com/Priyanshu-byte-coder/contextrot),
an observational diagnostic that reads an agent's own local transcripts and correlates
five behavioral failure signals (edit failures, retry loops, re-reads, self-corrections,
tool errors) with context fill. It measures a different thing than we do: behavioral
degradation on real workloads, not fact retention on controlled tasks.

We ran contextrot over our own eval transcripts (per-arm, isolated `HOME`+`XDG_DATA_HOME`
so it only sees the eval session). It did not produce a publishable rot curve, for two
principled reasons: our runs are 30-67 steps each (contextrot wants ~150 in both fresh and
deep context, so every verdict is "insufficient"), and our small function-writing tasks do
not induce the edit/retry/re-read behaviors it detects. One honest observation did fall
out: even at full context the no-memory arm threw no behavioral trouble at all, yet our
probe shows it had already lost most of the facts. Compaction's damage is silent, not a
visible failure. We cite contextrot as motivating/related work rather than folding its
metrics into our scorecard.

### Behavioral rot A/B (in-repo, real signals)

Rather than depend on contextrot's runtime, we reimplement its five signals in-repo
(`packages/core/src/degradation-signals.ts`) and run them over the driver's own per-turn
`sessions/*.json` streams (`trace-extract.mjs` → `TraceStep[]`), then build a rot curve per
arm and contrast lore vs each comparator (`rot-behavioral.mjs`). Two deliberate choices:

- **Absolute prompt tokens, not fill %.** Dividing by the context window confounds the A/B:
  with Lore on, per-step fill stays low, so Lore would "win" trivially by never reaching high
  fill. On the absolute-token axis the claim is falsifiable — *at the same real context size,
  does the arm degrade less?* Fresh/deep zones are derived from the pooled token spread so
  both arms are compared over the same populated range.
- **Wilson-gated significance.** Lore is reported as "significantly better deep in context"
  only when its deep-zone degraded rate is strictly lower AND the two arms' Wilson 95%
  intervals are disjoint — never on a bare numeric edge.

`(tool, target)` identity reuses the gateway's own `extractFilePath`/`extractCommand`
(`tool-trace.ts`, #1424) so retry/reread detection matches production exactly. Run it over a
completed matrix directory:

```
bun rot-behavioral.mjs --root /home/byk/.local/share/lore-eval-live/MATRIX-V4 [--json]
```

## How to run

The packaged command (reproduces the core Lore-on vs Lore-off comparison):

```
lore eval --model anthropic/claude-sonnet-4-6 --scenario both
```

The raw harness (all arms, full control), from this directory:

```
bun driver.mjs --task task-pref-combined.json --arm lore   --model <prov/model> --out /tmp/run-lore   --cap-context 200000
bun driver.mjs --task task-pref-combined.json --arm nolore --model <prov/model> --out /tmp/run-nolore --cap-context 200000
bun score.mjs /tmp/run-lore
```

Competitor arms additionally need `GEMINI_API_KEY` (mnemonic) or `MEM0_API_KEY`
(mem0) in the environment. `run-queue.sh <concurrency> <jobfile>` runs a batch with
per-provider concurrency limits.
