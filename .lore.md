* **Layer-1 raw message window construction via backward token scan**: Layer-1 raw window built by \`tryFitStable()\` (lines 1159-1232) using backward token scan. Splits budget: 25% distilled, 40% raw. Current turn always fully included (\`currentTurnStart()\` to end), then backward scan from most recent older messages until \`remainingBudget = rawBudget - currentTurnTokens\` exhausted. Cutoff drops oldest messages first. Three mechanisms cause leading-edge ID shifts: 1) \`currentTurnStart()\` boundary moves as tool-chains extend, 2) Pinned window cache overflow triggers fresh scan when appended messages exceed budget, 3) \`deduplicateToolOutputs()\` changes token estimates of older messages between transforms.

### Gotcha

<!-- lore:019e041a-a556-73ae-ac32-3ef302bb04c7 -->
* **Bust rate spikes from gradient cache thrashing in concurrent sessions**: High bust rates occur when multiple sessions simultaneously transform the same knowledge rows, causing gradient cache thrashing. Each session's \`transform()\` call invalidates cached gradients for overlapping knowledge, forcing expensive re-computation. Symptoms: burst of \`bust\` logs in journalctl with multiple active sessions hitting same knowledge base. Sessions show identical knowledge row counts but different gradient states. Monitor with \`journalctl -u opencode | grep bust\` and check concurrent session overlap. Consider gradient cache partitioning or session-aware batching to reduce thrashing during high concurrency periods.

<!-- lore:019e03e3-a2ec-72f5-9964-5f5e906ce46e -->
* **Cache TTL must use string literals '5m' or '1h', not seconds**: Anthropic's prompt caching API rejects numeric TTL values like \`"3600"\` (seconds), requiring ISO 8601 duration strings: \`"5m"\` (5 minutes) or \`"1h"\` (1 hour). This affects \`llm-adapter.ts\`, \`batch-queue.ts\`, and \`translate/anthropic.ts\`. Error: \`invalid\_request\_error: system.0.cache\_control.ephemeral.ttl: Input should be '5m' or '1h'\`. Use \`"1h"\` for worker system prompts (static, durable across bursts) and \`"5m"\` for conversation caching (expires between user turns).

<!-- lore:019e03e6-7c31-77a7-b839-09b8288417bc -->
* **Gateway needs restart after code changes during bun run from source**: Gateway needs restart after code changes when running from source: When running gateway via \`bun run /path/to/src/index.ts\`, code changes require process restart. Bun caches modules in memory; \`git pull\` updates files on disk but doesn't reload imported modules. Symptom: merged code on disk but runtime uses old code. Fix: restart systemd service after pulling changes. This particularly affects recall interception logic testing.

<!-- lore:019e041a-302c-708f-b026-333202aa4275 -->
* **Gradient prefix fingerprint changes cause universal transform busts**: All sessions bust on every transform when gradient prefix fingerprint changes between requests. Bust detection compares stored \`prefixFingerprint\` in session state against current conversation prefix hash. When prefix changes (new system messages, conversation context shifts), fingerprint mismatches trigger full gradient recomputation instead of incremental updates. Symptom: every active session shows bust logs on each transform. Check if system prompts, conversation preprocessing, or gradient context generation changed recently. High bust rates indicate upstream changes affecting conversation prefix consistency across requests.

<!-- lore:019e041a-a55c-7a31-bd2d-16e043633141 -->
* **High bust rates from gradient transform failures**: Bust detection in gradient system flags sessions when transform operations fail. Common causes: malformed gradients from upstream APIs, transform logic errors, or missing gradient data. Check \`journalctl -u opencode\` for transform failures and gradient processing errors. Sessions with busts show incomplete or corrupted gradient chains that can't be processed by the transform pipeline. Monitor session state via database queries to identify patterns in bust timing and frequency. Transform failures cascade to affect session quality scoring.

* **Mock type casting requires double cast through unknown**: TypeScript strict mode rejects \`mock() as typeof fetch\` because \`Mock\` interface lacks properties like \`preconnect\`. Use double cast: \`mock() as unknown as typeof fetch\` to bypass type checking. This pattern appears in gateway tests where vitest mocks need to satisfy complex built-in interfaces. Single cast fails with 'Property X is missing in type Mock' errors.

* **Model doesn't call recall when tool injection fails silently**: When gateway recall tool injection fails or is skipped, model receives no recall tool in toolset and won't attempt recall calls — no interception logs appear in journalctl. Model may literally type '\[Searching memory...]' text instead of using recall tool\_use. Debug by checking if recall tool appears in upstream request tools list. Silent injection failures (e.g., duplicate tool names, malformed tool definitions) cause model to operate without recall capability while appearing functional. Symptom: no \`recall (stream, only)\` or \`recall (stream, mixed)\` logs despite memory-relevant queries.

### Pattern

<!-- lore:019e041a-3027-7f62-b103-35db886d35bd -->
* **Gradient cache-bust detection via prefix fingerprints**: Gradient system detects cache-busts via \`prefixHash\` fingerprints combining transform version + system prompt hash. Format: \`version:hash\` (e.g., \`1:878af1f6\`). Generated in \`gradient.ts\` via \`transformVersion\` constant and \`hashSystemPrompt()\` of adapter's system prompt. Cache-bust triggers when current prefix differs from stored \`lastPrefixHash\` in \`kv\_meta\`. OpenCode adapter tracks system prompt changes in \`SYSTEM\_PROMPT\_HASH\_TRACKING\` section. Bust rate calculated as \`cacheBusts / transformCount \* 100\`. HIGH BUST RATE warning at >10% indicates frequent system prompt changes requiring investigation. Counter persists in \`kv\_meta\` as \`bust\_rate\_counter\`.

<!-- lore:019e0381-826b-7cf5-9586-f13d2c767f41 -->
* **Pending recall cross-request injection via SessionState**: Case 2 (recall + other tools) stores \`{ toolUseId, position, result, timestamp }\` in \`sessionState.pendingRecall\` with 60s TTL. On next request, \`injectPendingRecall()\` finds the last assistant message, inserts the recall tool\_use block at the recorded position, injects tool\_result into the following user message, and removes recall from tools list. Client never sees recall — model receives results as natural conversation history. Lazy cleanup on session access checks TTL.

<!-- lore:019e0428-694b-7aab-a42a-1ce98c4c3b1f -->
* **RawWindowCache requires pinnedBudget snapshot for stability**: \`RawWindowCache\` stores \`pinnedTokens\` and \`pinnedBudget\` (budget snapshot at pin creation). Pin validity check uses 10% hysteresis: \`pinnedTokens <= pinnedBudget \* 1.10\` instead of exact match against current budget. Decouples pin validity from volatile budget fluctuations caused by dedup token estimate changes and overhead drift. Absorbs ~8-17K token noise on ~47K budget without false evictions.

<!-- lore:019e0428-6964-7feb-b42b-d94e1cfb6aa6 -->
* **SessionState per-session isolation pattern for globals**: Move problematic module-level globals to \`SessionState\` for per-session isolation. Pattern: add optional \`sessionID\` parameter to setter functions. Store on session state if ID provided, else use module-level fallback for backward compatibility. Read from session state first in transforms. Example: \`setLtmTokens(tokens, sessionID?)\` stores in \`sessState.ltmTokens\` if sessionID provided. \`transformInner\` reads \`sessState.ltmTokens\` instead of global, eliminating cross-session contamination where one session's LTM pollutes another's budget calculations.

<!-- lore:019e0427-0281-75c6-bd89-e27b4e64a72f -->
* **Three-part fix for tryFitStable cache instability**: Fix tryFitStable cache busts with complementary changes: 1) Add \`pinnedBudget: number\` to \`RawWindowCache\` to snapshot budget at pin creation. 2) Use 5% hysteresis buffer in pin validation: \`totalTokens <= pinnedBudget \* 1.05\` to absorb token estimation noise. 3) Move \`ltmTokens\` from module global to \`SessionState.ltmTokens\` and update \`setLtmTokens(sessionID, tokens)\` signature. All three address different instability sources: budget snapshots prevent pin-check budget mismatches, hysteresis absorbs dedup estimation variance, per-session state eliminates cross-session contamination.

<!-- lore:019e0428-69dc-733b-8292-0775d1a1f6a4 -->
* **Three-part fix pattern for tryFitStable cache stability**: Comprehensive fix for tryFitStable cache busts uses three complementary approaches: 1) Budget snapshot - store \`pinnedBudget\` in \`RawWindowCache\` during cache write, use it (not current budget) in pin validity check, 2) Hysteresis buffer - use \`rawBudget \* 0.95\` threshold in pin check to absorb estimation noise from dedup changes, 3) Per-session isolation - move \`ltmTokens\` to \`SessionState\` to eliminate cross-session contamination. All three address different instability sources: budget drift, estimation variance, and shared globals. Pattern prevents 97%+ cache bust rates in tool-heavy sessions.

<!-- lore:019e0426-8c8a-7ca7-bba2-cfe65d79225e -->
* **Three-pronged fix for tryFitStable cache instability**: Three-pronged fix for tryFitStable cache instability: 1) Snapshot budget in \`RawWindowCache.pinnedBudget\` at pin creation time, 2) Hysteresis with 5% tolerance in pin overflow check (\`pinnedTokens > pinnedBudget \* 1.05\`) to absorb dedup estimation noise, 3) Move module-level \`calibratedOverhead\` and \`ltmTokens\` to per-session state to prevent cross-session contamination. The \`setLtmTokens(sessionID, tokens)\` API requires sessionID parameter - available as \`input.sessionID\` in adapters. All three address different aspects: budget drift, estimation variance, and session isolation.

Transport returns pre-recorded responses in sequence, enabling deterministic testing of streaming and non-streaming flows. Essential for testing recall interception Cases 1 and 2 where timing and response structure matter.
