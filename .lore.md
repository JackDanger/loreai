* **Embedding provider migration strategy with checkConfigChange**: Embedding config fingerprint enables zero-downtime provider migration: Config changes detected via fingerprint (\`provider:model:dimensions\`) in \`kv\_meta\`. On startup, \`checkConfigChange()\` compares current to stored fingerprint; on mismatch, stale embeddings auto-clear and full backfill regenerates. Enables seamless Voyage (1024d) ↔ Local (384d) ↔ OpenAI (1536d) migrations. Both OpenCode and Pi adapters call unified \`runStartupBackfill()\` at init. Coverage stats logged to stderr.

<!-- lore:019e03da-045c-7ce9-8fae-0f01de363b70 -->
* **Gateway as universal HTTP integration point**: Gateway as universal HTTP integration point: Gateway accepts Anthropic \`/v1/messages\` protocol, applies full Lore pipeline (gradient + LTM + distillation + curation), forwards to upstream. Works for any HTTP client by setting base URL. Enables recall tool interception universally without per-host plugins. Session tracking via base62 markers injected into assistant responses. Compaction requests intercepted before upstream forwarding to trigger distillation. Gateway also handles worker model validation and passes resolved models to \`distillation.run()\` and \`curator.run()\` via \`/v1/models\` upstream discovery, replacing per-adapter validation logic.

<!-- lore:019e0381-8263-71ec-b845-06b715363ce2 -->
* **Gateway recall interception — two-case stream-and-intercept design**: Gateway transparently intercepts recall tool\_use blocks mid-response. Case 1 (recall-only): suppress recall blocks from client stream, execute via follow-up request, inject '\[Searching memory...]' text at suppressed index, continue with re-indexed blocks in same HTTP response. Case 2 (recall + other tools): suppress recall, execute in parallel, store result with 60s TTL, inject into next request's assistant message. Logs show \`recall (stream, mixed): stored pending result\` for Case 2, then \`injected pending recall result\` on next request. Full implementation with 161 passing tests including end-to-end integration coverage.

<!-- lore:019e040a-c7db-7869-ab95-e2e5273e9511 -->
* **Gateway worker model discovery via /v1/models API**: Gateway discovers worker models via Anthropic's \`/v1/models\` API, then fetches real-time cost data from models.dev TOML files (\`https://raw.githubusercontent.com/sst/models.dev/dev/providers/anthropic/models/{model-id}.toml\`). Parses \`\[cost].input\` field from each TOML with simple regex (no full TOML parser needed). Falls back to hardcoded costs (Opus: $15/M, Sonnet: $3/M, Haiku 3.5: $0.80/M) only on GitHub fetch failures. Results cached 1-hour in \`kv\_meta\`. Automatically excludes deprecated models and feeds resolved candidates to \`selectWorkerCandidates\`/\`runValidation\` pipeline.

### Gotcha

<!-- lore:019e03e5-8e13-7441-9588-85efd3585be9 -->
* **.lore.md unstaged changes must be committed with feature branches**: Include .lore.md unstaged changes in commits: \`.lore.md\` is auto-updated by lore's knowledge management and contains meaningful project state. Before creating branches, PRs, or commits, always check for and stage unstaged changes to \`.lore.md\` to maintain knowledge consistency and avoid incomplete commits. Standard workflow: commit all changes including \`.lore.md\`, then create PR with auto-merge.

<!-- lore:019e03f4-aac0-76db-95fa-00d68a431b99 -->
* **Branch-to-main type drift breaks tests after merge**: Branch-to-main type drift breaks tests after merge: Long-lived feature branches cause type mismatches after merge when main evolves. Common: new required fields (\`protocol\` in GatewayRequest, \`input\` in PendingRecall), changed function signatures (argument count). Tests pass on branch but fail on main. Fix: rebase frequently or update tests to match current main types before/after merge. Always check CI post-merge.

<!-- lore:019e03e3-a2ec-72f5-9964-5f5e906ce46e -->
* **Cache TTL must use string literals '5m' or '1h', not seconds**: Anthropic's prompt caching API rejects numeric TTL values like \`"3600"\` (seconds), requiring ISO 8601 duration strings: \`"5m"\` (5 minutes) or \`"1h"\` (1 hour). This affects \`llm-adapter.ts\`, \`batch-queue.ts\`, and \`translate/anthropic.ts\`. Error: \`invalid\_request\_error: system.0.cache\_control.ephemeral.ttl: Input should be '5m' or '1h'\`. Use \`"1h"\` for worker system prompts (static, durable across bursts) and \`"5m"\` for conversation caching (expires between user turns).

<!-- lore:019e03e6-7c31-77a7-b839-09b8288417bc -->
* **Gateway needs restart after code changes during bun run from source**: Gateway needs restart after code changes when running from source: When running gateway via \`bun run /path/to/src/index.ts\`, code changes require process restart. Bun caches modules in memory; \`git pull\` updates files on disk but doesn't reload imported modules. Symptom: merged code on disk but runtime uses old code. Fix: restart systemd service after pulling changes. This particularly affects recall interception logic testing.

<!-- lore:019e03e6-7c40-7236-aca4-2e313520cf34 -->
* **Model doesn't call recall when tool injection fails silently**: When gateway recall tool injection fails or is skipped, model receives no recall tool in toolset and won't attempt recall calls — no interception logs appear in journalctl. Model may literally type '\[Searching memory...]' text instead of using recall tool\_use. Debug by checking if recall tool appears in upstream request tools list. Silent injection failures (e.g., duplicate tool names, malformed tool definitions) cause model to operate without recall capability while appearing functional. Symptom: no \`recall (stream, only)\` or \`recall (stream, mixed)\` logs despite memory-relevant queries.

<!-- lore:019e03f6-0ac6-7c88-a8d2-67db3c837060 -->
* **TypeScript interface requirements for gateway test setup**: Gateway test interface requirements: Gateway recall tests require specific interfaces: \`findRecallToolUse()\` takes 1 argument (response only), \`PendingRecall\` requires \`input\` field, \`GatewayRequest\` requires \`protocol\` field. Common failure: passing 2 args to \`findRecallToolUse\` or omitting required fields in mock structures. Check type definitions in \`test/gateway/utils.ts\` before creating fixtures.

### Pattern

<!-- lore:019e0381-826b-7cf5-9586-f13d2c767f41 -->
* **Pending recall cross-request injection via SessionState**: Case 2 (recall + other tools) stores \`{ toolUseId, position, result, timestamp }\` in \`sessionState.pendingRecall\` with 60s TTL. On next request, \`injectPendingRecall()\` finds the last assistant message, inserts the recall tool\_use block at the recorded position, injects tool\_result into the following user message, and removes recall from tools list. Client never sees recall — model receives results as natural conversation history. Lazy cleanup on session access checks TTL.

<!-- lore:019e0381-8268-77ac-9f9f-0362b425dcf8 -->
* **RecallAwareStreamAccumulator — selective block suppression with re-indexing**: RecallAwareStreamAccumulator — selective block suppression with re-indexing: Wraps \`StreamAccumulator\` to intercept SSE events and suppress recall tool\_use blocks from client stream. Detects \`content\_block\_start\` with name='recall', marks index as suppressed. Holds back \`message\_delta\`/\`message\_stop\` until caller decides Case 1 (continue) or Case 2 (forward). For continuation in Case 1, re-indexes content blocks: \`clientIndex = upstreamIndex - suppressedCount\`, starting at \`blockOffset\`. Zero overhead on non-recall streams.

<!-- lore:019e03ec-e1f1-7bab-8634-0a0a081f3f99 -->
* **Replay system for gateway integration testing**: Gateway tests use \`ReplayTransport\` to mock upstream API responses without real HTTP calls. Load golden response files with \`replay.loadResponse('path/to/response.json')\`, then create transport via \`new ReplayTransport(responses)\`. Transport returns pre-recorded responses in sequence, enabling deterministic testing of streaming and non-streaming flows. Essential for testing recall interception Cases 1 and 2 where timing and response structure matter. Responses include full SSE event sequences with proper \`content\_block\_start\`/\`content\_block\_delta\`/\`content\_block\_stop\` structure.
