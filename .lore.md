* **bspatch.ts: TRDIFF10 format, OldReader abstraction, SWAR diff-add, in-memory patch chaining**: TRDIFF10 header (32B): magic\[0:8], controlLen/diffLen/newSize each i64 LE via offtin's sign-magnitude decoding, followed by zstd(control)|zstd(diff)|zstd(extra). OldReader abstracts fd-backed (pread + 1MiB read-ahead cache, backed by a reflinked temp copy) vs in-memory (MemoryOldReader) sources so transformPatch/applyPatch read old bytes uniformly on-disk or mid-chain. Multi-patch chains (applyPatchChainInMemory) keep intermediates entirely in JS heap; only the final binary is persisted + SHA-256 hashed, avoiding N-1 redundant disk writes/hash passes per hop. Peak heap per hop ≈2x one binary's size, not cumulative, since each MemoryOldReader is closed/GC-eligible right after its hop. PR #1436 rewrote this for the 310MB gateway self-upgrade binary: SWAR carry-less Uint32Array add for diff-add (883ms→221ms, 4x); buffered 1MiB-highWaterMark writes; timing logs distinguish delta vs full-download path. Final binary is SHA-256-verified against expected hash before use; any failure (mismatch, partial write, RangeError) falls back to a full download.
