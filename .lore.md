Use imperative language ('never', 'always', 'must') and explain the why (error codes, regression tickets, failure modes). Use 🔴 markers with ALWAYS/NEVER language for hard invariants. Examples: 'Always record auth failure so worker-health ladder sees it even for session-less paths', 'opts.thinking is intentionally not forwarded — bare API calls never include thinking parameter'. When a behavioral directive is given, locate the enforcing call site and embed the rule, reason, and failure mode it prevents.

* **Always investigate why tests failed to catch issues, not just fix the issues themselves**: When a bug or production incident is resolved, the user consistently demands a follow-up investigation into why the existing test suite failed to catch the problem. Root cause: cache-stability suite asserted ONLY on cache \*inputs\* (byte-stability of system blocks between turns) and NEVER on cache \*output\*; replay fixtures reported ZERO cache usage so gateway oracles (categorizeBust/recordCacheUsage/calibrate) ran blindfolded. Fix: \`simulated-cache.ts\` computes each turn's cache\_read/cache\_creation from ACTUAL upstream-body prefix stability, feeding production-faithful numbers back to oracles. Expectations: (1) root cause analysis of test gap (structural/design gaps), (2) new regression tests that would have caught the issue, (3) hardening to prevent future misses. Simply fixing the bug is insufficient. Post-mortems must cite exact file:line references and verbatim code/assertions.

* **Always perform a discrimination check (evidence gate removal) to verify new tests can fail**: Discrimination check pattern (mandatory for new tests): (1) insert tests, (2) confirm they pass, (3) remove/disable the implementation logic being tested, (4) confirm at least one test fails with the expected message, (5) restore the logic, (6) confirm all pass. Applies especially to guard conditions and gating logic. Non-discriminating tests — where the fix lives in a different layer than the test exercises — must be called out as false confidence (M1 pattern).

* **Always plan before coding: produce a detailed file-and-line-level implementation plan before making any edits**: Before writing any code, the user expects a thorough read-only investigation phase that produces an explicit, ordered task list with exact file paths, line numbers, before/after code snippets, and invariant comments. * **Always run adversarial review before merging and fix all non-NIT findings**: Always run adversarial code review before merging any PR. Review must: focus on real bugs (logic errors, race conditions, incorrect assumptions, missing guards); categorize findings as BLOCKER / MAJOR/SHOULD-FIX / MINOR / NIT with exact file:line references; include actionable fix suggestions. * **Always treat 'never bust the cache' as an inviolable architectural invariant**: The user treats cache stability as a hard constraint, not a preference. When reviewing or fixing code, they expect: (1) frozen pin baselines (ltmPinKeys) must be preserved across ALL persist/reload paths — no path may wipe or advance the baseline while system\[2] bytes remain unchanged; (2) any fix must be verified with regression tests that confirm cumulative delta correctness across restarts; (3) the user will explicitly flag any code path that silently drops supersession entries or advances the rolling baseline as a BLOCKER. When implementing LTM pin changes, always audit every persistence site for baseline-wiping writes (e.g., \`ltmPinKeys: JSON.stringify(\[])\` or \`ltmPinKeys: null\`) and fix them before considering the work complete.

* **Always verify test discriminability by asking 'would a revert be caught?'**: After implementing a fix, the user requires that every new test actually discriminates against a revert of the fix — i.e., if the fix were removed, the test must fail. Tests that pass regardless of the fix (non-discriminating) are treated as a defect requiring correction. The standard check is: set test inputs so they exercise the new code path (e.g., warmupHits >= 1 to bypass a gate, not 0). This applies during adversarial/self-review of PRs and is treated as a hard requirement, not a suggestion. When a non-discriminating test is found, the fix is to adjust inputs so the test would catch a revert.

* **LTM pin invariants: never bust cache, ltmCacheText must never diverge from ltmPinText, always update cache with freshly ranked entries**: Three authoritative invariants for LTM pin behavior (absolute directives, repeated across 26-39 sessions): (1) NEVER bust the cache — all pin-preserving paths must keep system\[2] bytes stable. (2) \`ltmCacheText\` must NEVER diverge from \`ltmPinText\` — when pin is reused, both must be persisted together. (3) ALWAYS update the cache with freshly ranked entries (Layer-4 cache update, pipeline.ts line ~5715). Additionally: the coalesced seq-0 delta row must ALWAYS describe the full cumulative \`frozen→current\` delta — never a partial/rolling diff.
