## Long-term Knowledge

### Architecture

<!-- lore:019e25c5-5716-77a0-bcf9-d65f321e5736 -->
* **3-layer gradient model: layer 2 is transient, falls back to layer 1 via urgent distillation**: 3-layer gradient model: Layer 0 (all raw messages + LTM, append-only cache). Layer 1 (distilled prefix + pinned raw window, bust once on entry then warm). Layer 2/emergency (transient hard reset: fresh LTM via \`forSession()\`, 2-3 best distillations, current agentic turn always included — fires 1-2 turns then urgent distillation compresses history, falls back to Layer 1). Layer 2 must NOT set stickiness — stickiness (preventing oscillation) only applies to layers 1-3 where cache is warm; emergency already busts cache. Stickiness bug in \`gradient.ts\`: \`effectiveMinLayer = max(0, lastLayer)\` traps sessions in emergency indefinitely, causing per-turn cache busts. Fix: restrict stickiness to \`lastLayer >= 1 && lastLayer <= 3\`. Context budget caps (160K at Opus) are cost-driven (~$1.00/bust target), not context-window-driven.

<!-- lore:019e1c62-3208-7836-a531-f92d1bb20733 -->
* **Conversation import system: providers, detection, extraction pipeline**: Core import system lives in \`packages/core/src/import/\`. Key design: \`AgentHistoryProvider\` interface with \`detect()\`/\`load()\` methods; providers registered in a global registry (\`providers/index.ts\`). Detection scans all providers, returns \`DetectedSession\[]\`. Extraction calls curator LLM sequentially per chunk, deduplicating ops via \`parseOps()\`/\`applyOps()\`. Idempotency via \`import\_history\` table (DB migration v19). Built-in providers: Claude Code (\`~/.claude/projects/\`), OpenCode (SQLite), Aider (markdown), Codex (\`~/.codex/sessions/\` JSONL), Cline (VS Code globalStorage JSON), Continue (\`~/.continue/sessions/\` JSON), Pi (\`~/.pi/agent/sessions/\` tree-structured JSONL). Auto-import triggered in \`lore run\` via \`maybeAutoImport()\`. Copilot Chat skipped (opaque leveldb). The OpenCode plugin's \`reflect.ts\` (plugin-side recall tool) was dead code and has been removed — plugin uses \`tool: {}\` (empty), gateway handles all recall.

<!-- lore:019e2b12-6e94-7264-bada-88f5e1b68ffe -->
* **LORE\_REMOTE\_URL: remote gateway mode — CLI proxies to remote, skips local startup**: When \`LORE\_REMOTE\_URL\` env var is set, \`lore run\`/\`lore start\` skips starting a local gateway and instead configures agents to point at the remote gateway URL. The local CLI still runs on the developer's machine, so it can safely compute git remote and inject it as \`X-Lore-Git-Remote\` header. The remote gateway never calls \`getGitRemote()\` with client-controlled paths — it trusts the header. This cleanly separates filesystem trust: local CLI does trusted local FS work, remote gateway only receives derived values via headers.

<!-- lore:019e1de2-7650-76c3-9416-40d905dd998f -->
* **OpenAI streaming translation: stateful SSE translators in stream/openai.ts**: OpenAI streaming translation: stateful SSE translators in \`packages/gateway/src/stream/openai.ts\` and \`stream/openai-responses.ts\` consume Anthropic SSE events and emit OpenAI-format SSE events incrementally — clients see tokens as they arrive. The pipeline carries \`effectiveProtocol\` in \`UpstreamResult\` to dispatch to the right translator. All translators must implement: (1) \`cancelled\` flag + \`cancel()\` handler aborting upstream via \`AbortController\`; (2) \`safeEnqueue()\` wrapper that no-ops if \`cancelled\`; (3) error \`catch\` block emitting a protocol-appropriate terminal event (\`response.failed\` for Responses API, \`\[DONE]\` with error for Chat Completions) — otherwise clients hang. Adding a new upstream protocol requires both an accumulator branch and a streaming translator.

<!-- lore:019e1952-398c-753f-b92c-4a5fa5ecf15f -->
* **Pi plugin: which providers can be proxied through the gateway**: Pi plugin gateway proxy compatibility by wire protocol: \*\*Proxiable\*\* (redirect baseUrl to gateway): - \`anthropic-messages\` → \`/v1/messages\`: \`anthropic\`, \`fireworks\`, \`github-copilot\` - \`openai-completions\` → \`/v1/chat/completions\`: \`deepseek\`, \`xai\`, \`groq\`, \`cerebras\`, \`openrouter\`, \`huggingface\`, \`opencode\`, \`opencode-go\` - \`openai-responses\` → \`/v1/responses\`: \`openai\`, \`azure-openai-responses\`, \`openai-codex\`, \`azure-openai\`, \`lm-studio\`, \`ollama\` \*\*Cannot proxy\*\*: \`google\`, \`google-vertex\` (native Google SDK), \`amazon-bedrock\` (AWS SDK), \`mistral\` (\`mistral-conversations\` protocol). \`registerProvider(name, { baseUrl })\` overrides base URL; SDK appends correct path. Gateway routes by URL path only. OpenAI streaming clients now receive true incremental SSE (PR #267, \`stream/openai.ts\`).

### Decision

<!-- lore:019e2b12-6eb0-7996-9a6b-d7ee8ea743e7 -->
* **SaaS/CLI-less interaction pattern: ANTHROPIC\_CUSTOM\_HEADERS won't work without local CLI**: The current approach of injecting session context via \`ANTHROPIC\_CUSTOM\_HEADERS\` env var requires a local \`lore\` CLI process to set the env. In a future pure SaaS product (no local CLI, clients connect directly to hosted gateway), this mechanism breaks. Noted as a research item — need an alternative mechanism for CLI-less context injection (e.g. API key-scoped session metadata, SDK-level header injection, or a gateway-side project configuration API). For now, \`LORE\_REMOTE\_URL\` + local CLI is the supported pattern.

### Gotcha

<!-- lore:019e1c27-967c-7eb4-bd0e-afb195823970 -->
* **Bun NAPI crash on process.exit() — use safeExit() via libc \_exit()**: Bun NAPI crash on process.exit() with fastembed — use safeExit(): Loading fastembed (onnxruntime NAPI bindings) causes a C++ panic on \`process.exit()\` because Bun runs NAPI teardown destructors that throw. Fix: \`packages/gateway/src/cli/exit.ts\` exports \`safeExit(code)\` — uses \`\_exit()\` from libc via \`bun:ffi\` to skip teardown under Bun, falls back to \`process.exit()\` under Node.js. All gateway exit paths must use \`safeExit()\`. Add double-signal guard in shutdown functions to prevent concurrent \`shutdown()\` calls. Also: do NOT call \`embedding.resetProvider()\` in test teardown \`resetPipelineState()\` — this allows re-init to spawn new fastembed workers that crash at process exit. Move \`resetProvider()\` to \`shutdown()\` in \`start.ts\` only. \`resetPipelineState()\` must preserve the 'fastembed unavailable' cached state.

<!-- lore:019e21ed-e36a-78a8-8bba-ee04df1bdf8a -->
* **Curator LLM can corrupt .lore.md entries — title/content mismatch in DB**: The curator LLM can overwrite a knowledge entry's content with unrelated content from another entry, producing a title/content mismatch. This is detectable by comparing \`.lore.md\` (source of truth) against the DB. Fix: manually correct the DB row using direct SQL update, and ensure \`.lore.md\` matches. When a CI failure or weird behavior is traced to a knowledge entry, verify the DB content matches the file content — they can diverge silently during curation.

<!-- lore:019e2b12-6ea6-76dc-ab7a-a1532c60b312 -->
* **git remote -v in hosted gateway — skip when header present, never run with client-controlled cwd**: \`getGitRemote()\` calls \`execSync('git remote -v', { cwd: path })\` where \`path\` comes from \`X-Lore-Project\` header in hosted mode — a client-controlled path. Fix: when \`X-Lore-Git-Remote\` header is present, skip \`getGitRemote()\` entirely and use the header value. Consider a \`LORE\_REMOTE\_MODE=1\` config flag that disables all local FS operations (git subprocess, file reads, watchers) for client-controlled paths in hosted deployments, even if the header is absent.

<!-- lore:019e2828-f27e-7694-a4ed-07627c5b07fc -->
* **import.meta.url is undefined in CJS bundle — esbuild shims import.meta as {}**: In \`packages/gateway/script/bundle.ts\`, esbuild with \`format: 'cjs'\` replaces \`import.meta\` with an empty object \`{}\` — so \`import.meta.url\` evaluates to \`undefined\` at runtime. This breaks the embedding worker path resolution in \`packages/core/src/embedding.ts\` (LOREAI-GATEWAY-D/E are the same bug). Two co-required fixes: (1) Guard \`import.meta.url\` with a \`typeof\` check and fall back to \`\_\_filename\` + \`pathToFileURL\` for CJS; (2) Build \`embedding-worker.cjs\` as a separate output file alongside \`dist/index.cjs\` in the gateway bundle script — it is currently never built for the npm bundle, only for the Bun binary build.

<!-- lore:019e1de2-75fe-72f5-8f20-36b4923c1ea9 -->
* **LTM cache delete must be inside the 'changes made' guard in curator.ts**: LTM cache delete must be inside the 'changes made' guard: \`ltmSessionCache.delete(sessionId)\` must only fire when the curator actually makes DB writes. Placing it unconditionally invalidates the cache on every curation run — even no-ops — forcing expensive LTM rebuilds. Guard it inside \`if (changesApplied)\` / \`if (ops.length > 0)\`. Applies to both \`packages/core/src/curator.ts\` and \`packages/gateway/src/curator.ts\`. Also: recall follow-up requests must set \`cacheConversation: false\` — otherwise the modified message array triggers a full cache write at 5m TTL pricing. The non-streaming recall follow-up path must NOT re-issue the upstream request — capture response body once to prevent double token cost (~$0.56–0.94/occurrence on Opus) and double cache prime.

<!-- lore:019e2760-86be-7f0a-978e-8aafc873b9c8 -->
* **OpenAI/Responses API upstreams don't receive LTM — req.system passed through unchanged**: In \`packages/gateway/src/pipeline.ts\`, when the upstream protocol is OpenAI Chat Completions or Responses API, \`req.system\` is passed through without LTM concatenation. Only the Anthropic path injects LTM into the system prompt. This means sessions using OpenAI-protocol upstreams get no knowledge context. Fix: apply the same LTM injection logic to all upstream paths before forwarding, not just the Anthropic branch.

<!-- lore:019e1b2d-2bbb-7f18-b0cd-8c393aa6693d -->
* **OpenCode plugin spawns gateway via src/index.ts path, but npm publish ships only dist/**: OpenCode/Pi plugins start gateway in-process via \`startInProcess()\` (\`loadConfig()\` + \`startServer()\` from \`@loreai/gateway\`). In monorepo, Bun resolves \`@loreai/gateway\` to \`src/index.ts\` via \`'bun'\` export condition; npm resolves to \`dist/index.cjs\` (only exists post-build — CI typechecks fail). Fix: use variable indirection to break static analysis: \`const mod = '@loreai/gateway'; await import(mod)\`. Pi plugin must externalize \`@loreai/gateway\` in esbuild. \`NODE\_ENV === 'test'\` skips gateway init. Gateway CJS bundle (\`dist/index.cjs\`, \`dist/bin.cjs\`) built via esbuild with \`node-polyfills.ts\` shimming \`globalThis.Bun.serve()\` → \`node:http.createServer()\`. \`@sentry/bun\` remapped → \`@sentry/node\` at bundle time via esbuild plugin (alias fails for transitive-only deps). Use port \`0\` (ephemeral) in tests.

<!-- lore:019e1de4-cada-77cf-ac3b-79ab96b963e4 -->
* **Query expansion disabled by default AND LLM client not wired in gateway (IF-5 fix)**: Query expansion disabled by default AND LLM client not wired in gateway — two co-required fixes: (1) \`queryExpansion\` default in \`packages/core/src/config.ts\` must be \`true\` in BOTH the field-level \`.default()\` AND the section-level \`.default({})\` at ~line 199 — the section-level overrides field-level in Zod, so changing only one has no effect. (2) \`executeRecall()\` in \`packages/gateway/src/recall.ts\` must accept an optional \`LLMClient\` param and pass it to \`runRecall()\` — without it, \`if (searchConfig?.queryExpansion && llm)\` is always false. Both pipeline.ts call sites (streaming and non-streaming) must pass \`getLLMClient(config)\`.

<!-- lore:019e283f-60c0-7cf3-a541-241a9d982933 -->
* **Recall follow-up strips thinking blocks — Anthropic API rejects assistant messages missing thinking signatures**: In \`packages/gateway/src/recall.ts\` (lines ~401-404), the assistant message injected for recall follow-up is built with only the \`tool\_use\` block, stripping any preceding thinking blocks. When extended thinking is enabled, the Anthropic API requires thinking blocks (with their \`signature\` fields) to precede \`tool\_use\` blocks in assistant messages — omitting them causes a 400 error. Fix: when building the assistant message for recall follow-up, include all content blocks from the original response (thinking + tool\_use), not just the tool\_use block.

<!-- lore:019e1de2-7639-7b32-b4c1-e64486934c27 -->
* **TTL downgrade hysteresis: downgradeStreak field prevents compounding cache busts**: Auto-TTL downgrade hysteresis: In \`packages/gateway/src/pipeline.ts\`, downgrade from 1h→5m TTL requires 3 consecutive short-gap turns (\`ttlDowngradeStreak\` in \`SessionState\`). A single qualifying turn must not trigger the downgrade — it causes compounding cache busts at 2x pricing. Block downgrade if >50% of session tokens are cached. Reset streak to 0 on any long-gap turn. Also: subagent turns and tool-use continuations must be excluded from gap recording (they have near-zero gaps, biasing EMA low). Capture \`prevStopReason = sessionState.lastStopReason\` before line 1667 overwrites it, then skip gap recording when \`prevStopReason === 'tool\_use'\` or \`isSubagentTurn\`.

<!-- lore:019e1e9f-3131-733f-978e-dde6f41e29fd -->
* **Upgrade lock double-acquisition bug: same process re-locks same file**: In \`packages/gateway/src/cli/lib/binary.ts\`, \`downloadBinaryToTemp()\` acquires a lock on \`\<execPath>.lock\` and holds it. Then \`installBinary()\` computes the same install path and tries to \`acquireLock()\` again. \`handleExistingLock()\` only allows re-entry if \`existingPid === process.ppid\` (parent), but the lock was written by the same process (\`existingPid === process.pid\`), so it throws 'Another upgrade is already in progress'. Fix: in \`handleExistingLock\`, also allow re-entry when \`existingPid === process.pid\`. Double \`releaseLock()\` is safe — \`releaseLock\` swallows errors so the second call is a no-op after the file is deleted.

<!-- lore:019e1cd6-05d2-74c5-aea8-fd827a4a45e7 -->
* **vectorSearch() is unscoped — test cleanup must delete all embedding rows**: vectorSearch() is unscoped by project and ignores category filters — two gotchas: (1) \`vectorSearch()\` in \`packages/core/src/ltm.ts\` queries \`knowledge WHERE embedding IS NOT NULL AND confidence > 0.2\` with no \`project\_id\` filter (intentional for cross-project search). Test suites scoped to one project leak embedding rows into other vectorSearch tests — \`beforeEach\` must \`DELETE FROM knowledge WHERE embedding IS NOT NULL\`. (2) \`vectorSearch()\` also has no \`excludeCategories\` parameter, so category exclusions from \`forSession()\` callers have no effect — add optional \`excludeCategories\` param and propagate from callers. Also: global entries (pid=null) now force \`crossProject=true\`; confidence is clamped to \[0.0, 1.0] in \`update()\`.

### Pattern

<!-- lore:019e21e0-9f09-7dd1-a899-854579e160cc -->
* **Enhanced dedup: title overlap + vector similarity (Nomic v1.5)**: Nomic Embed v1.5 dedup threshold: same-domain cosine similarity spreads 0.46–0.70 (vs BGE Small which clusters at 0.93–0.97+, making dedup unusable). Correct dedup threshold: \*\*0.935\*\* — at-or-above is genuine duplicate. Range 0.85–0.91 contains 'related but distinct' entries; 0.85 produces false positives across project boundaries. \`deduplicate()\` in \`packages/core/src/ltm.ts\` uses both title word-overlap (0.7 Jaccard + 4+ shared words) AND vector cosine similarity. BGE Small embeddings are auto-nulled by \`checkConfigChange()\` on startup; \`backfillEmbeddings()\` re-embeds with Nomic v1.5. \`lore data reindex\` triggers backfill on-demand without gateway restart.

<!-- lore:019e25ee-179d-7027-b092-94179f47dd83 -->
* **State persistence strategy: immediate vs periodic vs per-turn**: State persistence tiers for gateway session state — never rely on shutdown handlers (SIGKILL/crashes bypass them): (1) \*\*Immediate\*\* — session identity fields (fingerprint, headerSessionId, headerName) on mutation (rare). (2) \*\*Per-turn\*\* — cost snapshot piggybacked on existing \`saveSessionTracking\` batched save in \`postResponse\`. (3) \*\*30s periodic\*\* — gradient EMAs and cache warming state via dirty flag + idle scheduler tick in \`idle.ts\`. DB migration v24 adds columns to \`session\_state\` for all three categories. \`MIGRATIONS.length\` governs migration version — never update \`SCHEMA\_VERSION\`. Max data loss on crash: ~30s of gradient/warmup state.

<!-- lore:019e2213-6dac-7392-a1b1-607e48c7ae19 -->
* **Token-budget batching for embedding backfill — nextBatch() replaces fixed BACKFILL\_CHUNK\_SIZE**: Embedding backfill: token-budget batching + OOM handling. Distillation observations (4476+ chars) and knowledge entries (~1200 chars) have very different token counts — a fixed \`BACKFILL\_CHUNK\_SIZE\` causes ONNX OOM (error codes \`287180544\`, \`144786472\`). Fix: implement \`nextBatch(texts, budget)\` approximating tokens as \`chars/4\`, greedily filling until \`batch\_size × max\_seq\_len > budget\` (budget=8192). Apply to both \`backfillEmbeddings()\` and \`backfillDistillationEmbeddings()\` in \`packages/core\`. Pass \`truncation: true\` to pipeline call. In the embedding worker, catch OOM errors and re-throw with human-readable message; \`log.error()\` routes to Sentry automatically — no direct Sentry import needed.

### Preference

<!-- lore:019e2820-3ed0-7cc0-97a7-2c654df763ec -->
* **IDs starting with LOREAI-GATEWAY- are Sentry issue IDs**: Any identifier starting with \`LOREAI-GATEWAY-\` (e.g. \`LOREAI-GATEWAY-F\`) is a Sentry issue ID for the gateway project. Always treat these as Sentry issue references when encountered in conversation — fetch the issue via Sentry CLI/API to get stack traces, user counts, and release info before investigating the codebase.

<!-- lore:019e1c54-02ef-7ca8-90ce-b75d97917b96 -->
* **Never use --admin flag for PR merges unless explicitly told**: Always watch CI after submitting a PR — never use \`--admin\` to bypass branch protection and never use \`gh pr merge --auto\`. Fix any CI failures until green, then merge with \`gh pr merge --squash\`. Also: always check for merge conflicts when CI doesn't start — a PR with merge conflicts won't trigger CI. Resolve by rebasing onto main (\`git rebase origin/main\`), then force-pushing the branch.

<!-- lore:019e2745-83af-766d-8e97-c8d4ca202388 -->
* **Pin stable LTM (preferences) for ≥1h so cache survives curation changes**: When splitting LTM into stable (preferences) vs context-bound blocks, pin stable LTM with a 1h TTL even if the curator updates a preference entry mid-session — the old cached prefix remains usable for ~1h, avoiding an unnecessary cache bust. The 1h TTL on block 2 (stable LTM) also implicitly covers block 1 (host prompt) via Anthropic prefix caching, making a separate cache\_control on block 1 redundant.

<!-- lore:019e2168-2fa4-77bd-a557-9d6dbcb40d81 -->
* **Prefer WASM backend over native onnxruntime-node for compiled binaries**: WASM backend for Bun \`--compile\` binaries with transformers.js: \`binaryExternalsPlugin\` in esbuild redirects \`onnxruntime-node\` → \`onnxruntime-web\` via \`onResolve\` (static imports only — does NOT redirect dynamic \`import()\` calls) and patches transformers.js CDN fallback via \`onLoad\` to read \`wasmPaths\` from \`globalThis.\_\_LORE\_VENDOR\_WASM\_PATHS\_\_\` (object form \`{ mjs, wasm }\` with exact hashed \`$bunfs\` filenames — directory strings fail because Bun hashes bundled WASM filenames). WASM files embedded as Bun \`{ type: 'file' }\` assets in the wrapper; wrapper sets \`globalThis.\_\_LORE\_VENDOR\_WASM\_PATHS\_\_\` before importing the worker. No onnxruntime import in wrapper or worker. For npm/CJS builds, \`onnxruntime-node\` stays external. WASM is ~2x faster on batches than native. Importing \`onnxruntime-web\` explicitly alongside the redirect creates two ort instances — 'cannot register backend cpu using priority 10' error.
