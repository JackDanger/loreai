* **A1 quota trigger: counter-based enforce\_row\_quota — only INSERTs and revivals grow live-row count**: Post-A1 enforce\_row\_quota() (migration 0007) reads O(1) from user\_table\_usage(scope\_id, table\_name, row\_count, byte\_count). Advisory lock keyed on hashtextextended(scope\_id||'|'||tg\_table\_name,0) MUST be taken before the exists\_pk check (0005 ordering) — 0007 moved exists\_pk outside the lock (SHOULD-FIX). * **Append-only knowledge versioning: invariants + metadata forward-copy**: Two identities per entry: logical\_id (stable) + version (immutable, UUIDv7). Version rows never mutate except is\_current — content/title/category/is\_deleted/version/logical\_id fixed after insert. Never resurrect a logical\_id whose latest version has is\_deleted=1 (no tombstones table needed). All non-FTS reads use knowledge\_current view (is\_current=1 AND is\_deleted=0); FTS5 indexes only current+live rows, removed on append. Outbox keyed by logical\_id so deletes match their remote row. appendVersion() uses COALESCE(?, metadata) to forward-copy prior metadata when no override given (stringifyMetadata({}) → NULL). update() only passes metadata to appendVersion in the content-changed branch — metric/sensitivity-only updates leave gitHead untouched (A2 immutability). remove(id, metadata?) passes metadata to the death-cert version.

* **bspatch.ts: OldReader abstraction (FileOldReader/MemoryOldReader) + applyPatchChainInMemory**: PR #1436 rewrote packages/gateway/src/cli/lib/bspatch.ts for delta-upgrade speed: vectorized SWAR wrapping-add (addDiffChunk, carry-less Uint32Array formula) replaces the per-byte add loop. * **bspatch.ts: TRDIFF10 format, OldReader abstraction, in-memory patch chaining**: TRDIFF10 header (32B): magic\[0:8], controlLen/diffLen/newSize each i64 LE via offtin's sign-magnitude decoding (not two's-complement), followed by zstd(control)|zstd(diff)|zstd(extra). \`OldReader\` abstracts fd-backed (pread + single-block read-ahead cache, backed by a reflinked temp copy) vs in-memory (\`MemoryOldReader\`) sources so \`transformPatch\`/\`applyPatch\` read old bytes uniformly whether patching from disk or mid-chain. Multi-patch chains (\`applyPatchChainInMemory\`) keep intermediates entirely in JS heap and only the FINAL binary is persisted + SHA-256 hashed — avoids N-1 redundant disk writes/temp-copies/hash passes per hop vs the old on-disk ping-pong approach. Peak heap per hop ≈ 2x one binary's size (prev buffer + next output), not cumulative across chain length, because each \`MemoryOldReader\` is \`.close()\`d and GC-eligible right after its hop.
