## Long-term Knowledge

### Architecture

<!-- lore:019e044a-a65f-7b25-bbcf-2c73b017a05e -->
* **OpenCode uses compaction not front-trimming for context management**: OpenCode does NO front-trimming/slicing of messages before API calls. Context management uses compaction: when \`isOverflow()\` detects token count ≥ usable context, \`compaction.create()\` generates AI summary of old messages. \`filterCompacted()\` in message-v2.ts walks messages newest-first, stops at compaction boundary. Messages before the boundary excluded from API calls. Tool output pruning clears old tool outputs beyond 40K tokens (marks \`time.compacted\`). Plugin hook (\`experimental.chat.messages.transform\`) delegates windowing to external systems like Lore gradient.

### Gotcha

<!-- lore:019e0434-7467-7475-a924-1c865f2f846e -->
* **Dedup annotation changes affect pinned window token estimates**: \`deduplicateToolOutputs()\` runs fresh every turn and can annotate different messages as duplicates when new tool outputs arrive. When current turn contains duplicate of earlier output, the earlier message gets annotated (smaller token count), changing \`estimateMessage()\` for messages inside pinned window. With offset-from-end pinning and 10% hysteresis + re-budgeting, most dedup-driven token shifts are absorbed. Only large dedup changes (>10% of pinnedBudget) still cause pin-overflow and rescan.

<!-- lore:019e0453-3e5f-78ea-95ab-6d942b8c503b -->
* **Gateway generates random message IDs causing all cache busts**: Gateway's temporal adapter (line 108) calls \`randomUUID()\` for every message on every request. Same messages get different IDs each turn, breaking both layer 0 and layer 1+ caching. Layer 0: first message ID changes → prefix fingerprint changes → API cache bust. Layer 1+: windowed messages have new IDs → prefix fingerprint changes → gradient cache bust despite pin-hit. Fix: replace \`randomUUID()\` with deterministic hash of message content so identical messages get stable IDs across requests.

<!-- lore:019e0453-3e48-7722-84b4-14bff052d17e -->
* **Gateway generates random UUIDs causing all cache busts**: Gateway's temporal adapter (line 108) calls \`randomUUID()\` for every message on every HTTP request, generating fresh IDs even for identical message content. This breaks both layer 0 and layer 1+ caching: layer 0 busts because first message ID changes (prefix fingerprint mismatch), layer 1+ busts because pinned window messages get new IDs despite pin-hit. Fix requires deterministic content-based IDs (hash of message content) instead of random UUIDs to ensure same message content gets same ID across requests.

<!-- lore:019e044a-a63c-7bb5-b7fa-a8bf816fba9a -->
* **Layer 0 busts from front-trimming: not preventable at gateway**: Layer 0 passthrough returns \`input.messages\` unchanged. When the host (OpenCode) changes message IDs at the front between turns — from \`filterCompacted()\` boundary shifts, compaction, or message regeneration — the Anthropic prompt cache invalidates (byte prefix changed). The gateway cannot prevent this at layer 0. Sessions with \`cap=off\` (missing model pricing) stay at layer 0 indefinitely, busting every turn. Fix: ensure sessions escalate to layer 1 (set \`maxLayer0Tokens\` in config or provide model cost data) where \`tryFitStable\` controls the window. Layer 0 busts are real API cache misses, not diagnostic artifacts.

<!-- lore:019e0445-9ede-77ac-bf07-b4745feca17d -->
* **Layer 0 cache busts are expected with OpenCode front-trimming**: Layer 0 sessions bust on every turn when host applications change front messages (compaction boundaries, message regeneration). At layer 0, gradient returns messages unchanged (passthrough), so first message ID changes from host-internal management despite growing count. Sessions stay at layer 0 when \`input.model.cost?.cache?.read\` is missing - the plugin checks this field at line 1082-1089 to set \`activeSessionModel\`. Without cost data, cost formula never executes and \`cap=off\`. OpenCode passes resolved model from \`provider.getModel()\` which should include cost field. Fix: ensure model provider returns complete cost structure or set explicit \`cfg.budget.maxLayer0Tokens\`.

<!-- lore:019e044b-0be7-7bb4-88ed-85ee05a3c73f -->
* **Offset-from-end pin resilience to front-trimming**: Gradient pins use offset-from-end positioning (\`pinnedRawCount\` + \`pinnedTotalCount\`) instead of absolute message IDs to survive front-trimming. When messages are removed from conversation start, pin maintains validity by counting backward from conversation end. Re-pins at current budget when old pinned window still fits (budget growth scenario). This prevents cache busts from external message trimming affecting pin boundaries while preserving cache reuse stability.

<!-- lore:019e0453-3e9b-7bf6-bca2-dff7731d74fa -->
* **OpenCode front-trimming despite compaction-only architecture**: OpenCode removes ~2 messages from front each turn despite architectural intent to use compaction-only. Evidence: gateway pinned window at fixed offset shows different firstID each turn (c2c4cdf2→354747ea) while pinnedIdx stays constant. Message count grows by +2 net (195→197→199) but messages\[2] changes identity, indicating front removal + rear append. Not from filterCompacted() boundary shifts (no compaction active) or trailing message drops (removes from end). Unknown mechanism in OpenCode process trims front before gateway receives messages, breaking offset-from-end pin assumptions.

<!-- lore:019e0453-3e9f-7a90-bfe3-8b82b65a0ac1 -->
* **OpenCode message array instability affects gradient firstID tracking**: OpenCode's message array isn't append-only between turns. Messages at fixed indices change IDs due to front-trimming, compaction boundary shifts, or message regeneration. When gradient pins at \`pinnedIdx=2\` across turns, \`messages\[2].info.id\` changes even though the pin hits. This causes firstID changes in prefix fingerprints despite stable window positions. Total message count grows (+2 per turn) but front messages are replaced, shifting content at same array indices. Gateway receives pre-processed messages from OpenCode SDK, not raw internal message objects.

<!-- lore:019e0453-3ea8-7045-8fdf-762d8fe34122 -->
* **OpenCode message array instability breaks gradient pins despite stable indexing**: OpenCode changes messages at fixed array positions between turns, not just appending. Even with stable \`pinnedIdx\` calculation (offset-from-end), the message at that index gets different ID each turn. Causes \`firstID\` changes in prefix fingerprints despite pin hits. Not traditional front-trimming - could be message regeneration, tool result updates, or other in-place modifications. Gradient pin logic assumes append-only behavior but OpenCode's message management is more dynamic. Results in stable pin geometry but unstable content identity.

<!-- lore:019e0452-c9d6-78ab-83b0-bef6cec3b890 -->
* **OpenCode message array not append-only causes firstID instability**: OpenCode's message array changes at fixed positions between turns, not just appends. Even when total count grows (+2 per turn), messages at same index have different IDs turn-over-turn. This breaks offset-from-end pinning assumptions. Pattern observed: \`pinnedIdx=2\` stable, but \`messages\[2].info.id\` changes every turn (c2c4cdf2→354747ea→...). Likely causes: message regeneration, compaction boundary updates, or tool result modifications affecting middle of array. Unlike pure append-only growth, this makes any position-based caching unstable since message identity at fixed indices isn't preserved.

* **OpenCode message ID instability breaks pin-hit despite stable offsets**: OpenCode modifies messages in-place between turns, changing message IDs at fixed positions even when total count grows predictably. Pin offset from end stays stable (\`pinnedIdx=2\`), but \`messages\[2].id\` changes each turn despite no front-trimming. Causes \`firstID\` mismatches in gradient prefix fingerprints even with perfect offset-based pinning. <!-- lore:019e0451-69b3-7ca4-b6d8-08c7f830f3df -->
* **OpenCode mutates messages in-place, breaking offset-from-end pinning**: OpenCode mutates messages in-place, breaking offset-from-end pinning: OpenCode changes messages at fixed array positions between turns via regeneration, compaction, or tool results. When \`pinnedIdx\` calculation finds correct index but \`messages\[pinnedIdx].id\` differs from cached \`firstID\`, it indicates in-place mutation. Deterministic ID generation (MD5 of content + role + tool calls) helps identify mutations but doesn't prevent them. Pin-miss occurs because offset-from-end assumes append-only + front-trim, but OpenCode's array is fully mutable at any position.

<!-- lore:019e0452-c9e1-7321-a421-aa2d01ae7a72 -->
* **OpenCode sliding compaction boundary causes apparent front-trimming**: OpenCode's \`filterCompacted()\` creates sliding window effect that mimics front-trimming. Messages array stays stable, but compaction boundary advances as tool outputs exceed 40K tokens or compaction completes. Each turn, boundary moves forward, changing which messages pass the filter. Results in same net effect as front-trimming: first message ID in API call changes, causing layer 0 cache busts. Offset-from-end pinning still works because it tracks from actual message array end, not filtered boundary.

<!-- lore:019e044c-7305-759b-a2ba-406df61496d9 -->
* **Pin-miss and pin-overflow fixes stabilize layer 1+ sessions completely**: Recent fixes to gradient pin system achieve perfect layer 1+ stability: 1) Pin-miss fixed by replacing absolute message ID pinning with offset-from-end (\`pinnedRawCount\` + \`pinnedTotalCount\`), resilient to front-trimming. 2) Pin-overflow fixed by re-pinning at current budget when window grows beyond old hysteresis but fits current \`rawBudget\`, avoiding expensive rescans. Gateway sessions show 6+ consecutive turns with 0 busts at layer 1. Only remaining busts are layer 0 sessions with \`cap=off\` experiencing expected front-trimming busts.

### Pattern

<!-- lore:019e0436-e3ff-7307-bf65-1e943ea15b11 -->
* **Budget snapshot + hysteresis pattern for tryFitStable pin stability**: \`RawWindowCache\` stores \`pinnedRawCount\` (messages in window), \`pinnedTotalCount\` (total messages when pinned), and \`pinnedBudget\` (rawBudget snapshot). Pin validity: \`pinnedTokens <= pinnedBudget \* 1.10\`. When pin overflows hysteresis but \`pinnedTokens <= currentRawBudget\`, re-pins at current budget without rescan. When \`currentRawBudget\` also insufficient, falls through to \`tryFit()\` for fresh scan. Offset-from-end approach (\`messages.length - pinnedRawCount - newMessages\`) is resilient to front-trimming by host applications.

<!-- lore:019e0453-b388-76cb-a258-1adf8c0e178f -->
* **Deterministic ID generation for gradient stability**: Gateway uses deterministic ID generation to prevent gradient cache busts from ID instability. \`deterministicId()\` creates stable IDs from content hash + position: \`${position}-${contentHash.slice(0, 8)}\`. Applied in \`contentBlockToPart()\` for message parts and \`gatewayMessagesToLore()\` for messages themselves. Position ensures uniqueness when content repeats; hash ensures same content gets same ID. This prevents pin-miss scenarios where identical conversations get different IDs between turns, which would break offset-from-end pinning in \[\[019e044a-a640-7b87-bab4-24a7c9f9085f]]. Deterministic IDs maintain gradient cache validity across gateway restarts and message transformations.

<!-- lore:019e041a-3027-7f62-b103-35db886d35bd -->
* **Gradient cache-bust detection via prefix fingerprints**: Prefix fingerprint: \`layer:first5messageIDs\` (gradient.ts:1641-1642). Layer 0 uses raw message IDs; layer 1+ uses distilled prefix IDs + raw window IDs. Cache-bust detected when \`lastPrefixHash !== prefixHash\`. Layer 0 busts from host front-trimming are API-level cache misses, expected behavior. Layer 1+ busts indicate gradient problems — offset-from-end pinning + hysteresis should keep bust rate near 0%. HIGH BUST RATE warning fires at ≥20 transforms with >50% bust rate. \`bustCount\`/\`transformCount\` are per-session, in-memory, reset on restart.

<!-- lore:019e044b-0beb-7f98-a363-a88c0481b43a -->
* **Gradient pin stability via offset-from-end and re-pin on budget growth**: Pin-miss fixed by replacing absolute message IDs with offset-from-end (\`pinnedRawCount\` + \`pinnedTotalCount\`), making pins resilient to front-trimming. Pin-overflow fixed by re-pinning when window grows beyond old hysteresis but fits within current \`rawBudget\`, avoiding full rescan. Combined fix eliminates layer 1+ cache busts - gateway sessions show sustained 0 busts across multiple turns with growing token counts. Pin holds perfectly as new messages arrive.

<!-- lore:019e044a-a640-7b87-bab4-24a7c9f9085f -->
* **Offset-from-end pinning prevents front-trim eviction**: \`RawWindowCache\` uses \`pinnedRawCount\` (window size at pin creation) + \`pinnedTotalCount\` (total messages at pin creation) instead of absolute message IDs. At read time: \`pinnedIdx = messages.length - pinnedRawCount - newMessages\` where \`newMessages = max(0, messages.length - pinnedTotalCount)\`. Resilient to front-trimming because offset is relative to tail. When host evicts old messages from front, the pinned window stays intact since it's measured from the end. Previous ID-based approach caused \`pin-miss\` on every turn when host trimmed front messages.

<!-- lore:019e044b-0bef-7782-abbd-6563486d483a -->
* **Offset-from-end re-pin strategy prevents gradient cache busts**: \`tryFitStable()\` with offset-from-end pinning + re-pin on budget growth eliminates layer 1+ cache busts. When budget increases, creates new pin at same offset from conversation end instead of invalidating. Pin tracks growing conversations naturally without busting. Combined with 10% hysteresis from \[\[019e0436-e3ff-7307-bf65-1e943ea15b11]], provides stable cache reuse even as token budgets fluctuate from EMA drift. Gateway sessions show zero layer 1 busts after implementing this pattern, with tokens growing naturally (62K→63K) while maintaining pin hits.

<!-- lore:019e0436-e406-7438-989b-e54de73cd1f9 -->
* **Per-session ltmTokens isolation via optional sessionID parameter**: \`setLtmTokens(tokens, sessionID?)\` stores on \`SessionState.ltmTokens\` when sessionID provided, falls back to module-level \`ltmTokensFallback\`. \`getLtmTokens(sessionID?)\` reads session state first. \`transformInner\` resolves \`sid\`/\`sessState\` before computing \`usable\`, reads \`sessState.ltmTokens\`. \`calibrate()\` reads \`state.ltmTokens\` for \`lastKnownLtm\`. All adapter call sites (OpenCode, Gateway, Pi) pass sessionID. Eliminates cross-session contamination where one session's LTM token count polluted another's budget calculation.

<!-- lore:019e044a-a643-770f-97ee-198396886116 -->
* **Pin re-budgeting avoids rescan when current budget fits**: When pinned window overflows \`pinnedBudget \* 1.10\` hysteresis but fits within current \`rawBudget\`, re-pin at current budget without full backward rescan. Updates \`pinnedBudget\`, \`pinnedRawCount\`, and \`pinnedTotalCount\` in cache. Avoids cache bust from prefix fingerprint change that a full \`tryFit()\` rescan would cause. This handles the common case where new messages grow the window past the old budget's hysteresis, but the current budget (which may be larger due to overhead EMA drift) can absorb the growth.

<!-- lore:019e0442-6979-7219-a153-0d39b4b92879 -->
* **tryFit backward window scanning algorithm**: \`tryFit()\` implements backward window scanning for layers 1-3. Identifies current turn via last user message through tool chains, walks backward accumulating tokens until budget exhausted. \`tryFitStable()\` reuses previous window when it still fits budget with 10% hysteresis buffer. Pin validity uses offset-from-end (\`pinnedRawCount\` + \`pinnedTotalCount\`) rather than absolute message IDs, making it resilient to front-trimming. When pinned window grows beyond old budget but fits current budget, re-pins at new budget instead of full rescan. Both use splice-in-place for layers 1-4, passthrough for layer 0.
