## Long-term Knowledge

### Architecture

<!-- lore:019e0238-28bb-72e4-8551-f308662d86c0 -->
* **Embedding backfill unified at startup across all hosts**: P0 fix: moved embedding backfill from OpenCode-only (\`packages/opencode/src/index.ts:1480-1487\`) to a unified \`runStartupBackfill()\` in core. Both OpenCode and Pi now call this function at init, ensuring all adapters embed knowledge + distillations equally. Logs coverage stats to stderr (always visible, not gated on debug mode), making embedding failures immediately surfaced. Before: Pi had 0% distillation embedding coverage (~1,333 rows); after: both adapters trigger backfill identically.

<!-- lore:019e0222-5198-7a02-bdda-f44d4b7b3f3b -->
* **Embedding provider migration strategy with checkConfigChange**: Embedding provider migration strategy with checkConfigChange: Config changes (provider swap, model/dimensions) detected via fingerprint (\`provider:model:dimensions\`) in \`kv\_meta\` table. On startup, \`checkConfigChange()\` compares current config to stored fingerprint; on mismatch, stale embeddings clear and regenerate. Enables seamless migrations without manual intervention. Default is now local (fastembed, offline, 384 dims) via \`@xenova/transformers\`. Users can opt back to Voyage via \`.lore.json\` config. Local embeddings require no API keys; Voyage (1024 dims) available as premium option.

<!-- lore:019e03da-045c-7ce9-8fae-0f01de363b70 -->
* **Gateway as universal HTTP integration point**: Gateway accepts Anthropic \`/v1/messages\` protocol, applies full Lore pipeline (gradient + LTM + distillation + curation), forwards to upstream. Works for any HTTP client by setting base URL. Enables recall tool interception universally without per-host plugins. Session tracking via base62 markers injected into assistant responses. Compaction requests intercepted before upstream forwarding to trigger distillation.

<!-- lore:019e0236-1d10-7b16-abdc-a9c4505ce47a -->
* **Gateway buffer-or-stream strategy for recall interception**: Gateway selective recall interception for streaming: When recall tool\_use appears mid-response, suppress only its content blocks from the client stream (not other concurrent tools). Execute recall in background. If recall is sole tool, execute & send follow-up upstream, forward continuation in same HTTP response ('pause & continue' UX). If mixed with other tools, store pending recall result, inject it as tool\_use+tool\_result into next request before forwarding upstream. Requires cross-request session state (pending recalls with TTL). Tradeoff: selective buffering removes the 2-8s delay penalty from full buffering while maintaining transparent tool execution flow.

<!-- lore:019e0381-8263-71ec-b845-06b715363ce2 -->
* **Gateway recall interception — two-case stream-and-intercept design**: Gateway recall interception — two-case stream-and-intercept design: Gateway transparently executes recall tool calls mid-response. Case 1 (recall-only): suppress recall tool\_use blocks from client stream, execute recall via follow-up request, inject synthetic '\[Searching memory...]' text block at suppressed index, re-index continuation content blocks starting at \`blockOffset\`, splice into same HTTP response ('pause & continue' UX). Case 2 (recall + other tools): suppress recall, execute in parallel, store result with 60s TTL, inject into next request's assistant message. Both cases handled in streaming (Anthropic) and non-streaming (OpenAI) paths via \`RecallAwareStreamAccumulator\` wrapper and \`buildRecallFollowUp()\` helper.

Uniform segments (low c\_norm) score higher; old bursty segments lower. The list deduplicates across BM25+vector results and injects as an additional RRF list. Keeps existing relevance signals intact — quality is a mild secondary signal. Enables temporal clustering heuristic (discovered in diagnostic analysis) to modestly improve recall quality for memory segments with uneven conversation pacing.

### Gotcha

<!-- lore:019e03e4-2eb4-767b-bd6b-22a2b04908f2 -->
* **Auto-generated .lore.md creates uncommitted changes**: Lore's knowledge management automatically updates \`.lore.md\` with extracted insights, creating unstaged changes that need manual handling during development workflow. When creating branches or PRs, check for and include these auto-generated changes to maintain knowledge state consistency. The file tracks distilled conversation knowledge and should be committed alongside code changes to preserve the full context of development decisions.

<!-- lore:019e022f-7149-73dc-a105-d090a7cf1426 -->
* **c\_norm temporal clustering signal defined but not used in recall scoring**: \`temporalCnorm()\` (temporal.ts:297–320) computes \[0,1] attention imbalance but only feeds distillation segmentation. NOT wired into recall search pipeline despite comment (line 289) mentioning "recall time-biasing" as future use case. Recency boost exists (recall.ts:368–379 re-sorts temporal results by created\_at DESC) but doesn't account for conversation time windows or clustering patterns. Opportunity: use c\_norm to weight temporal vs semantic relevance when query context is clustered vs evenly distributed.

<!-- lore:019e03e3-a2ec-72f5-9964-5f5e906ce46e -->
* **Cache TTL must use string literals '5m' or '1h', not seconds**: Anthropic's prompt caching API rejects numeric TTL values like \`"3600"\` (seconds), requiring ISO 8601 duration strings: \`"5m"\` (5 minutes) or \`"1h"\` (1 hour). This affects \`llm-adapter.ts\`, \`batch-queue.ts\`, and \`translate/anthropic.ts\`. Error: \`invalid\_request\_error: system.0.cache\_control.ephemeral.ttl: Input should be '5m' or '1h'\`. Use \`"1h"\` for worker system prompts (static, durable across bursts) and \`"5m"\` for conversation caching (expires between user turns).

<!-- lore:019e03da-d196-7f42-bc82-bef0af38b6fd -->
* **Case 1 continuation follow-up must exclude pre-recall text**: Case 1 continuation follow-up must exclude pre-recall text: When executing recall follow-up in Case 1 (streaming, recall-only), the follow-up assistant message must contain ONLY the \`tool\_use\` block for recall — exclude any pre-recall response text. Including pre-recall text causes duplication in the continuation; the model then regenerates the entire response after receiving recall results. Symptom: empty/duplicate content blocks in continuation, long delays. Fix: in \`buildRecallFollowUp()\`, filter assistant message to tool\_use blocks only. Implementation verified in gateway/src/pipeline.ts and recall.ts.

<!-- lore:019e0381-8272-77d7-88e4-29caa47b2635 -->
* **Case 1 streaming continuation requires block re-indexing across message\_start boundary**: When continuing after recall execution in Case 1 (streaming recall-only), the continuation response's \`message\_start\` must be suppressed (client already has one), and its content blocks must be re-indexed to start at \`firstSuppressedIndex + 1\`. The continuation's \`message\_delta\`/\`message\_stop\` close the message naturally. Without re-indexing, client sees duplicate/incorrect block indices.

<!-- lore:019e03df-fd74-7c78-bee5-a60179d41e02 -->
* **Debug logging cleanup required before Case 1/Case 2 merge**: Verbose per-event debug logs (e.g., \`log.debug()\` for every SSE delta) must be removed or downgraded to \`log.trace()\` before merging recall interception changes. Operational logs (\`recall (stream, only): executing follow-up\`, \`recall (stream, mixed): stored pending result\`) should remain at \`info\` level for production monitoring. Implemented in pipeline.ts and stream/anthropic.ts.

<!-- lore:019e03e3-a2e8-7a27-8458-971ccc3be1f0 -->
* **Deprecated worker models persist in kv\_meta cache**: When Anthropic deprecates a model (e.g., \`claude-3-haiku-20240307\`), the OpenCode adapter's \`resolveWorkerModel()\` returns the stale entry from \`kv\_meta\` indefinitely, causing 404 errors on all worker calls (distillation, curation). Validation re-runs when the model landscape changes, but if no candidate passes (all deprecated), the stale entry isn't cleared. Fix: \`runValidation()\` now calls \`clearValidatedWorkerModel()\` when no candidate passes, and \`resolveWorkerModel()\` applies 24h TTL — entries older than 24h fall back to session/config model, allowing re-validation on next idle cycle.

<!-- lore:019e022f-7139-7f0d-bc07-7d37fd122a18 -->
* **Embedding backfill only runs in OpenCode adapter, not Pi**: Embedding backfill is now centralized in core \`runStartupBackfill()\` and called by both OpenCode and Pi adapters at startup. Pre-fix: Pi adapter had no backfill call, leaving all Pi-served projects with ~0% distillation embedding coverage (~1,333 rows). Fire-and-forget design hides failures — errors logged but not surfaced. Backfill depends on embedding provider availability; if no key is set, it silently no-ops. Coverage stats now logged to stderr at startup.

<!-- lore:019e0222-519d-7e99-9436-5a2695ddb6c2 -->
* **Silent Voyage-to-local migration for existing users without explicit config**: Users relying on implicit Voyage default (no \`.lore.json\`, just \`VOYAGE\_API\_KEY\` env var) will silently switch to local embeddings on upgrade. Embeddings auto-clear and re-embed, so no data loss, but they lose cloud provider quality (1024 dims → 384 dims). Risk is low for knowledge entries (negligible quality impact), but consider logging if \`VOYAGE\_API\_KEY\` is set but provider resolves to \`local\` — nudges power users they have an available option.

<!-- lore:019e0225-f018-78f2-9d1c-fe2a4c47ef9e -->
* **Exact keyword matching boost in RRF pipeline**: RRF (Reciprocal Rank Fusion) search combines vector and keyword results. Exact keyword matches now receive a boost via \`keywordBoost\` parameter during ranking. This improves relevance for queries with specific technical terms that may not embed well (e.g., library names, acronyms). Boost is applied as a multiplier to keyword match scores before RRF combination. Controlled via search config; default boost enables better recall on technical/domain-specific knowledge.

<!-- lore:019e0225-f01c-79fb-9f2b-c25a200ce6ed -->
* **Pattern extraction at distillation time**: New \`pattern-extract\` module identifies and extracts regex-based patterns (code snippets, commands, structured data) during knowledge distillation. Extracted patterns are indexed separately, enabling targeted queries like "show me all shell commands" or "find all SQL queries". Patterns are stored as metadata and used in search ranking. Reduces noise in semantic search by providing explicit structure to unstructured conversation knowledge.

<!-- lore:019e0381-826b-7cf5-9586-f13d2c767f41 -->
* **Pending recall cross-request injection via SessionState**: Case 2 (recall + other tools) stores \`{ toolUseId, position, result, timestamp }\` in \`sessionState.pendingRecall\` with 60s TTL. On next request, \`injectPendingRecall()\` finds the last assistant message, inserts the recall tool\_use block at the recorded position, injects tool\_result into the following user message, and removes recall from tools list. Client never sees recall — model receives results as natural conversation history. Lazy cleanup on session access checks TTL.

<!-- lore:019e0381-8268-77ac-9f9f-0362b425dcf8 -->
* **RecallAwareStreamAccumulator — selective block suppression with re-indexing**: RecallAwareStreamAccumulator — selective block suppression with re-indexing: Wraps \`StreamAccumulator\` to intercept SSE events and suppress recall tool\_use blocks from client stream. Detects \`content\_block\_start\` with name='recall', marks index as suppressed. Holds back \`message\_delta\`/\`message\_stop\` until caller decides Case 1 (continue) or Case 2 (forward). For continuation in Case 1, re-indexes content blocks: \`clientIndex = upstreamIndex - suppressedCount\`, starting at \`blockOffset\`. Zero overhead on non-recall streams.

<!-- lore:019e0238-28be-7892-98b8-07bdcda39d0c -->
* **Retroactive metric backfill (P3) for pre-v12 distillations**: \`backfillMetrics()\` in \`distillation.ts\` computes missing \`r\_compression\` and \`c\_norm\` for rows with NULL metrics. Loads source temporal messages via \`source\_ids\` JSON array, computes compression ratio and temporal clustering from timestamps/tokens, UPDATEs the row. Idempotent — only touches NULL values. Gracefully skips pruned source messages. Runs synchronously at startup so P2 recall weighting has full historical data (fills 2,863 pre-v12 rows).

Emits text block \`\[Searching memory...]\` to user, replacing the suppressed recall tool\_use. Continuation blocks re-index starting at \`clientBlockCount() + 1\` to maintain sequential indices. Implemented in pipeline.ts \`buildStreamingResponse()\`, line ~515.
