## Long-term Knowledge

### Architecture

<!-- lore:019e045f-273f-7c3d-b7a0-13684981368c -->
* **Gateway cost-aware layer 0 cap from MODEL\_SPECS pricing**: Gateway computes \`layer0Cap\` from model cache-read pricing in \`MODEL\_SPECS\` via \`computeLayer0Cap()\`. \`MODEL\_SPECS\` now includes \`cacheReadCost\` field (e.g., claude-opus-4: \`$15/M \* 0.1 = $1.50/M\`). When \`cfg.budget.maxLayer0Tokens\` is undefined, auto-computes cap from \`targetCacheReadCostPerTurn / cacheReadCost\`. For opus: \`$0.10 / $0.0000015 = 66,666 tokens\`. Ensures sessions escalate to layer 1 where \`tryFitStable\` controls the window. Without this, missing model cost data keeps sessions in layer 0 passthrough mode with 100% bust rate from front-trimming.

<!-- lore:019e044a-a65f-7b25-bbcf-2c73b017a05e -->
* **OpenCode uses compaction not front-trimming for context management**: OpenCode uses compaction not front-trimming for context management. When overflow detected, \`compaction.create()\` generates AI summary. \`filterCompacted()\` excludes messages before boundary. Tool output pruning clears beyond 40K tokens. Plugin hook delegates windowing to external systems like Lore gradient. However, between turns the message array can shift (compaction boundaries move, new compactions complete) causing different messages at the same array positions — this breaks absolute ID-based pinning but is handled by offset-from-end pins in gradient.

### Gotcha

<!-- lore:019e045f-9d5a-787f-83aa-79cc9e8a7021 -->
* **Dedup annotation changes can trigger legitimate cache busts**: Dedup annotation changes in \`deduplicateToolOutputs()\` alter tool output text, but content-based fingerprinting uses \`p.type\` ("tool") not output content for tool parts, so dedup changes don't directly trigger cache bust false positives. However, dedup changes token estimates which can cause \`tryFitStable\` pin-overflow when pinned window cost shifts. 15% hysteresis + high-water mark budget absorbs this variance — field data shows 0 pin-overflows over 63 turns. Dedup does change bytes sent to Anthropic (real cache miss), but this is expected and unavoidable.

<!-- lore:019e0466-4b94-7ec0-9cb8-aea9bfe6b7a2 -->
* **Gateway randomUUID generates unstable message IDs on every request**: Gateway randomUUID generates unstable message IDs causing 98% cache bust rates: \`gatewayMessagesToLore()\` originally used \`randomUUID()\` for message/part IDs, generating fresh UUIDs per request. Same conversation messages got different IDs each turn, breaking ID-based fingerprinting. Fixed with \`deterministicId()\` using SHA-256 of \`role + index + content\`. Combined with content-based fingerprinting \[\[019e045f-2777-79f6-b22a-13431690182d]], achieved 98% → ~2% bust rate reduction (6-7x cost savings for Opus sessions).

<!-- lore:019e0445-9ede-77ac-bf07-b4745feca17d -->
* **Layer 0 cache busts are expected with OpenCode front-trimming**: Layer 0 cache busts are expected with OpenCode front-trimming and distillation creation. At layer 0, gradient returns messages unchanged (passthrough), so front message changes affect Anthropic cache. Distillation prefix appearing for the first time causes a one-time bust that won't repeat. Sessions escalate to layer 1 when \`maxLayer0Tokens\` is set or model cost data provided, allowing \`tryFitStable\` to control the window and eliminate busts. Observed 1.6% bust rate (1/63 turns) with single unavoidable distillation bust; subsequent 50-turn sessions project ~2% rate.

<!-- lore:019e0466-4b98-7135-a8f9-ad674552c503 -->
* **OpenCode front-trims messages between turns causing pin-miss on absolute IDs**: OpenCode's message compaction shifts which messages appear at array boundaries between turns. Absolute ID pinning in \`tryFitStable\` fails when the pinned ID disappears. Fix: use offset-from-end pinning (\`pinnedIdx = messages.length - (pinnedRawCount + pinnedTotalCount)\`) instead of \`firstMessageID\`. Resilient to front-trimming since offset is relative to array tail.

<!-- lore:019e047e-71c9-7d95-80d0-157b4a3579c4 -->
* **Releases must use craft through the release workflow**: Do NOT create releases manually with \`gh release create\` or bump versions directly. This project uses a Release GitHub Actions workflow (triggered via workflow\_dispatch) that auto-detects version from changelog, creates a release branch, and opens a publish issue. Label the issue \`accepted\` to trigger the Publish workflow, which runs Craft to publish packages to npm. Manual version bumps bypass this and cause downstream failures.

<!-- lore:019e0483-8a1e-72e5-bf12-1224de1543b0 -->
* **TypeScript narrowing failure on generic parts with 'text' property**: \`LoreGenericPart\` has \`\[key: string]: unknown\` indexing, so \`"text" in p\` always passes but \`p.text\` remains \`unknown\`. Calling \`.slice()\` fails TypeScript strict mode. Fix: use \`isTextPart(p)\` or \`isToolPart(p)\` type guards before accessing \`.text\` as string, or cast as \`(p.text as string).slice()\`. Critical in \`computeContentBasedFingerprint()\` where fingerprint code accesses part text properties.

### Pattern

<!-- lore:019e045f-2777-79f6-b22a-13431690182d -->
* **Content-based cache fingerprinting matches Anthropic byte-identity behavior**: Gradient fingerprint uses \`role + text.slice(0, 30)\` instead of message IDs to match Anthropic's byte-identity caching. Format: \`${layer}:${role}:${text.slice(0,30)}\` per message. Part text must be explicitly typed as \`string\` before calling \`.slice()\` to avoid TypeScript errors. Reduces false positive busts from ~98% to ~7-9%, with remaining busts being genuine window changes from memory updates or dedup shifts.

<!-- lore:019e0453-b388-76cb-a258-1adf8c0e178f -->
* **Deterministic ID generation for gradient stability**: Gateway uses \`deterministicId()\` in \`temporal-adapter.ts\` to create stable IDs from SHA-256 of \`role + index + contentSnippet\`: first 8 hex chars of hash. Applied to all message IDs and part IDs so identical conversations produce identical prefix fingerprints across requests. Combined with content-based fingerprints (role + text snippets matching Anthropic's byte-identity cache behavior), achieves 0% bust rates in gateway sessions. Cache-bust detection fingerprint uses \`role:text.slice(0,60)\` per message instead of IDs.

<!-- lore:019e0436-e406-7438-989b-e54de73cd1f9 -->
* **Per-session ltmTokens isolation via optional sessionID parameter**: \`setLtmTokens(tokens, sessionID?)\` stores on \`SessionState.ltmTokens\` when sessionID provided, uses module-level fallback otherwise. \`getLtmTokens(sessionID?)\` reads session state first. \`transformInner()\` computes \`sessLtmTokens\` from session state before budget calculation. \`calibrate()\` reads \`state.ltmTokens\` for \`lastKnownLtm\`. All adapter call sites (OpenCode, Gateway, Pi) pass sessionID. Eliminates cross-session contamination where one session's LTM token count polluted another's budget calculation, causing \`usable\` swings of 20K+ tokens.

<!-- lore:019e0483-8a09-7626-80cd-f1f1f136b63a -->
* **Projected stable cache bust rates after hysteresis tuning**: With 15% hysteresis and offset-from-end pinning, achieved 1.6% cache bust rate (1/63 turns) in final testing. Single bust is unavoidable one-time distillation prefix appearance. Pin-overflow events eliminated (0 vs 4 with 10% hysteresis). Projected 50-turn opus session: ~2% bust rate, ~$5.30 cost. Stable enough for production release. Single busts from legitimate content changes (distillation, dedup) are expected and won't repeat in same conversation.

<!-- lore:019e047f-696e-7272-93ce-f75e6f8f5f90 -->
* **Release workflow uses GitHub CLI with automated changelog**: Bump version in \`package.json\`, commit, push, then \`gh release create v$VERSION --generate-notes\`. GitHub CLI auto-generates changelog from commit history. Always check CI passes before release. Use semantic versioning: patch for fixes, minor for features, major for breaking changes. No manual cross-reference updates needed—version is read from package.json.

<!-- lore:019e0442-6979-7219-a153-0d39b4b92879 -->
* **tryFit backward window scanning algorithm**: Implements backward window scanning for layers 1-3 using offset-from-end pinning (\`pinnedRawCount + pinnedTotalCount\`) resilient to host front-trimming. Pin validity uses high-water mark budget (\`max(pinnedBudget, currentBudget)\`) with 15% hysteresis to absorb overhead EMA drift and dedup changes. When pinned window fits within hysteresis, re-pins at current budget without rescan. 15% hysteresis eliminated pin-overflow events (0 in 63-turn benchmark). Projected: ~2% bust rate (one-time distillation), 0 overflows.
