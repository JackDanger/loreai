## Long-term Knowledge

### Architecture

<!-- lore:019e045f-273f-7c3d-b7a0-13684981368c -->
* **Gateway cost-aware layer 0 cap from MODEL\_SPECS pricing**: Gateway computes \`layer0Cap\` from model cache-read pricing in \`MODEL\_SPECS\` via \`computeLayer0Cap()\`. \`MODEL\_SPECS\` now includes \`cacheReadCost\` field (e.g., claude-opus-4: \`$15/M \* 0.1 = $1.50/M\`). When \`cfg.budget.maxLayer0Tokens\` is undefined, auto-computes cap from \`targetCacheReadCostPerTurn / cacheReadCost\`. For opus: \`$0.10 / $0.0000015 = 66,666 tokens\`. Ensures sessions escalate to layer 1 where \`tryFitStable\` controls the window. Without this, missing model cost data keeps sessions in layer 0 passthrough mode with 100% bust rate from front-trimming.

<!-- lore:019e044a-a65f-7b25-bbcf-2c73b017a05e -->
* **OpenCode uses compaction not front-trimming for context management**: OpenCode uses compaction not front-trimming for context management. When overflow detected, \`compaction.create()\` generates AI summary. \`filterCompacted()\` excludes messages before boundary. Tool output pruning clears beyond 40K tokens. Plugin hook delegates windowing to external systems like Lore gradient. However, between turns the message array can shift (compaction boundaries move, new compactions complete) causing different messages at the same array positions — this breaks absolute ID-based pinning but is handled by offset-from-end pins in gradient.

### Gotcha

<!-- lore:019e045f-9d5a-787f-83aa-79cc9e8a7021 -->
* **Dedup annotation changes can trigger legitimate cache busts**: \`deduplicateToolOutputs()\` changes tool \`state.output\` text, but the content-based fingerprint maps tool parts to \`p.type\` ("tool"), not their output text. So dedup changes do NOT directly cause cache bust false positives in the fingerprint. However, dedup changes token estimates which can cause \`tryFitStable\` pin-overflow when the pinned window's measured cost shifts. The 15% hysteresis + high-water mark budget absorbs most of this variance. Dedup annotations also change the actual bytes sent to Anthropic, which IS a real cache miss.

<!-- lore:019e0466-4b94-7ec0-9cb8-aea9bfe6b7a2 -->
* **Gateway randomUUID generates unstable message IDs on every request**: \`gatewayMessagesToLore()\` in \`temporal-adapter.ts\` originally used \`randomUUID()\` for message and part IDs, generating fresh UUIDs on every HTTP request. Since the same conversation messages get different IDs each turn, any ID-based fingerprinting (cache-bust detection, pin lookups) breaks 100% of the time. Fix: replaced with \`deterministicId()\` using SHA-256 of \`role + index + content\` so identical messages always produce identical IDs across requests.

<!-- lore:019e0445-9ede-77ac-bf07-b4745feca17d -->
* **Layer 0 cache busts are expected with OpenCode front-trimming**: Layer 0 cache busts are expected behavior when host applications change front messages through compaction boundaries, message regeneration, or other management. At layer 0, gradient returns messages unchanged (passthrough), so first message ID changes affect Anthropic cache. Sessions stay at layer 0 when \`input.model.cost?.cache?.read\` is missing. Fix: ensure sessions escalate to layer 1 by setting \`maxLayer0Tokens\` or providing model cost data where \`tryFitStable\` can control the window.

<!-- lore:019e0466-4b98-7135-a8f9-ad674552c503 -->
* **OpenCode front-trims messages between turns causing pin-miss on absolute IDs**: OpenCode's \`filterCompacted()\` and internal message management can change which messages appear at the front of the array between turns (compaction boundaries shift, pruned outputs). Absolute message ID pinning in \`tryFitStable\` fails because the pinned ID disappears from the array (\`pinnedIdx === -1\`). Fix: use offset-from-end (\`pinnedRawCount + pinnedTotalCount\`) instead of \`firstMessageID\`. Compute \`pinnedIdx = messages.length - (pinnedRawCount + newMessages)\`. Resilient to front-trimming since offset is relative to the tail.

### Pattern

<!-- lore:019e045f-2777-79f6-b22a-13431690182d -->
* **Content-based cache fingerprinting matches Anthropic byte-identity behavior**: Gradient fingerprint uses \`role + text.slice(0, 30)\` instead of message IDs to match Anthropic's actual byte-prefix caching behavior. Function \`computeContentBasedFingerprint()\` extracts content snippets from first 5 messages in layer-appropriate format. Format: \`${layer}:${role}:${text.slice(0,30)}\` per message prefix. Reduces false positive busts from ~98% to ~7-9%, with remaining busts being genuine window content changes from memory context updates, distillation completions, or dedup annotation shifts. More accurate than ID-based detection since it reflects what actually gets sent to the API.

<!-- lore:019e0453-b388-76cb-a258-1adf8c0e178f -->
* **Deterministic ID generation for gradient stability**: Gateway uses \`deterministicId()\` in \`temporal-adapter.ts\` to create stable IDs from SHA-256 of \`role + index + contentSnippet\`: first 8 hex chars of hash. Applied to all message IDs and part IDs so identical conversations produce identical prefix fingerprints across requests. Combined with content-based fingerprints (role + text snippets matching Anthropic's byte-identity cache behavior), achieves 0% bust rates in gateway sessions. Cache-bust detection fingerprint uses \`role:text.slice(0,60)\` per message instead of IDs.

<!-- lore:019e0436-e406-7438-989b-e54de73cd1f9 -->
* **Per-session ltmTokens isolation via optional sessionID parameter**: \`setLtmTokens(tokens, sessionID?)\` stores on \`SessionState.ltmTokens\` when sessionID provided, uses module-level fallback otherwise. \`getLtmTokens(sessionID?)\` reads session state first. \`transformInner()\` computes \`sessLtmTokens\` from session state before budget calculation. \`calibrate()\` reads \`state.ltmTokens\` for \`lastKnownLtm\`. All adapter call sites (OpenCode, Gateway, Pi) pass sessionID. Eliminates cross-session contamination where one session's LTM token count polluted another's budget calculation, causing \`usable\` swings of 20K+ tokens.

<!-- lore:019e0442-6979-7219-a153-0d39b4b92879 -->
* **tryFit backward window scanning algorithm**: \`tryFit()\` implements backward window scanning for layers 1-3. Walks backward from current turn accumulating tokens until budget exhausted. \`tryFitStable()\` uses offset-from-end pinning (\`pinnedRawCount + pinnedTotalCount\`) instead of absolute message IDs, resilient to host front-trimming. Pin validity uses high-water mark budget (\`max(pinnedBudget, currentBudget)\`) with 15% hysteresis to absorb overhead EMA drift and dedup token estimate changes. When pinned window overflows old hysteresis but fits current \`rawBudget\`, re-pins at current budget without full rescan. The 15% hysteresis (up from 10%) eliminates pin-overflow events that were causing unnecessary rescans.
