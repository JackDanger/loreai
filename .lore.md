### Pattern

<!-- lore:019e0d91-d623-7753-9867-8e728c6fac37 -->
* **Auto-detected conversation TTL based on cold-cache turn progression**: Auto-upgrade conversation TTL by turn count: TTL selection logic in ModelSpec.conversationTTL: 5m for turns 1-3, 1h for turn 4+. Gateway SessionState tracks \`coldCacheTurns\` incremented on cache misses. \`getTTLBasedOnTurns()\` in pipeline.ts determines TTL before AnthropicCacheOptions construction. Prevents unnecessary long-term cache on quick interactions while preserving context for sustained conversations.

<!-- lore:019e0d92-4bd6-7eaf-8fc2-53ce15cd7d97 -->
* **Auto-sync idle resume threshold with 1h TTL conversation context**: Auto-sync idle resume with 1h TTL: When 1h conversation TTL is active (\`session.contextTTLMinutes === 60\`), auto-sync \`idleResumeMinutes\` to ~50-55min before idle check. Prevents idle resume disconnect when conversation context would refresh within resume window, reducing cache busts and reconnection overhead on long sessions.

<!-- lore:019e0d89-1f32-764a-ba85-e5b7018ac3fe -->
* **Bust rate feedback loop in postResponse calibration**: Bust rate feedback loop in postResponse: Updates \`SessionState.gradient.burstFrequencyEMA\` via EMA (alpha ~0.3) in pipeline.ts postResponse() after cache analytics. Tracks write:read ratio and break frequency per session for adaptive cap tightening on frequent breakers. Call \`calibrate()\` callback with recent metrics to update EMA for next turn's context cap calculation.

<!-- lore:019e0d8c-cb2f-7834-8dba-670edf682fe4 -->
* **Cache bust metric emission for observability and feedback**: Cache bust metric emission: Emit \`emitCacheBustMetric(sessionId, bustCause, context)\` in postResponse() after categorizing bust cause. Captures sessionId, bustCause enum (CONTEXT\_OVERFLOW, TOKEN\_LIMIT\_EXCEEDED, EXPLICIT\_REQUEST, UNKNOWN), model, session duration, active layers. Uses Sentry distribution/count metrics for dashboards tracking bust patterns. Data feeds back into calibrate() EMA feedback loop.

<!-- lore:019e0d8a-0c17-7807-8c9c-854650d88535 -->
* **Cache write cost per model in ModelSpec — derived from pricing + TTL**: ModelSpec cache write cost calculation: ModelSpec.cacheWriteCost pre-computed as \`cache\_write\_price \* base\_price / input\_price\` per model+TTL. Defaults: 1.25x (5m TTL), 2x (1h TTL). Exposed via \`getModelSpec()\` for context caps and curation decisions. Gateway's cost-aware worker defaults use this to select sonnet-4 when session model cacheWriteCost >= 2x. Includes \`conversationTTL\` field for auto-detection based on cold-cache turn count.

<!-- lore:019e0d8b-e42b-7cb5-a89c-9ebe0af8b4bb -->
* **Cost-aware curation frequency scaling by model tier**: Cost-aware curation frequency by model tier: Curation frequency scales by model cost via \`afterTurns \*= getModelSpec(model).cacheWriteCost\` in pipeline.ts. Expensive models (Opus ~2x) curate less frequently. LTM diff threshold scales inversely: \`ltmDiffThreshold /= cacheWriteCost\` so expensive models need larger changes to trigger updates. Prevents costly model churn while maintaining quality for cheaper models.

Uses \`getModelSpec()\` to check session model cost. Resolution chain: explicit config → session model → cost-aware default → host fallback. Prevents expensive models from running background tasks while preserving quality for cheaper session models.

* **Human-readable cache invalidation reasons from JSON paths**: Human-readable cache invalidation reasons: Map technical JSON paths to readable reasons in \`inferDivergenceReason()\`: \`system\[1].text\` → "LTM knowledge updated", \`messages\[N]\` at conversation end → "new message", mid-conversation → "message modified". Bare \`system\` → "system prompt changed". Context: system\[0] is host prompt, system\[1] is LTM injection in array format. Requires message count to distinguish append vs edit patterns.

<!-- lore:019e0da8-7f29-7f15-9438-480825db5444 -->
* **SSE stream parsing with ReadableStream controller lifecycle**: SSE response streaming uses ReadableStream with \`parseSSEStream()\` helper that parses chunks and calls \`controller.enqueue()\` for each event. Controller lifecycle: start→parse upstream→enqueue events→close on completion. Error handling wraps parsing with try/catch, finally block ensures cleanup. Key insight: \`parseSSEStream\` callback both enqueues parsed events AND closes controller when upstream ends, requiring careful coordination with error cleanup paths to avoid double-close.

<!-- lore:019e0da9-69ef-79d4-87e6-6206f11dcfb0 -->
* **SSE streaming pipeline recall interception architecture**: Pipeline streaming works in phases: 1) Parse upstream SSE via parseSSEStream() async generator, forward through accumulator.processEvent() 2) Post-stream recall check - if recall tool detected, execute query and emit synthetic marker 3) Branch by tool mix: A) mixed tools = forward held events + close, B) recall-only = make follow-up API call and pipe continuation stream with re-indexed content blocks, C) no recall = normal close. All controller.enqueue() calls use encoder.encode(sseText). Mutually exclusive branches via if/return gating prevent double-close.
