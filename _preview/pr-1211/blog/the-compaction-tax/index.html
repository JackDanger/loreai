The compaction tax

We benchmarked what memory actually costs you, in dollars and in focus

July 11, 2026

Two posts ago we argued that memory is not enough and that you never have to remember to remember. Here are the receipts. A real coding-agent benchmark, and the two taxes it puts a number on.

We have made two arguments on this blog. In Why memory is not enough we said a store on the sidelines cannot save the session that is overflowing right now, because compaction runs whether or not it is about to drop something you still need. In How Lore remembers, forgets, and changes its mind we said the capture has to be automatic, because memory you have to reach for is memory you will forget to reach for.

Arguments are cheap. So we built a benchmark and ran it, and this post is the receipts.

There are two taxes hiding in how agents handle memory today. One the machine charges you automatically, in dollars and in lost context. One you charge yourself, in attention. We can now put a number on both.

How we measured it

The full protocol is written up and versioned in the repo (see the methodology), and you can reproduce the core of it with lore eval. The short version:

Every run drives a real OpenCode agent, turn by turn, against a real model. Nothing is replayed. We probe on four arbitrary values stated once in passing, an order status, a sales channel, a region, a warehouse, none of which can be guessed by reading the code. If they show up in the file the agent writes at the end, something carried them. The scorer is a mechanical check for those four values, so there is no model grading its own homework. We ran it on a cheaper, everyday model (MiniMax-M3, capable but not a frontier model) and on a frontier flagship (Claude Sonnet-5), because that split is exactly where the approaches part ways.

The first tax: compaction

Put the agent in one long session and keep working until the window fills. Every major client does the same thing at that point: it compacts, crushing the older turns into a lossy summary and dropping the originals. That is the moment the facts you mentioned an hour ago quietly fall out.

Here is that session, sixteen turns, with the four facts mentioned once near the start:

retentioncompactionstotal tokensstepspeak context
no memory33%35.03M55194K
Lore100%04.44M33150K

Two things are happening at once. The no-memory agent compacts three times and loses two of the three probes to it. Lore never compacts, because it manages the window on every turn instead of waiting for it to overflow, and it keeps all four facts.

And it does this for less money, not more. Compaction is not free: it busts the prompt cache and rewrites the prefix, and rewritten cache is the expensive kind of token. The no-memory run wrote noticeably more to cache and burned 13% more tokens across 67% more steps to end up in a worse place. This is the part people miss when they treat compaction as a free built-in. You are paying for it every time it fires, and it is charging you to forget.

The loss is also quiet, which is what makes it easy to miss. Independent research on context rot (Chroma’s report, and the contextrot tool that measures it on your own sessions) shows models degrade as their window fills. When we ran contextrot over these same benchmark sessions, one thing stood out: even at full context the no-memory run threw no visible trouble, no failed edits, no retries, no re-reads, while our probe shows it had already dropped most of the facts. Compaction does not announce itself. The agent keeps going, confident, having quietly forgotten.

The second tax: remembering to remember

Now the cross-session case, which is what most people mean by memory: something said in one session, needed in a later, separate one. This is the job a memory store is built for, so we put Lore next to two good ones: mem0 (cloud) and mnemonic (a local SQLite store, the closest analogue to how Lore keeps things on your own disk).

On the frontier model, everyone with memory got it right:

Sonnet-5, cross-sessionretention
no memory0%
Lore100%
mnemonic100%
mem0100%

That is the honest headline: a good store works, and on a capable model it works reliably. We are not going to pretend otherwise.

Retention is not the whole story, though. Even here Lore reached 100% with the fewest tool calls of any arm, roughly 45% fewer than the no-memory run (13 versus 24), and the smallest bill, which we come to next. On a frontier model the win is efficiency: the same answer for less work, because managing the window in the loop beats compacting and re-deriving it.

The difference shows up on the cheaper, everyday model, the kind you actually run at scale:

MiniMax-M3, cross-sessionretention
no memory0%
Lore85%
mnemonic33%
mem017%

The gap is not about storage quality. It is about who has to remember to use it. A store the agent drives only works if the agent decides, on its own, to save the offhand value when it is mentioned, and then to go looking for it later. An everyday model does not reliably do either. In the runs where the competitors missed, they had dutifully stored the project’s coding conventions and skipped the four incidental values entirely, which is exactly the kind of fact nobody flags as important and everybody needs later. Lore does not ask the model to decide. It captures as the conversation happens and surfaces what is relevant on its own, so it does not hang on the model’s discipline on any given turn.

That is the attention tax made concrete. Every store the agent reaches for is a store that depends on remembering to reach, and the further a model sits from the frontier, the less reliably it reaches.

What it costs, honestly

Cost is where memory tools get quiet, so here is ours out loud. On the cross-session run with Sonnet:

answering costmemory backendthird-party cost
no memory$4.03nonenone
mnemonic$3.97Gemini (embeddings + extraction)metered, external
mem0$4.10mem0 cloudmetered, external
Lore$3.34your own model, on your own billnone

Lore’s answering call is the cheapest of the four, 17% below the no-memory run, for the same reason as before: it does not pay the compaction tax. Its memory work, the background distillation, runs on the same model you are already using, so it lands on the same bill and you can predict it from your own usage. The alternatives add a separate, metered service on top: Gemini for one, a hosted cloud for the other. That is a real cost, and it is one you cannot read off your model spend. We did not bill it here because we could not, which is rather the point.

We hold ourselves to a simple bar: same price with better results, or cheaper at a tie. On the cheaper model Lore is far more reliable at a transparent cost. On the frontier model it ties on retention with pricing you can actually account for.

What we are not claiming

The benchmark has limits and we would rather say them than have you find them. We could not measure the competitors’ third-party backend costs, so the totals above are their visible cost plus an unknown, not a full accounting. The reliability gap on cheaper models is real, but on frontier models it closes, and we showed that. One M3 Lore run out of five dropped to a single probe, genuine run-to-run variance on a cheaper model that we kept in the average rather than trim. And we could not evaluate Lore on an anonymous free endpoint, because its background worker needs a real key to run.

All of it, the tasks, the scorer, the raw per-run numbers, is in the repo. Run lore eval and check our work.

Back to the one question

We ended the first post with a question to ask anything sold to you as agent memory: what happens at 200K tokens? Now there is a number attached. What happens is the client compacts, you lose a third of what mattered, and you pay more to get there. A store on the sidelines does not change that, however good the store is. Managing the window in the loop does, and it is cheaper, and you never have to think about it.

Memory is table stakes. The compaction tax is the bill nobody was reading.
