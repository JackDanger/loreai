Let your past decisions review the code

The reviewer can't remember every rule the team ever agreed on. Something should.

July 20, 2026

"We read the code" and "automate the review" are usually framed as opposites. The part worth automating isn't judgment. It's memory: catching the change that contradicts a decision nobody on the PR happens to recall.

There’s a real argument happening right now about what review is for. One camp says the point of writing code with agents is that a human still reads every line before it ships. The other says manual review is a bottleneck we should be automating away. We land on the automation side, but we think the way the debate is usually drawn misses the part that actually matters.

Because most of what “read the code” is defending is worth defending. Taste, judgment, whether the change is the right change at all, whether it fits where the codebase is going. A machine is not about to take that over, and we are not trying to. What we want to talk about is the part of review that has nothing to do with judgment and everything to do with memory, because that part is where humans are weakest and where a system can genuinely help.

The decision nobody on the PR remembers

Here is the failure we keep seeing. Six months ago the team decided, for good reasons, not to do a certain thing. Maybe you don’t mock the database in integration tests, because the last time someone did it hid a migration bug for a week. Maybe deploys only ever go out from the release branch. The decision was made, it was sound, and everyone moved on.

Then a pull request quietly reintroduces exactly that thing. Not out of carelessness. The author simply wasn’t in the room when the decision was made, or was, and it was a year ago. The one reviewer who would catch it in a heartbeat isn’t on this PR. So it merges, and the lesson gets relearned the expensive way.

Notice what kind of failure this is. Nobody exercised bad judgment. The change looks reasonable in isolation. What was missing was not a smarter reviewer. It was a reviewer who happened to remember one specific decision at the exact moment it was being contradicted. That is not a judgment problem. It is a recall problem wearing a review problem’s clothes.

You already wrote the rule down

If your team uses Lore, that decision is not lost. It’s sitting in .lore.md, a version-controlled file of the decisions, patterns, and gotchas the team has accumulated, exported as plain Markdown and reviewed in pull requests like any other file. We’ve written before about why that knowledge lives in text rather than baked into a model: text is something you can read, diff, carry to the next model, and correct a line at a time.

But a written rule that only a human can act on has the same weakness as the human. It only helps if someone reads it at the right moment. The file knows the team decided not to mock the database. The question is whether that knowledge can do anything on its own, or whether it just waits, again, for a person to remember it’s there.

Memory that can intervene

This is the shift we care about. A memory store that only answers when asked is passive. It recalls; it never intervenes. The more useful thing is knowledge that shows up on its own at the moment it’s relevant, which for a team decision is the moment a change is about to contradict it.

So we built a check that reads the diff against the rules. On every pull request, Lore’s semantic linter takes the changes and the invariants from .lore.md and asks, for the pairs that look related, a narrow question: does this change contradict this documented decision? When it thinks the answer is yes, it says so, as an annotation on the PR, next to the line.

It is a judge, not a rule engine, and that distinction is the whole point. “Never mock the database in integration tests” is not a pattern you can match with a regex. It’s a meaning. You can only check it by understanding what the change does and what the rule intends, which is exactly the kind of reading we used to need a person for. The tradeoff is that a judge produces suspicions, not verdicts. It can be wrong. We designed around that rather than pretending otherwise.

Why it doesn’t fail your build

If you’re in the “read the code” camp, the thing you’re rightly afraid of is a noisy automated check that blocks work on a bad guess. We share that fear, so the linter ships advisory only. It never fails a build. It leaves a note and a human decides. A probabilistic check that can break CI is just a probabilistic build-breaker, and it would teach everyone to ignore it inside a week.

Cost gets the same restraint. A large PR against a full knowledge base is thousands of change-and-rule pairs, and calling a model on each one would be slow and expensive. Almost all of those pairs are unrelated, so we throw them out for free first, with a local embedding pass, and only spend a model call on the handful that actually look like they might conflict. The model is the last stage of a funnel, not the first.

There’s an enforcement ladder above advisory, for teams that watch the false-positive rate on their own repo and decide a particular rule is worth gating on, with an override path for the author who knows better. But the default is deliberately the humble end of that ladder. The system earns trust before it’s given teeth.

The narrow claim

We’re not claiming the reviewer goes away. The judgment part of review, the part the “we read the code” camp is protecting, is not what this replaces, and we’d be suspicious of anyone selling that. What we’re claiming is narrower and, we think, harder to argue with: no human reviewer can hold every decision the team has ever made in their head and recall the right one at the right moment. That specific job, remembering, is one a system does better, and handing it over doesn’t cost you any of the judgment you were worried about losing. It gives the reviewer back the attention to spend on the parts that actually need a person.

If you’re skeptical of automating review, this is the version we’d ask you to try. Not a bot that approves your PRs. A memory that speaks up when you’re about to undo a decision you’d forgotten you made, and otherwise stays quiet.

Because a check that reads every diff against your team’s rules is a lot to take on faith, the code that decides what it flags is open to read. Lore is Fair Source (FSL-1.1-Apache-2.0), and it turns into Apache 2.0 on a timer. A rule you can watch enforce itself is worth more than one you have to remember.
