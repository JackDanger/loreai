# Quality Constitution — Lore

## 1. Project Identity

Lore is a **three-tier memory architecture** for AI coding agents. It intercepts LLM API calls (as a transparent proxy or native plugin), stores raw conversation turns, distills them into compressed summaries, and extracts long-term knowledge entries that persist across sessions. The system ships as four packages in a TypeScript monorepo:

- **`@loreai/core`** — shared memory engine: SQLite (WAL + FTS5), distillation, knowledge (LTM), recall, gradient context management, prompts.
- **`@loreai/gateway`** — transparent LLM proxy + CLI (`lore`): intercepts API calls across Anthropic/OpenAI/Responses protocols, manages context windows, serves a web dashboard, runs cache warming.
- **`@loreai/opencode`** — OpenCode plugin adapter.
- **`@loreai/pi`** — Pi coding-agent extension adapter.

**Runtime:** Bun (development/tests) and Node.js >= 22.5 (production npm bundles). **Database:** SQLite with WAL mode, FTS5 full-text search.

### What "correct" means for this project

A correct Lore deployment:

1. **Never loses user data.** Temporal messages stored atomically; distillation store+mark in a single SQLite transaction; knowledge entries survive process crashes via immediate or 30s-periodic persistence.
2. **Produces faithful memory.** Distillation summaries are bounded by expansion guards; knowledge entries are deduplicated; recall search returns relevant results via vector+FTS5 hybrid scoring.
3. **Is protocol-transparent.** Clients communicating via Anthropic, OpenAI Chat Completions, or OpenAI Responses API receive equivalent behavior: LTM injection, tool handling, streaming error propagation, and recall interception all work identically regardless of wire protocol.
4. **Manages context economically.** The gradient context manager selects the lowest-cost compression layer that fits the model's context window, preserves prompt cache byte-identity across tool-call chains, and never oscillates between layers.
5. **Degrades gracefully.** Embedding provider failures fall back (local ONNX → remote Voyage/OpenAI), FTS5 errors fall back to LIKE queries, session identification falls through three tiers, port binding retries through four candidates.

## 2. Quality Priorities (ranked)

| Rank | Priority | Rationale |
|------|----------|-----------|
| 1 | **Data Integrity** | Temporal messages, distillations, and knowledge entries are the user's memory. Loss or corruption is unrecoverable. Atomic transactions, crash-safe persistence, and dedup guards are non-negotiable. |
| 2 | **Protocol Correctness** | The gateway translates between three LLM API protocols. A tool_result dropped in translation, a system prompt not injected, or a streaming error swallowed silently breaks the agent's ability to function. |
| 3 | **Resource Management** | Long-running gateways serve many sessions over days/weeks. Unbounded Maps, leaked p-limit instances, unevicted session state, and permanently-tripped circuit breakers degrade the process over time. |
| 4 | **Cache Optimization** | Anthropic prompt caching is the primary cost lever. Cache busts from layer oscillation, prefix non-determinism, or unnecessary distillation refreshes cost $1-2 per incident at 500K context. |
| 5 | **Cross-Protocol Parity** | All three protocol paths (Anthropic, OpenAI, Responses) must inject LTM, handle recall markers, propagate streaming errors, and preserve tool blocks equivalently. Asymmetric behavior is a defect. |

## 3. Fitness-to-Purpose Scenarios

### FTP-1: Session state survives process crash

**Situation:** The gateway process is killed (SIGKILL, OOM) mid-session. The user restarts the gateway and resumes their session.

**Correct behavior:** Gradient EMAs, cost data, and `lastLayer` are recoverable to within 30 seconds of the last checkpoint (the idle scheduler's periodic flush interval). `forceMinLayer` — the critical "prompt too long" recovery flag — is recoverable immediately (persisted on every mutation). Session identity (fingerprint, header session ID) is recoverable immediately.

**Verification:** After simulated SIGKILL, `loadSessionTracking()` returns `lastTurnAt > 0` with `lastLayer`, `lastKnownInput` values matching the last periodic save. `loadForceMinLayer()` returns the value set by the most recent overflow handler.

**What breaks if violated:** Session re-calibrates from scratch on restart, wasting an entire turn's cache budget (~$1.88 at 500K Sonnet). If `forceMinLayer` is lost, the "prompt too long" error repeats indefinitely.

### FTP-2: Tool results round-trip through all protocol translators

**Situation:** An agent session uses tools (file reads, code execution). The conversation contains `tool_use` blocks from the assistant and `tool_result` blocks from the user, interleaved across multiple turns. The client talks OpenAI Chat Completions; the upstream is Anthropic.

**Correct behavior:** `tool_result` content blocks in the gateway's internal representation are translated to `role: "tool"` messages with `tool_call_id` in OpenAI format, and to `tool_result` content blocks in Anthropic format, preserving the tool use ID linkage in both directions. No tool results are silently dropped.

**Verification:** A multi-turn tool-using conversation round-trips through `parseOpenAIRequest` → gateway internal → `buildAnthropicRequest` → response → `buildOpenAIResponse` with all tool results present and correctly linked by tool call ID.

**What breaks if violated:** The LLM receives a conversation with tool_use blocks but no corresponding results, causing it to re-issue tool calls or hallucinate results. Exploration finding #9 identified this gap in the OpenAI translator.

### FTP-3: Embedding provider failure is transient, not permanent

**Situation:** The local ONNX embedding provider hits a transient OOM (error codes `287180544`, `144786472`) on a particularly large batch during backfill. The next recall query arrives 5 minutes later with a normal-sized input.

**Correct behavior:** The system falls back to the remote provider (Voyage/OpenAI) for the failed batch, but retries the local provider on subsequent requests. The local provider is not permanently disabled by a single transient failure.

**Verification:** After a single `LocalProviderUnavailableError`, `isAvailable()` eventually returns `true` again (either on a timer or on next request). The `localProviderKnownBroken` flag is recoverable.

**What breaks if violated:** A single OOM permanently degrades to higher-latency, higher-cost remote embeddings for the entire process lifetime. Every recall query pays API costs and adds ~200ms latency. Exploration finding #11 identified this as a current gap — `cachedProvider` at `embedding.ts:700` is permanently set to the remote provider.

### FTP-4: Long-running gateway doesn't leak memory

**Situation:** A hosted gateway serves 200+ sessions over 72 hours. Sessions come and go — some last 5 minutes, some last hours.

**Correct behavior:** The `sessionStates` Map (`gradient.ts:304`), per-session p-limit instances (`session-limiter.ts:17`), `globalHeaderValues` (`session.ts:436`), `stableLtmCache` (`pipeline.ts:327`), and `sessions` Map (`pipeline.ts:280`) all have eviction policies. Dead sessions are cleaned up when their survival probability drops below threshold or after a configurable TTL.

**Verification:** After 200 sessions where 180 have been idle for >2 hours, memory usage is bounded. The session-related Maps contain ≤ 30 active entries (20 active + 10 recently-idle buffer).

**What breaks if violated:** Gateway RSS grows monotonically. At ~50KB per session state (gradient + warming + analytics + cost tracking), 200 sessions consume ~10MB — manageable. But at 10,000 sessions (hosted mode over weeks), this becomes 500MB of dead state. Exploration finding #7.

### FTP-5: Recall queries with special characters produce correct results

**Situation:** A user's recall query contains double quotes, parentheses, or unicode characters. The agent calls the recall tool with `query: 'how to use "async"'`.

**Correct behavior:** The recall marker regex at `recall.ts:109` correctly captures the full query including embedded quotes. The FTS5 query at `search.ts:162` escapes special characters that would break FTS5 syntax. The marker round-trips through expansion (`expandRecallMarkers`) without truncation.

**Verification:** `parseRecallMarker('📚 Searching all archives for "how to use \\"async\\""…')` returns the full query. `ftsQuery('error "file not found"')` produces a valid FTS5 MATCH expression.

**What breaks if violated:** Recall searches for truncated queries, returning irrelevant or no results. The agent provides incorrect context to the user. Exploration finding #6 identified the lazy quantifier truncation bug in `MARKER_REGEX`.

---

### FTP-6: LTM entries are never duplicated in system prompt

**Situation:** A knowledge entry has `project_id = X` and `cross_project = 1`. The `forSession()` function runs for a session in project X.

**Correct behavior:** The entry appears in exactly one of the two pools (project-specific or cross-project), not both. The final merged pool passed to `formatKnowledge()` contains no duplicate entry IDs.

**Verification:** For any entry with `cross_project = 1`, the project query (`WHERE project_id = ? AND cross_project = 0`) excludes it. The cross-project query (`WHERE project_id IS NULL OR cross_project = 1`) includes it exactly once. The greedy packing loop in `formatKnowledge()` at `prompt.ts:543-548` never emits the same entry twice.

**What breaks if violated:** The agent sees duplicate knowledge entries, wasting token budget and potentially confusing the model with redundant context. Exploration finding #13 noted this edge case in the non-preference path.

---

### FTP-7: Streaming errors reach clients in all protocols

**Situation:** The Anthropic upstream sends an `error` event during streaming. The client is connected via OpenAI Chat Completions protocol.

**Correct behavior:** The OpenAI streaming translator at `stream/openai.ts` detects the Anthropic error event, emits an OpenAI-format error chunk with the error reason, and closes the stream with `data: [DONE]`. The client receives the error details, not a silent hang or empty completion.

**Verification:** An Anthropic `error` event with `{"type":"error","error":{"type":"overloaded_error","message":"..."}}` is translated to an OpenAI streaming chunk with `finish_reason: "error"` and the error message. The Responses API translator (`stream/openai-responses.ts`) emits `response.failed` with error details.

**What breaks if violated:** Clients hang waiting for more tokens that never arrive, or receive an empty completion with `finish_reason: "stop"` — indistinguishable from a normal empty response. Exploration finding #14 identified missing error event handling in the OpenAI translator.

---

### FTP-8: Multi-model sessions get correct context limits

**Situation:** Two concurrent sessions use different models: Session A uses Claude Sonnet (200K context) and Session B uses Claude Haiku (200K context with different output reservation). Both sessions are active simultaneously in hosted mode.

**Correct behavior:** Each session's gradient transform uses the context limit and output reservation from its own model, not from the last `setModelLimits()` call by any session. The `contextLimit`, `outputReserved`, and `maxLayer0Tokens` module-level globals at `gradient.ts:37-38,87` are either per-session or last-writer-wins is safe because the values converge.

**Verification:** After Session A calls `setModelLimits(200000, 32000)` and Session B calls `setModelLimits(200000, 16000)`, Session A's next `transform()` uses `outputReserved = 16000` (Session B's value). If Session A's model actually requires 32000 output tokens, the `maxInput` calculation underestimates available space.

**What breaks if violated:** A session with a smaller output reservation gets a larger-than-safe `maxInput`, potentially causing "prompt too long" errors. A session with a larger output reservation gets an unnecessarily compressed context window, wasting cache budget. Exploration finding #2 identified this module-level global contamination.

## 4. Coverage Theater Prevention

The following patterns **look like** adequate test coverage but provide false confidence:

### 4.1 Testing only the Anthropic protocol path

The gateway supports three wire protocols. If tests only exercise the Anthropic path (which is the most mature), they miss:
- Tool result blocks silently dropped in OpenAI translation (`translate/openai.ts:523-527`)
- Streaming error events unhandled in the OpenAI translator (`stream/openai.ts`)
- System prompt injection missing in non-Anthropic upstream paths (historically present, now fixed)

**Requirement:** Every behavioral contract that involves protocol translation MUST have test cases for all three protocols: Anthropic, OpenAI Chat Completions, and OpenAI Responses.

### 4.2 Testing only happy-path tool calls

Tool handling has complex edge cases:
- Orphaned `tool_result` without matching `tool_use` (recall marker expansion creates synthetic pairs)
- Mixed tool calls (recall + other tools in the same response)
- Tool call chains during gradient compression (current turn protection)
- `JSON.parse` on malformed tool arguments (`translate/openai.ts:166,205`)

**Requirement:** Tool handling tests MUST include malformed inputs, mixed tool responses, and orphaned result blocks.

### 4.3 Testing distillation with small segments only

The `splitSegments()` infinite recursion bug (now fixed with `messages.length <= 1` guard at `distillation.ts:153`) was triggered by a single message exceeding `maxSegmentTokens`. Tests using small, well-formed segments never hit this path.

**Requirement:** Distillation tests MUST include oversized single messages (>16384 tokens estimated), empty segments, and segments at exact boundary sizes.

### 4.4 Testing LTM scoring with embeddings always available

When `embedding.isAvailable()` returns `false`, `forSession()` falls through to FTS5-only scoring. When FTS5 fails (syntax error from special characters), it falls through to LIKE queries. Tests that always have embeddings available miss both fallback paths.

**Requirement:** LTM scoring tests MUST include scenarios where embeddings are unavailable and where FTS5 queries fail.

### 4.5 Testing gradient layers in isolation

The gradient system's correctness depends on cross-layer interactions: the sticky layer guard, `forceMinLayer` persistence and consumption, calibrated vs uncalibrated safety multipliers, and the post-idle compact flag. Testing each layer's `tryFit()` in isolation misses these state transitions.

**Requirement:** Gradient tests MUST include multi-turn sequences that cross layer boundaries, idle-resume transitions, and `forceMinLayer` set-consume-reset cycles.

## 5. Quality Bar

### Minimum standard for this project

1. **Every behavioral contract** in `CONTRACTS.md` has at least one test that exercises both the invariant and the violation condition. The test name references the contract ID.

2. **Protocol parity** is enforced: any feature that works for Anthropic protocol MUST work for OpenAI and Responses protocols, verified by parameterized tests across all three.

3. **Crash recovery** is tested: state persistence tests simulate process crashes (clear in-memory state, reload from DB) and verify that critical fields (`forceMinLayer`, session identity, cost data) are recoverable within their documented tolerance.

4. **Resource bounds** are asserted: long-running simulation tests verify that session-scoped Maps, caches, and limiters are bounded. Any new `Map<string, ...>` keyed by session ID must have a documented eviction policy.

5. **Fallback paths are exercised**: every fallback chain (embedding local→remote, FTS5→LIKE, port probing, session tier fallback) has tests that force each fallback level and verify the degraded behavior is correct.

6. **No silent data loss**: any code path that drops, truncates, or transforms user data (tool results, recall queries, distillation content) must log the transformation at `info` level or higher. Silent drops are defects.

7. **Token budget accounting is consistent**: the `chars/3` estimation heuristic is used uniformly. Any budget calculation that mixes estimated and real (API-reported) token counts must apply the `UNCALIBRATED_SAFETY` multiplier or equivalent.
