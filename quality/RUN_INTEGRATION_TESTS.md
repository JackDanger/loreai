# Integration Test Protocol

Integration test groups verifying end-to-end behavior across Lore's gateway, core, and plugin packages. Each group targets specific use cases (UC) derived from EXPLORATION.md. Tests run against a real gateway instance using ephemeral port 0 and a temporary SQLite database.

## Prerequisites

- Node.js >= 22.5 installed
- All workspace dependencies installed (`pnpm install`)
- No production database used — tests use `packages/core/test/setup.ts` preload for DB isolation
- Environment: `NODE_ENV=test`, `LORE_HOSTED_MODE=0`
- Upstream API mocks: use `packages/gateway/test/helpers/harness.ts` test harness where available

## Group 1: Gateway Startup and Port Fallback

**Use cases:** UC-01 (Multi-session gateway with different models)
**Requirements:** REQ-003
**Related bugs:** None

### Test 1.1: Gateway starts on explicit port

**Steps:**
1. Start gateway with `--port 0` (ephemeral port assignment)
2. Read `resolvedPort` from server handle
3. Send HTTP GET to `http://127.0.0.1:<port>/health`

**Expected results:**
- Server starts without error
- Health endpoint returns 200 with JSON body
- `resolvedPort` is a valid port number > 0

**Pass/fail criteria:** Health endpoint returns HTTP 200.

### Test 1.2: Gateway falls back when preferred port is occupied

**Steps:**
1. Start a dummy TCP server on port 3207
2. Start gateway without explicit `--port` flag (defaults to 3207 -> 5673 -> 0)
3. Verify gateway started on a different port (5673 or ephemeral)
4. Stop dummy server

**Expected results:**
- Gateway detects port 3207 is in use
- Gateway falls back to port 5673 or ephemeral port 0
- No crash or unhandled exception

**Pass/fail criteria:** Gateway starts successfully on a fallback port.

### Test 1.3: Multi-host bind failure on resolved port (Quality Risk 7)

**Steps:**
1. Start gateway with `port: 0` and `hosts: ['127.0.0.1', '::1']`
2. Observe whether `resolvedPort` from first bind is used for second bind
3. If second bind fails, observe error handling

**Expected results:**
- Either both binds succeed on the same ephemeral port, OR
- Failure on second bind is caught and gateway falls back gracefully

**Pass/fail criteria:** No unhandled exception on multi-host bind. Gateway either succeeds on both interfaces or logs a warning and continues with the first.

## Group 2: Anthropic Protocol Round-Trip with Tool Calls

**Use cases:** UC-03 (Agentic workflow), UC-TOOL-001.a (Anthropic tool result round-trip)
**Requirements:** REQ-006
**Related bugs:** None (Anthropic path is the reference implementation)

### Test 2.1: Tool_result blocks preserved in Anthropic upstream request

**Steps:**
1. Start gateway with mock Anthropic upstream
2. Send a conversation turn containing a `tool_result` content block in the Anthropic message format
3. Capture the request forwarded to the mock upstream

**Expected results:**
- Upstream request contains the `tool_result` block unchanged
- Block has correct `tool_use_id` matching the prior `tool_use`
- Content text is preserved verbatim

**Pass/fail criteria:** `tool_result` block present in upstream request body with matching `tool_use_id`.

### Test 2.2: Multi-turn tool call round-trip

**Steps:**
1. Send turn 1: user message with tool definitions
2. Mock upstream returns `tool_use` response
3. Send turn 2: `tool_result` from tool execution
4. Mock upstream returns final text response
5. Verify both turns are forwarded correctly

**Expected results:**
- Turn 1 request includes tool definitions
- Turn 2 request includes `tool_result` block
- Gateway does not modify tool call IDs
- Response is returned to client

**Pass/fail criteria:** Complete tool call cycle completes without errors; `tool_result` visible to upstream on turn 2.

## Group 3: OpenAI Protocol Round-Trip with Tool Calls

**Use cases:** UC-03 (Agentic workflow), UC-TOOL-001.b (OpenAI tool result round-trip)
**Requirements:** REQ-006
**Related bugs:** CB-006 (tool_result blocks silently dropped)

### Test 3.1: Tool_result blocks translated to OpenAI `role: "tool"` messages

**Steps:**
1. Start gateway with mock upstream, client using OpenAI Chat Completions protocol
2. Build a conversation containing a gateway-internal `tool_result` block
3. Invoke `buildOpenAIMessages` (or send through pipeline)
4. Capture the upstream request

**Expected results:**
- Upstream request contains a message with `role: "tool"`
- Message has `tool_call_id` matching the prior `tool_use`
- Content is the tool result text

**Pass/fail criteria:** `role: "tool"` message present in upstream request. **EXPECTED TO FAIL (CB-006)** — tool_result is currently silently dropped at `translate/openai.ts:523-527`.

### Test 3.2: OpenAI tool call arguments with malformed JSON

**Steps:**
1. Send a request through the OpenAI translator where tool call arguments are malformed JSON (e.g., `"{invalid"`)
2. Observe error handling at `translate/openai.ts:166,205`

**Expected results:**
- Request handler does NOT crash with unhandled exception
- Error is caught and returned as a structured error response

**Pass/fail criteria:** No unhandled exception. **EXPECTED TO FAIL (CB-008)** — JSON.parse has no try/catch.

## Group 5: Streaming Error Propagation Across Protocols

**Use cases:** UC-STREAM-001.a (Anthropic), UC-STREAM-001.b (OpenAI), UC-STREAM-001.c (Responses)
**Requirements:** REQ-007
**Related bugs:** CB-012 (OpenAI streaming translator missing error handling)

### Test 5.1: Anthropic streaming error event forwarded to client

**Steps:**
1. Start gateway with mock upstream that sends an SSE `error` event after partial content
2. Client connects via Anthropic native protocol (streaming)
3. Collect all SSE events received by client

**Expected results:**
- Client receives partial `content_block_delta` events
- Client receives the `error` event with error type and message
- Stream terminates after error event

**Pass/fail criteria:** Client SSE stream contains an `error` event with the upstream error details.

### Test 5.2: OpenAI Chat Completions streaming error propagation

**Steps:**
1. Start gateway with mock Anthropic upstream that sends an SSE `error` event
2. Client connects via OpenAI Chat Completions protocol (streaming)
3. Collect all SSE events received by client

**Expected results:**
- Client receives partial `chat.completion.chunk` events
- Client receives an error indication (error object in final chunk or dedicated error event)
- Stream does NOT terminate silently with just `[DONE]`

**Pass/fail criteria:** Client receives error details, not just silent `[DONE]`. **EXPECTED TO FAIL (CB-012)** — error event falls through silently in `stream/openai.ts`.

### Test 5.3: Responses API streaming error propagation

**Steps:**
1. Start gateway with mock Anthropic upstream that sends an SSE `error` event
2. Client connects via OpenAI Responses API protocol (streaming)
3. Collect all SSE events received by client

**Expected results:**
- Client receives partial response events
- Client receives `response.failed` event with error details (lines 687-711 of `stream/openai-responses.ts`)
- Error includes type and message from upstream

**Pass/fail criteria:** Client SSE stream contains `response.failed` event with error details.

## Group 6: Session Identification and State Persistence

**Use cases:** UC-01, UC-02 (Long-running gateway session lifecycle)
**Requirements:** REQ-004, REQ-011
**Related bugs:** CB-001 (saveForceMinLayer row deletion), CB-002 (shared globals), CB-003 (unbounded maps)

### Test 6.1: Session identified via known header

**Steps:**
1. Send request with `x-session-id: test-session-001` header
2. Send second request with same header
3. Verify both requests are assigned the same session ID

**Expected results:**
- Both requests share the same internal session ID
- Session state is created on first request
- Session state is reused on second request

**Pass/fail criteria:** Same session ID returned for both requests.

### Test 6.2: Session state survives saveForceMinLayer(sid, 0)

**Steps:**
1. Create session via request, triggering session_state row creation
2. Store cost data and tracking data via `saveSessionCosts` and `saveSessionTracking`
3. Call `saveForceMinLayer(sessionId, 0)`
4. Call `loadSessionTracking(sessionId)`

**Expected results:**
- Session tracking data is still present after saveForceMinLayer
- Cost data is still present
- Only the `force_min_layer` column is cleared/reset

**Pass/fail criteria:** All session_state columns except force_min_layer are preserved. **EXPECTED TO FAIL (CB-001)** — entire row is deleted.

### Test 6.3: Concurrent fingerprint-based identification (race condition)

**Steps:**
1. Send two simultaneous requests with identical message content but no session header
2. Both should resolve to the same fingerprint
3. Check whether they receive the same or different session IDs

**Expected results:**
- Both requests should receive the same session ID
- No duplicate session creation

**Pass/fail criteria:** Same session ID for both. **MAY FAIL (REQ-004)** — no serialization of fingerprint lookups.

### Test 6.4: Module-level model limits not shared across sessions

**Steps:**
1. Start gateway handling two concurrent sessions
2. Session A: send request for `claude-opus-4-20250514` model (200K context)
3. Session B: send request for `gpt-4o` model (128K context)
4. Session A: send another request and check what context limit is used

**Expected results:**
- Session A uses Claude Opus context limits (200K)
- Session B uses GPT-4o context limits (128K)
- Session A's second request still uses Claude limits, not GPT-4o limits

**Pass/fail criteria:** Each session uses its own model's context limits. **EXPECTED TO FAIL (CB-002)** — module-level globals are shared.

## Group 7: Recall Marker Expansion and Follow-Up

**Use cases:** UC-02 (Long-running session lifecycle)
**Requirements:** REQ-014
**Related bugs:** CB-004 (untracked follow-up tokens), CB-005 (regex truncation)

### Test 7.1: Recall marker with simple query expands correctly

**Steps:**
1. Seed database with knowledge entries for the project
2. Send a request where the assistant response contains a recall marker: `📚 Searching project for "database migration"…`
3. Verify recall is executed and marker is expanded with search results

**Expected results:**
- Recall marker is detected by regex at `recall.ts:109`
- Search is executed against temporal + knowledge stores
- Marker is replaced with formatted recall results
- Follow-up request is issued to LLM

**Pass/fail criteria:** Recall marker expanded; follow-up request contains recall results.

### Test 7.2: Recall marker with double-quote in query

**Steps:**
1. Send a request where assistant response contains: `📚 Searching project for "how to use "async" patterns"…`
2. Observe extracted query

**Expected results:**
- Full query `how to use "async" patterns` is extracted
- Search uses the complete query

**Pass/fail criteria:** Full query extracted without truncation. **EXPECTED TO FAIL (CB-005)** — lazy `(.+?)` stops at first `"`.

### Test 7.3: Recall follow-up output tokens tracked

**Steps:**
1. Trigger a recall marker expansion that causes a follow-up LLM request
2. After follow-up completes, check `postResponse` usage accounting
3. Verify output tokens from follow-up are included in cost tracking

**Expected results:**
- Follow-up response tokens are counted in session usage
- Cost tracker includes follow-up costs
- Calibration receives complete token counts

**Pass/fail criteria:** Total output tokens include both initial and follow-up responses. **EXPECTED TO FAIL (CB-004)** — follow-up tokens not fed to accumulator.

### Test 8.3: LTM injected into Responses API instructions

**Steps:**
1. Seed database with same knowledge entries
2. Send Responses API request through gateway
3. Capture upstream request instructions field

**Expected results:**
- Instructions field contains LTM content

**Pass/fail criteria:** LTM content present in upstream instructions field.

### Test 8.4: LTM deduplication across pools

## Group 9: Distillation and Knowledge Curation Lifecycle

**Use cases:** UC-02 (Long-running gateway session lifecycle)
**Requirements:** REQ-011
**Related bugs:** CB-001 (state persistence), CB-011 (stale SCHEMA_VERSION)

### Test 9.1: Temporal storage → distillation → knowledge pipeline

**Steps:**
1. Store 20+ messages via `temporal.store()` for a session
2. Trigger `distillation.run()` (or wait for idle scheduler)
3. Verify gen-0 distillations created in `distillations` table
4. Trigger `curator.run()` with a mock LLM that returns create ops
5. Verify knowledge entries created in `knowledge` table

**Expected results:**
- Messages stored in `temporal_messages` with FTS5 indexing
- Distillation segments created as gen-0 entries
- Curator creates knowledge entries from distillation content
- `.lore.md` export triggered if entries changed

**Pass/fail criteria:** End-to-end pipeline produces knowledge entries from raw messages.

### Test 9.2: Meta-distillation consolidation

**Steps:**
1. Create 25+ gen-0 distillations for a session (exceeding threshold of 20)
2. Trigger `metaDistill()` with mock LLM
3. Verify gen-1 meta-distillation created
4. Verify gen-0 distillations are archived (not deleted)

**Expected results:**
- Gen-1 distillation created summarizing gen-0 content
- Gen-0 distillations marked as archived
- Archived distillations still searchable via recall
- Archived distillations excluded from in-context prefix

**Pass/fail criteria:** Gen-1 distillation exists; gen-0 distillations archived but not deleted.

### Test 9.3: Database schema version consistency

**Steps:**
1. Open test database
2. Read `PRAGMA user_version`
3. Compare with `MIGRATIONS.length`
4. Note value of `SCHEMA_VERSION` constant at `db.ts:24`

**Expected results:**
- `PRAGMA user_version` equals `MIGRATIONS.length` (currently 26)
- `SCHEMA_VERSION` constant is stale (16) but unused

**Pass/fail criteria:** Database version matches migration count. **INFORMATIONAL (CB-011)** — stale constant has no runtime impact.

## Group 10: Embedding Provider Fallback and Recovery

**Use cases:** UC-02 (Long-running gateway session lifecycle)
**Requirements:** REQ-001
**Related bugs:** CB-010 (permanent fallback)

### Test 10.1: Local embedding provider processes text successfully

**Steps:**
1. Initialize embedding system with local ONNX provider
2. Call `embed()` with a short text string
3. Verify embedding vector returned

**Expected results:**
- Local provider initializes successfully
- Embedding vector has expected dimensionality
- No fallback triggered

**Pass/fail criteria:** Non-null embedding vector returned from local provider.

### Test 10.2: Local provider failure triggers fallback to remote

**Steps:**
1. Initialize embedding system
2. Force local provider to fail (e.g., mock ONNX OOM error)
3. Call `embed()` after failure
4. Check `cachedProvider` state

**Expected results:**
- First call fails on local provider
- System falls back to remote provider
- Subsequent calls use remote provider

**Pass/fail criteria:** Fallback occurs; remote provider used for subsequent calls.

### Test 10.3: Fallback is permanent (no recovery to local)

**Steps:**
1. Trigger fallback as in Test 10.2
2. Wait or simulate passage of time
3. Call `embed()` again
4. Check whether local provider is retried

**Expected results:**
- Local provider is NOT retried after fallback
- `cachedProvider` remains set to remote provider indefinitely

**Pass/fail criteria:** Confirms CB-010 — fallback is permanent. **EXPECTED BEHAVIOR (current code)** but violates REQ-001.

### Test 10.4: `_shutdownAndDisable()` permanently disables embeddings

**Steps:**
1. Call `_shutdownAndDisable()` to set `cachedProvider = null`
2. Call `embed()` — should return null/skip
3. Verify this is distinct from the auto-fallback path

**Expected results:**
- `cachedProvider` is `null` (not a remote provider)
- Embeddings are fully disabled, not degraded to remote
- Three-state distinction maintained: `undefined` (not resolved), `null` (permanently off), `EmbeddingProvider` (active)

**Pass/fail criteria:** `null` state is distinct from fallback state; embeddings fully disabled.
