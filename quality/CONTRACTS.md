## 1. Gradient Context Manager

### GCM-01: Sticky layer guard restricts to layers 1-3 only

**Function:** `transformInner()`
**Location:** `packages/core/src/gradient.ts:1673-1674`

**Invariant:** The sticky layer guard (`effectiveMinLayer = Math.max(effectiveMinLayer, sessState.lastLayer)`) is only applied when `sessState.lastLayer >= 1 && sessState.lastLayer <= 3` AND the session is calibrated AND `input.messages.length >= sessState.lastKnownMessageCount`. Layer 0 is excluded (no stickiness for passthrough). Layer 4 (emergency) is excluded to prevent permanently trapping the session at emergency compression.

**Breaks if violated:**
- If layer 0 gets stickiness: sessions can never escalate to compression, causing "prompt too long" errors on growing conversations.
- If layer 4 gets stickiness: sessions are permanently trapped at emergency compression even after the context shrinks (e.g., after compaction), wasting context budget indefinitely.
- If the `messages.length >= lastKnownMessageCount` guard is removed: post-compaction sessions (where message count drops) can't recover to lower layers.

---

### GCM-02: forceMinLayer is one-shot consumed and persisted

**Function:** `transformInner()`
**Location:** `packages/core/src/gradient.ts:1628-1630`

**Invariant:** `forceMinLayer` is consumed exactly once per `transformInner()` call: read into `effectiveMinLayer`, immediately reset to 0 in memory (`sessState.forceMinLayer = 0`), and persisted to disk (`saveForceMinLayer(sid, 0)`). It is set by the overflow error handler and persisted to SQLite on mutation.

**Breaks if violated:**
- If not consumed (reset): every subsequent turn skips layers below the forced minimum, permanently over-compressing the session.
- If not persisted on set: a process restart between the overflow error and the next turn loses the escalation, causing the overflow to repeat indefinitely.
- If `saveForceMinLayer(sid, 0)` deletes the entire `session_state` row (the documented db.ts:1019 bug): cost data, tracking state, and gradient calibration are destroyed.

---

### GCM-03: Cache frozen during tool-call chains

**Function:** `loadDistillationsCached()`
**Location:** `packages/core/src/gradient.ts:266-278` (distillationSnapshot field)

**Invariant:** The distillation row snapshot (`distillationSnapshot`) is refreshed only at turn boundaries — when a new user message ID appears. During autonomous tool-call chains (assistant→user-tool-result→assistant→...), the snapshot stays frozen, keeping the distilled prefix byte-identical across consecutive API calls to preserve the prompt cache.

**Breaks if violated:** Each tool-call continuation busts the prompt cache (~$1.88 per bust at 500K Sonnet). A 10-step tool chain would cost ~$19 in unnecessary cache writes instead of ~$0.20 in cache reads.

---

### GCM-04: tryFit rejects when prefix exceeds distilled budget

**Function:** `tryFit()`
**Location:** `packages/core/src/gradient.ts:2081-2082`

**Invariant:** `if (input.prefixTokens > input.distilledBudget && input.prefix.length > 0) return null`. When the distilled prefix is non-empty and exceeds the distilled budget, the layer is rejected and the caller escalates to the next compression stage.

**Breaks if violated:** An oversized distilled prefix crowds out raw messages. The model sees a long summary but minimal recent context, degrading response quality. The total may also exceed `maxInput`, causing upstream "prompt too long" errors.

---

### GCM-05: Module-level globals are shared across all sessions

**Function:** `setModelLimits()`
**Location:** `packages/core/src/gradient.ts:37-38, 87-88`

**Invariant:** `contextLimit`, `outputReserved`, and `maxLayer0Tokens` are process-global. The last `setModelLimits()` call wins for ALL sessions. In multi-model hosted deployments, this means sessions using different models share the same limits.

**Breaks if violated:** This is a known architectural constraint, not a bug per se. But if a test assumes per-session isolation of these values, it will produce false results. Any future per-session context limit implementation must migrate these globals into `SessionState`.

---

### GCM-06: Uncalibrated safety multiplier on first turn

**Function:** `transformInner()`
**Location:** `packages/core/src/gradient.ts:1648, 1718`

**Invariant:** On the first turn of a session (`calibrated = false`, i.e. `sessState.lastKnownInput === 0`), the `UNCALIBRATED_SAFETY` multiplier of 1.5 is applied to the estimated token count before the layer-0 gate decision. This compensates for the chars/3 heuristic undercounting by ~1.68x.

**Breaks if violated:** Without the multiplier, a session estimated at 146K tokens passes the layer-0 gate but actually costs 214K tokens (real tokenizer) — causing an immediate "prompt too long" overflow. The overflow handler sets `forceMinLayer=2`, but one wasted API call has already been billed.

---

### GCM-07: Compression stages table drives layer escalation

**Function:** `COMPRESSION_STAGES` array
**Location:** `packages/core/src/gradient.ts:1557-1561`

**Invariant:** The three compression stages define the escalation path:
- Stage 0 (Layer 1): `strip: "none"`, all distillations, stable window (Approach B cache).
- Stage 1 (Layer 2): `strip: "old-tools"`, all distillations, 50% raw budget, 2 protected turns.
- Stage 2 (Layer 3): `strip: "all-tools"`, top 5 distillations (`distLimit: 5`), 55% raw + 15% distilled budgets.

Emergency (Layer 4) is hardcoded separately at `gradient.ts:1906` with `selectDistillations(distillations, 2)`.

**Breaks if violated:** Adding a stage without updating the `SafetyLayer` type (0-4) or the `effectiveMinLayer` guards causes type errors or skipped layers. Changing `distLimit` without updating `selectDistillations` scoring weights degrades distillation quality.

---

### GCM-08: selectDistillations re-sorts chronologically after scoring

**Function:** `selectDistillations()`
**Location:** `packages/core/src/gradient.ts:1232-1249`

**Invariant:** After selecting the top N distillations by score (70% recency + 30% importance bonus), the selected set is re-sorted by `created_at` ascending. This preserves chronological order in the prefix, which is critical for the Approach C append-only cache — the prefix must be byte-identical between turns.

**Breaks if violated:** Score-sorted distillations produce a different byte sequence on each turn (as scores change with new distillations), busting the prompt cache every turn.

## 2. Protocol Translation

### PT-01: LTM injection works for all three upstream protocols

**Function:** `handleConversationTurn()`
**Location:** `packages/gateway/src/pipeline.ts:1024-1051` (approximate)

**Invariant:** LTM knowledge entries are injected into the system prompt for all three upstream protocols: Anthropic (`/v1/messages`), OpenAI Chat Completions (`/v1/chat/completions`), and OpenAI Responses API (`/v1/responses`). The injection happens in the gateway's internal representation before protocol-specific translation.

**Breaks if violated:** Sessions using a non-Anthropic upstream protocol receive no long-term knowledge context. The agent has no memory of past decisions, patterns, or preferences. This was a historical bug (documented in `.lore.md` gotcha) that has since been fixed.

---

### PT-02: Tool result blocks preserved in OpenAI translation

**Function:** `buildOpenAIMessages()`
**Location:** `packages/gateway/src/translate/openai.ts:523-527`

**Invariant:** `tool_result` content blocks in the gateway's internal representation MUST be translated to `role: "tool"` messages with `tool_call_id` in OpenAI Chat Completions format. The tool call ID linkage between `tool_use` (assistant) and `tool_result` (user) must be preserved.

**Breaks if violated:** The LLM receives tool_use blocks without corresponding results. It either re-issues the tool calls (wasting tokens and causing duplicate side effects) or hallucinates results. Exploration finding #9 identified that the current implementation silently drops tool_result blocks.

---

### PT-03: Streaming error events propagated in OpenAI translator

**Function:** `translateAnthropicStreamToOpenAI()`
**Location:** `packages/gateway/src/stream/openai.ts:71-332`

**Invariant:** When the Anthropic upstream sends an `error` event type, the OpenAI streaming translator MUST emit an error chunk with the error reason to the client and close the stream with `data: [DONE]`. The error message content must be preserved, not swallowed.

**Breaks if violated:** Clients hang waiting for tokens that never arrive, or receive an empty completion indistinguishable from a normal response. The Responses API translator at `stream/openai-responses.ts:687-711` correctly emits `response.failed` — the Chat Completions translator must have equivalent handling.

---

### PT-04: Tool arguments JSON.parse is error-safe

**Function:** `parseOpenAIRequest()`
**Location:** `packages/gateway/src/translate/openai.ts:166, 205`

**Invariant:** `JSON.parse(fn.arguments)` on tool call arguments from OpenAI clients MUST be wrapped in try/catch. Malformed JSON should produce a fallback (empty object or the raw string) rather than an unhandled exception.

**Breaks if violated:** A single malformed tool call argument from any OpenAI-protocol client crashes the entire request handler, returning a 500 to the client and losing any partial response.

### PT-05: anthropic-beta header captured for cache warmer

**Function:** `handleConversationTurn()` (post-response)
**Location:** `packages/gateway/src/pipeline.ts:2114-2115`

**Invariant:** `sessionState.lastAnthropicBeta` is updated on every request from `req.rawHeaders["anthropic-beta"]`. When the header is absent, the field is set to `undefined` (not left stale). The cache warmer at `cache-warmer.ts:1057-1058` forwards this header on warmup requests.

**Breaks if violated:** Warmup requests containing beta-gated body fields (e.g., `context_management`) are sent without the required `anthropic-beta` header, causing Anthropic 400 errors. Or a stale header from a previous request causes unexpected beta behavior on the warmup.

## 3. Session Management

### SM-01: 3-tier session identification with deterministic precedence

**Function:** `identifySession()` pipeline
**Location:** `packages/gateway/src/session.ts:267-289` (Tier 1), `session.ts:387` (Tier 2), `session.ts:180` (Tier 3)

**Invariant:** Session identification follows strict precedence:
1. **Tier 1:** Known session headers (`x-claude-code-session-id`, `x-session-affinity`, `x-lore-session-id`) — first match wins.
2. **Tier 2:** Learned headers — heuristic header promotion from prior fingerprinted turns.
3. **Tier 3:** Message fingerprint — SHA-256 of first user message content + auth suffix, combined with message-count proximity (threshold: 20 messages).

**Breaks if violated:**
- If Tier 1 is skipped: deterministic session IDs from Claude Code/OpenCode are ignored, causing session splits.
- If Tier 3 lacks the proximity guard: forked sessions (same first message, different conversation paths) merge into one session, corrupting both sessions' memory.

---

### SM-02: State persistence lifecycle — 3 tiers

**Function:** Multiple — `saveForceMinLayer()`, `saveSessionTracking()`, `saveSessionCosts()`
**Location:** `packages/core/src/db.ts:1016-1031`, `packages/gateway/src/idle.ts:141-173`

**Invariant:** Session state persists in three tiers:
1. **Immediate:** Session identity fields (fingerprint, headerSessionId, headerName) and `forceMinLayer` — persisted on mutation.
2. **Per-turn:** Cost snapshots — piggybacked on existing `saveSessionTracking` in `postResponse`.
3. **30s periodic:** Gradient EMAs, cache warming state, `lastTurnAt`, `lastLayer`, `lastKnownInput` — via dirty flag + idle scheduler tick.

**Breaks if violated:**
- If immediate-tier fields use periodic persistence: a crash between the identity change and the next 30s tick loses the session mapping.
- If the periodic tier uses per-turn persistence: high-frequency state writes (every tool-call continuation) create unnecessary DB load.
- Maximum data loss on crash: ~30s of gradient/warmup state (acceptable), 0s of session identity (critical).

---

### SM-03: Fingerprint excludes model string

**Function:** `fingerprintMessages()`
**Location:** `packages/gateway/src/session.ts:180-203`

**Invariant:** The fingerprint hash includes the first user message content and an optional auth suffix. Model string is intentionally excluded because model strings change mid-session (e.g., `claude-opus-4-7` → `claude-opus-4-6`).

**Breaks if violated:** Including the model string causes session splits when the model is upgraded/downgraded mid-session — a confirmed production bug that motivated this exclusion.

## 4. LTM System

### LTM-01: forSession scoring pipeline — vector → FTS5 → confidence fallback

**Function:** `forSession()`
**Location:** `packages/core/src/ltm.ts:415-576`

**Invariant:** The scoring pipeline is:
1. If `sessionContext` has >20 chars AND `embedding.isAvailable()`: vector cosine similarity scoring with FTS5 fallback for entries without embeddings.
2. If `sessionContext` has >20 chars but embeddings unavailable: FTS5 BM25 scoring only.
3. If no session context (first turn): top entries by confidence only, capped at `NO_CONTEXT_FALLBACK_CAP` per pool.

Two mutually exclusive pools are scored independently:
- Pool 1: `project_id = ? AND cross_project = 0` (project-specific)
- Pool 2: `project_id IS NULL OR cross_project = 1` (cross-project)

Pool 1 gets a safety net: top `PROJECT_SAFETY_NET` entries by confidence are always included (at score `0.001 * confidence`). Pool 2 has NO safety net.

**Breaks if violated:**
- If pools are not mutually exclusive: entries with `cross_project = 1` appear in both pools, doubling their token cost.
- If the safety net is removed: high-confidence project entries with no keyword overlap with the current session context are silently excluded.
- If the no-context fallback is skipped: first-turn sessions get zero knowledge entries.

---

### LTM-02: Preference fast path skips relevance scoring

**Function:** `forSession()` — preference fast path
**Location:** `packages/core/src/ltm.ts:461-478`

**Invariant:** When `categories === ['preference']`, scoring is skipped entirely. Both pools are merged into `allPrefs`, sorted by `confidence DESC, updated_at DESC`, and greedily packed into the token budget with no count cap.

**Breaks if violated:** If relevance scoring is applied to preferences, session-irrelevant preferences (e.g., "always use English for changelogs") would be excluded because they don't match any session keywords. Preferences are unconditional user directives — relevance scoring harms them.

---

### LTM-03: formatKnowledge uses continue, not break, on budget overflow

**Function:** `formatKnowledge()`
**Location:** `packages/core/src/prompt.ts:543-548`

**Invariant:** When an entry's estimated token cost exceeds the remaining budget, the loop uses `continue` (skip this entry, try the next one) rather than `break` (stop entirely). This allows smaller entries that fit the remaining budget to be included even after a large entry is skipped.

**Breaks if violated:** If `break` is used: the first oversized entry terminates packing, potentially leaving 40% of the budget unused. Critical small entries (gotchas, patterns) after a large entry would be excluded.

---

### LTM-04: Dedup guard checks title match across project and cross-project pools

**Function:** `create()`
**Location:** `packages/core/src/ltm.ts:66-115`

**Invariant:** Before creating a new entry, three dedup checks run in order:
1. Exact title match within the same `project_id` (lines 68-80).
2. Exact title match across cross-project entries (lines 96-100).
3. Fuzzy title match via FTS5 + word overlap (lines 111-115).

If any check finds a match, the existing entry is updated instead of creating a duplicate. When an explicit `id` is provided (cross-machine import via agents-file), dedup is skipped — the caller handles duplicate detection by UUID.

**Breaks if violated:** Without the cross-project check (#2): the curator creates project-scoped duplicates of entries that already exist as cross-project knowledge. Without the fuzzy check (#3): near-duplicate titles like "Upgrade lock bug" vs "Upgrade binary lock re-entry bug" both exist as separate entries.

---

### LTM-05: 3-block system prompt — stable preferences vs context-bound

**Function:** Pipeline LTM injection
**Location:** `packages/gateway/src/pipeline.ts:1024-1051` (approximate)

**Invariant:** LTM is injected as two blocks:
- `system[1]`: stable preferences only, 1h cache TTL, pinned on turn 1. Budget: `preferenceLtm` config field (default 2% usable context).
- `system[2]`: context-bound entries (all non-preference categories), 5m cache TTL, deferred until turn 2+. Budget: `ltmBudget` (5% usable).

**Breaks if violated:**
- If preferences use the 5m TTL: they bust the cache every 5 minutes, costing ~$0.50 per bust for the system prompt prefix.
- If context-bound entries use the 1h TTL: stale knowledge entries (e.g., from an old session) persist for an hour after becoming irrelevant.

## 5. Distillation

### DIST-01: splitSegments base case prevents infinite recursion

**Function:** `splitSegments()`
**Location:** `packages/core/src/distillation.ts:143-153`

**Invariant:** `if (messages.length <= 1) return [messages]`. A single message that exceeds `maxSegmentTokens` is returned as an indivisible segment. Without this guard, `findSplitIndex()` returns `messages.length` (=1), producing `left = messages.slice(0, 1)` — an identical recursive call that never terminates.

**Breaks if violated:** Infinite recursion on any message exceeding 16384 estimated tokens (~49KB of text). The process crashes with a stack overflow. This is triggered by large tool outputs (file reads, search results) which are common in coding agent sessions.

---

### DIST-02: Atomic store + mark in single transaction

**Function:** `distillSegment()`
**Location:** `packages/core/src/distillation.ts:817-834`

**Invariant:** `storeDistillation()` and `temporal.markDistilled()` are wrapped in a single `BEGIN IMMEDIATE` / `COMMIT` transaction. On error, `ROLLBACK` ensures neither operation persists.

**Breaks if violated:** A crash between `storeDistillation` and `markDistilled` leaves messages in the `distilled=0` state with an existing distillation row containing their content. On restart, `run()` re-processes these messages, creating duplicate distillation content that inflates the prefix.

---

### DIST-03: Expansion guard discards bloated output

**Function:** `distillSegment()`
**Location:** `packages/core/src/distillation.ts:802-810`

**Invariant:** If `distilledTokens > maxAllowedExpansion(sourceTokens)`, the distillation output is discarded. Source messages are still marked as distilled (line 804) to prevent infinite retry loops. The expansion limits are:
- `sourceTokens < 100`: 5x expansion allowed
- `sourceTokens < 500`: 2x expansion allowed
- `sourceTokens >= 500`: must compress (1x limit)

**Breaks if violated:**
- If expanded output is kept: the distilled prefix grows larger than the raw messages it replaced, defeating the purpose of compression. At layer 1, the prefix could exceed the distilled budget, causing `tryFit` to reject (GCM-04) and escalate unnecessarily.
- If source messages are NOT marked distilled on discard: the same segment is re-distilled every cycle, burning LLM tokens indefinitely.

---

### DIST-04: Meta-distillation threshold clamped to minimum 2

**Function:** `run()`
**Location:** `packages/core/src/distillation.ts:723-726`

**Invariant:** `effectiveMetaThreshold = Math.max(2, input.metaThresholdOverride ?? cfg.distillation.metaThreshold)`. Meta-distillation is pointless with fewer than 2 gen-0 segments — there's nothing to consolidate. The minimum of 2 prevents wasted LLM calls.

**Breaks if violated:** A threshold of 1 triggers meta-distillation on every gen-0 distillation, consuming an LLM call to "consolidate" a single segment into itself. This also busts the prefix cache (meta-distillation rewrites gen-0 rows into gen-1), costing ~$1.88 per unnecessary bust.

---

### DIST-05: Tiny segments merged or absorbed, never distilled alone

**Function:** `splitSegments()`, `run()`
**Location:** `packages/core/src/distillation.ts:165-167`, `distillation.ts:690-701`

**Invariant:** Trailing segments with fewer than `MIN_SEGMENT_TOKENS` (64) tokens are merged into the previous segment during splitting. In `run()`, segments below `cfg.distillation.minSegmentTokens` are either absorbed (marked distilled without LLM call, when `force=true`) or left undistilled to accumulate with future messages.

**Breaks if violated:** Tiny segments produce low-quality distillations (too little context for the LLM to generate useful observations). The LLM call is wasted, and the resulting distillation may inflate the prefix without adding value.

## 6. Embedding

### EMB-01: Local provider failure sets permanent flag

**Function:** `LocalProvider.ensureWorker()` — `init-error` handler
**Location:** `packages/core/src/embedding.ts:349-366`

**Invariant:** On worker init failure, `localProviderKnownBroken = true` is set permanently. All pending requests are rejected with `LocalProviderUnavailableError`. Future calls to `isAvailable()` short-circuit to `false`. The auto-fallback at line 700 sets `cachedProvider` to the remote provider for the remaining process lifetime.

**Breaks if violated:** This is a known architectural limitation (Exploration finding #11). A transient OOM permanently degrades to remote embeddings. The contract documents the current behavior; a future fix should make this recoverable.

---

### EMB-02: Config change clears all embeddings for re-backfill

**Function:** `checkConfigChange()`
**Location:** `packages/core/src/embedding.ts:999-1038`

**Invariant:** When the embedding config fingerprint (provider + model + dimensions) changes, ALL existing embeddings in `knowledge`, `distillations`, and `temporal_messages` tables are set to `NULL`. The new fingerprint is stored in `kv_meta`. The next `backfillEmbeddings()` call re-embeds everything with the new provider.

**Breaks if violated:** Mixed embeddings from different providers/models in the same table produce nonsensical cosine similarity scores. A BGE Small embedding (384 dims) compared against a Nomic v1.5 embedding (768 dims) via `cosineSimilarity()` would use `Math.min(a.length, b.length)` — computing a partial dot product that is mathematically meaningless.

---

### EMB-03: Token-budget batching prevents ONNX OOM

**Function:** `nextBatch()`
**Location:** `packages/core/src/embedding.ts:1175-1191`

**Invariant:** Backfill batches are sized by estimated token area (`batch.length * max_tokens_in_batch`), not by fixed chunk count. The budget is `MAX_BATCH_TOKEN_AREA` (8192). This prevents ONNX OOM on batches containing long texts (distillation observations can be 4000+ chars).

**Breaks if violated:** A fixed batch size of 64 with one 4000-char observation creates a tensor area of `64 * 1000 = 64,000 tokens`, far exceeding the ONNX runtime's allocation limit. Error codes `287180544`, `144786472` crash the worker thread.

---

### EMB-04: Pre-truncation at LOCAL_MAX_CHARS before worker

**Function:** `LocalProvider.embed()`
**Location:** `packages/core/src/embedding.ts:38, 44-49`

**Invariant:** Before sending texts to the ONNX worker thread, `safeLocalTruncate()` truncates each text to `LOCAL_MAX_CHARS` (16384 chars, ~4096 tokens). The truncation respects UTF-16 surrogate pairs (backs up one char if the cut falls on a high surrogate).

**Breaks if violated:** A single max-length input (~8192 tokens) causes ONNX error `284432024` inside the worker, which then sets `localProviderKnownBroken = true` (EMB-01), permanently disabling local embeddings for the entire process.

## 7. Cache Warmer

### CW-01: Survival function drives warming decisions

**Function:** `shouldWarm()`
**Location:** `packages/gateway/src/cache-warmer.ts:644+`

**Invariant:** Cache warming is only performed when:
1. Circuit breaker is not tripped (`circuitBreakerTripped === false`).
2. Session has `>= MIN_TURNS_FOR_WARMING` (3) completed turns.
3. The blended survival probability (session histogram + global histogram with `BLEND_PSEUDOCOUNT` Bayesian smoothing) indicates P(return before TTL) × savings > warmup cost.

**Breaks if violated:**
- Without the minimum turns gate: one-shot sessions get warmed, wasting money on sessions that will never return.
- Without the circuit breaker: a cache key mismatch causes every warmup to write (not read), burning money at `cacheWriteCostPerToken` rates.

---

### CW-02: Circuit breaker trips after 3 uncached warmups

**Function:** `recordWarmupOutcome()`
**Location:** `packages/gateway/src/cache-warmer.ts:163-190`

**Invariant:** If `circuitBreakerFailures >= CIRCUIT_BREAKER_MAX_FAILURES` (3), `circuitBreakerTripped` is set to `true` and persisted to SQLite via `saveCircuitBreaker()`. Once tripped, ALL warming is disabled for the process lifetime. The circuit breaker state is loaded from DB on first access (`ensureCircuitBreakerLoaded()`).

**Breaks if violated:** Without the circuit breaker: if the warmup body doesn't match the cached prefix (e.g., due to a changed cache key algorithm), every warmup pays full cache-write cost. At 500K tokens and $3.75/MTok write cost, that's ~$1.88 per warmup — potentially hundreds of dollars per day across active sessions.

---

### CW-03: anthropic-beta header forwarded on warmup requests

**Function:** `executeWarmup()`
**Location:** `packages/gateway/src/cache-warmer.ts:1057-1058`

**Invariant:** `if (state.lastAnthropicBeta) { headers["anthropic-beta"] = state.lastAnthropicBeta; }`. The warmup request includes the same `anthropic-beta` header as the last real request from that session.

**Breaks if violated:** Warmup bodies containing beta-gated fields (e.g., `context_management`) are sent without the beta header, causing Anthropic 400 errors. These errors don't trip the circuit breaker (they're HTTP errors, not "uncached" responses), so warming silently fails on every tick.

## 8. Recall

### RCL-01: Marker-and-expand protocol round-trips correctly

**Function:** `buildRecallMarker()`, `expandRecallMarkers()`
**Location:** `packages/gateway/src/recall.ts:103-106`, `recall.ts:158-255`

**Invariant:** The recall lifecycle is:
1. On response: `tool_use` block with `name: "recall"` → replaced with marker text (`📚 Searching <scope> for "<query>"…`).
2. Recall is executed internally; result stored in `RecallStore` keyed by `scope:query`.
3. On next request: marker text blocks in assistant messages → expanded back to `tool_use` + synthetic `tool_result` user message.

The expansion correctly handles:
- Continuation content after the marker (split into separate assistant message).
- Missing following user message (synthetic tool_result message inserted).
- Prepending to existing user message (unshift to preserve tool_use ordering).

**Breaks if violated:**
- If marker text is not replaced on the response path: the client sees raw `📚` text instead of the recall result.
- If expansion is not performed on the request path: the upstream API receives a conversation with a text block where it expects a `tool_use` + `tool_result` pair, causing API errors or confused model behavior.

---

### RCL-02: Recall marker regex truncates on embedded quotes

**Function:** `MARKER_REGEX`
**Location:** `packages/gateway/src/recall.ts:109`

**Invariant (KNOWN VIOLATION):** `MARKER_REGEX = /📚 Searching (.+?) for "(.+?)"…/` uses lazy quantifiers. If the query contains a `"` character, the regex stops at the first `"` and truncates. This is a documented bug (Exploration finding #6).

**Expected fix:** Use a greedy quantifier for the query capture group with an anchor on the trailing `"…`: `/📚 Searching (.+?) for "(.+)"…/` — or escape quotes in `buildRecallMarker()`.

**Breaks if violated (current state):** Recall searches for partial queries when the user's query contains double quotes. Results are irrelevant or empty.

---

### RCL-03: Query expansion gating requires both config and LLM client

**Function:** `runRecall()`
**Location:** `packages/core/src/recall.ts:470-472`

**Invariant:** `if (searchConfig?.queryExpansion && llm)` — query expansion is only attempted when BOTH the config flag is true AND an LLM client is provided. The config default is `true` (in both field-level and section-level `.default()` in `config.ts:137,195`). The LLM client must be passed by the caller (gateway pipeline or CLI).

**Breaks if violated:**
- If the LLM client is not wired in the gateway caller: `llm` is undefined, and `queryExpansion && undefined` is always false. Query expansion is silently disabled despite being configured. This was a historical bug (`.lore.md` gotcha "Query expansion disabled by default AND LLM client not wired").
- If only the field-level default is changed but not the section-level `.default({})`: Zod's object-level default overrides the field-level default, silently keeping the old value.

---

### RCL-04: Follow-up request strips recall tool and preserves thinking blocks

**Function:** `buildRecallFollowUp()`
**Location:** `packages/gateway/src/pipeline.ts:1332-1391` (streaming path), `pipeline.ts:3288-3312` (non-streaming path)

**Invariant:** The recall follow-up request:
1. Strips `recall` from the tools list to prevent re-invocation.
2. Converts the `tool_use`/`tool_result` pair to plain text (keeping `tool_use` with `recall` stripped causes API rejection; orphaned `tool_result` without matching `tool_use` also fails).
3. Preserves thinking blocks in assistant messages when extended thinking is enabled (Anthropic rejects assistant messages missing thinking signatures).
4. Sets `cacheConversation: false` to avoid caching the follow-up's modified conversation.

**Breaks if violated:**
- If `recall` is not stripped from tools: the model may call recall again in the follow-up, creating an infinite loop.
- If thinking blocks are stripped: Anthropic API rejects the request with "assistant message missing thinking signature".
- If `cacheConversation` is not false: the modified conversation (with inline tool results instead of markers) is cached, corrupting future turns.

## 9. Background Work

### BG-01: p-limit(2) caps concurrent background LLM calls

**Function:** `runBackground()`
**Location:** `packages/gateway/src/background-limiter.ts:27`

**Invariant:** All background LLM work (idle distillation, curation, pipeline-triggered incremental distillation) is routed through a single `p-limit(2)` instance. At most 2 background LLM operations run concurrently across all sessions. Urgent distillation is excluded (bypasses the limiter).

**Breaks if violated:** Without the limiter, N idle sessions fire N×4 simultaneous background calls (distillation + meta-distillation + curation + knowledge consolidation per session), overwhelming the upstream rate limit and causing cascading 429 failures that starve conversation turns of their rate limit budget.

---

### BG-02: Circuit breaker pauses background work on 429

**Function:** `tripCircuitBreaker()`, `isBackgroundPaused()`
**Location:** `packages/gateway/src/background-limiter.ts:43-68`

**Invariant:** When any background LLM call receives a 429, `tripCircuitBreaker()` pauses ALL background work for `Retry-After` seconds (default 60s). The pause only extends, never shortens (`if (until > circuitOpenUntil)`). Background work checks `isBackgroundPaused()` before submitting to the limiter. The circuit breaker auto-resets when `Date.now() >= circuitOpenUntil`.

**Breaks if violated:**
- Without the circuit breaker: background retries compete with conversation turns for the rate limit budget, degrading interactive response times.
- If the breaker is permanent (like the cache warmer's breaker): a single transient 429 disables all background work for the process lifetime, preventing distillation and curation from ever running again.
