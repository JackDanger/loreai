# TDD Red-Green Verification Protocol

For each confirmed bug from `quality/EXPLORATION.md` (CB-001 through CB-012), this protocol defines:

1. **Red phase:** Write a regression test that exercises the buggy code path. Run it against the current (unfixed) codebase. The test MUST FAIL to confirm the bug exists.
2. **Green phase:** Apply the minimal fix patch. Run the same test. The test MUST PASS to confirm the fix works.
3. **Log and verdict:** Record red/green logs and produce a machine-readable sidecar.

## Directory Structure

```
quality/results/
  BUG-001.red.log        # test output from red phase (expected: FAIL)
  BUG-001.green.log      # test output from green phase (expected: PASS)
  BUG-002.red.log
  BUG-002.green.log
  ...
  tdd-results.json        # machine-readable sidecar
```

## Log Capture

All test runs are captured with:

```bash
# Red phase
vitest run <test-file> --bail 2>&1 | tee quality/results/BUG-NNN.red.log
echo "EXIT_CODE=$?" >> quality/results/BUG-NNN.red.log

# Green phase (after applying fix)
vitest run <test-file> --bail 2>&1 | tee quality/results/BUG-NNN.green.log
echo "EXIT_CODE=$?" >> quality/results/BUG-NNN.green.log
```

## Verdict Values

| Verdict | Meaning |
|---------|---------|
| `TDD verified` | Red phase FAILED (bug confirmed), green phase PASSED (fix works). Full TDD cycle complete. |
| `red failed` | Red phase PASSED unexpectedly — the test does not reproduce the bug. Either the bug was already fixed, the test is wrong, or the bug is environment-specific. Investigate. |
| `green failed` | Red phase FAILED (bug confirmed) but green phase also FAILED — the fix is incomplete or introduces a regression. Fix needs revision. |
| `confirmed open` | Red phase FAILED (bug confirmed), fix not attempted or not feasible in this cycle. Bug is real and remains open. |
| `deferred` | Bug not tested in this cycle. Reason documented. |

## Execution Workflow

### Prerequisites

```bash
# Ensure clean working tree (or stash changes)
# Create results directory
mkdir -p quality/results
```

### Per-Bug Execution

For each BUG-NNN:

```bash
# 1. Write the regression test (add to existing test file or create new)
# 2. Red phase — run test against CURRENT code
vitest run <test-file> -t "<test-name>" --bail 2>&1 | tee quality/results/BUG-NNN.red.log
echo "EXIT_CODE=$?" >> quality/results/BUG-NNN.red.log

# 3. Verify red: exit code should be non-zero (test FAILED)
#    If exit code is 0, verdict = "red failed" — investigate

# 4. Apply minimal fix patch to the source file(s)

# 5. Green phase — run same test against FIXED code
vitest run <test-file> -t "<test-name>" --bail 2>&1 | tee quality/results/BUG-NNN.green.log
echo "EXIT_CODE=$?" >> quality/results/BUG-NNN.green.log

# 6. Verify green: exit code should be 0 (test PASSED)
#    If exit code is non-zero, verdict = "green failed" — fix is incomplete

# 7. Run full test suite to check for regressions
vitest run 2>&1 | tee quality/results/BUG-NNN.regression.log

# 8. Revert fix (we're only verifying, not shipping)
#    OR: keep fixes if Phase 6 will integrate them
```

### Sidecar Generation

After all bugs are processed, generate `quality/results/tdd-results.json` using the template in the Sidecar Format section above. Populate:

- `red_exit_code` and `red_passed` from the `.red.log` file
- `green_exit_code` and `green_passed` from the `.green.log` file
- `verdict` based on the decision matrix:

| Red exit code | Green exit code | Verdict |
|--------------|----------------|---------|
| non-zero (FAIL) | 0 (PASS) | `TDD verified` |
| 0 (PASS) | — | `red failed` |
| non-zero (FAIL) | non-zero (FAIL) | `green failed` |
| non-zero (FAIL) | not attempted | `confirmed open` |
| not attempted | not attempted | `deferred` |
