## Pass 1 — Structural Review (Anti-Hallucination Guardrails)

### Purpose

Identify code-level defects that exist regardless of requirements: missing error handling, resource leaks, null/undefined hazards, async error propagation gaps, and unbounded growth patterns. This pass uses no requirement references — findings are purely structural.

### Scope

All `role: "code"` files from `quality/exploration_role_map.json`. Priority order:

1. **Critical path files** (read first):
   - `packages/core/src/gradient.ts` — module-level globals, session state map
   - `packages/gateway/src/pipeline.ts` — central orchestrator, streaming, recall follow-up
   - `packages/core/src/db.ts` — saveForceMinLayer row deletion, persistence functions
   - `packages/core/src/embedding.ts` — provider fallback, permanent replacement
   - `packages/gateway/src/stream/openai.ts` — streaming error propagation
   - `packages/gateway/src/translate/openai.ts` — tool_result handling, JSON.parse
   - `packages/gateway/src/idle.ts` — periodic persistence, dirty-flag flush
   - `packages/gateway/src/session.ts` — unbounded globalHeaderValues map

2. **Secondary files** (read if time permits):
   - `packages/gateway/src/server.ts` — multi-host bind error handling
   - `packages/gateway/src/cache-warmer.ts` — circuit breaker recovery
   - `packages/gateway/src/recall.ts` — marker regex, query expansion
   - `packages/gateway/src/cli/bin.ts` — process.exit vs safeExit
   - `packages/core/src/ltm.ts` — pool dedup, forSession scoring
   - `packages/core/src/session-limiter.ts` — unbounded p-limit map
   - `packages/core/src/distillation.ts` — splitSegments recursion guard

### Checklist

For each file, check the following categories. Record findings as `P1-NNN`.

#### 1.1 Error Handling Completeness
- [ ] Every `JSON.parse()` call is wrapped in try/catch or uses a safe parser
- [ ] Every `fetch()` / HTTP call has error handling for network failures and non-2xx responses
- [ ] Every database operation handles SQLite errors (BUSY, LOCKED, CORRUPT)
- [ ] Every file I/O operation handles ENOENT, EACCES, EISDIR
- [ ] No bare `throw` without error type/message

**Entry points:**
- `packages/gateway/src/translate/openai.ts:166,205` — JSON.parse without try/catch (CB-008)
- `packages/gateway/src/pipeline.ts:1371-1440` — recall follow-up stream error path
- `packages/core/src/db.ts:1019-1031` — saveForceMinLayer DELETE vs UPDATE (CB-001)

#### 1.2 Resource Cleanup
- [ ] Every `AbortController` is aborted on all exit paths (success, error, timeout)
- [ ] Every stream (ReadableStream, WritableStream) is closed/cancelled on error
- [ ] Every database statement is finalized or using `db.exec`/`db.run` (auto-finalize)
- [ ] Timers (`setTimeout`, `setInterval`) are cleared on cleanup/shutdown

**Entry points:**
- `packages/gateway/src/stream/openai.ts` — `cancelled` flag + `cancel()` handler
- `packages/gateway/src/stream/openai-responses.ts` — `cancelled` flag + `cancel()` handler
- `packages/gateway/src/pipeline.ts` — AbortController for upstream requests
- `packages/gateway/src/idle.ts` — interval timers for periodic flush

#### 1.3 Null/Undefined Guards
- [ ] Optional chaining used where objects may be null/undefined from DB queries
- [ ] Map.get() results checked before use (no implicit undefined-as-value)
- [ ] Array indexing guarded against out-of-bounds (especially `[0]` on empty arrays)
- [ ] Function parameters with `| undefined` types are checked before use

**Entry points:**
- `packages/core/src/gradient.ts:304` — `sessionStates.get(sessionId)` callers
- `packages/gateway/src/session.ts:436-455` — `globalHeaderValues.get()` callers
- `packages/core/src/ltm.ts:463-464` — pool array merging without null check

#### 1.4 Async Error Propagation
- [ ] Every `async` function has its Promise rejection handled by the caller
- [ ] Fire-and-forget async calls use `.catch()` or are explicitly documented as intentional
- [ ] Stream pipeline errors propagate to the client (not swallowed silently)
- [ ] Worker thread errors are caught and re-thrown with context

**Entry points:**
- `packages/gateway/src/pipeline.ts:1371-1440` — recall follow-up stream piped to client (CB-004)
- `packages/gateway/src/stream/openai.ts` — Anthropic error events not propagated (CB-012)
- `packages/core/src/embedding-worker.ts` — ONNX OOM error handling
- `packages/gateway/src/idle.ts:141-173` — periodic flush error handling

#### 1.5 Unbounded Growth Patterns
- [ ] Every in-memory Map/Set has an eviction strategy or bounded size
- [ ] Every cache has a TTL or LRU eviction policy
- [ ] Process-global state is documented as intentionally global or per-session
- [ ] Module-level mutable variables are documented as safe for concurrent access

**Entry points:**
- `packages/core/src/gradient.ts:37-38` — module-level `contextLimit`, `outputReserved` (CB-002)
- `packages/core/src/gradient.ts:304` — `sessionStates` Map, never evicted (CB-003)
- `packages/core/src/session-limiter.ts:17` — p-limit instances, never evicted (CB-003)
- `packages/gateway/src/session.ts:436` — `globalHeaderValues` Map, never evicted (CB-003)
- `packages/gateway/src/pipeline.ts:280,327-332` — `sessions`, `stableLtmCache` Maps

## Pass 2 — Requirement Verification

### Purpose

For each REQ-001 through REQ-014, verify whether the current code satisfies the requirement. Cite file:line for each check. Focus on cross-protocol parity requirements.

### Procedure

For each requirement:
1. Read the requirement text from EXPLORATION.md (Derived Requirements section)
2. Read the referenced file(s) at the cited lines
3. Determine: SATISFIED, PARTIALLY SATISFIED, NOT SATISFIED, or CANNOT VERIFY
4. Cite the exact file:line evidence for the determination
5. If NOT SATISFIED, cross-reference to the matching CB-NNN

## Pass 3 — Cross-Requirement Consistency

### Purpose

Check that requirements don't contradict each other, and that cross-cutting concerns (session state persistence, LTM injection, protocol parity) are consistent across all code paths.

### Consistency Checks

#### 3.1 Session State Persistence Consistency

All writers to the `session_state` SQLite table must be consistent. Verify that:

- **Writers to check:**
  - `packages/core/src/db.ts` — `saveForceMinLayer()` (line 1019), `saveSessionTracking()` (line 891), `saveSessionCosts()`, `saveGradientState()`
  - `packages/gateway/src/idle.ts` — periodic persistence (line 141-173)
  - `packages/gateway/src/pipeline.ts` — `postResponse` cost snapshot

- **Consistency checks:**
  - [ ] All writers use UPSERT (INSERT OR REPLACE / ON CONFLICT) consistently
  - [ ] No writer deletes rows that another writer depends on (CB-001: saveForceMinLayer DELETE)
  - [ ] Column set is the same across all writers (no partial row updates that lose other columns)
  - [ ] Concurrent writes from different sessions don't collide on the same row
  - [ ] REQ-011 (state persistence lifecycle) is not contradicted by any writer

#### 3.2 LTM Injection Consistency

LTM must be injected into all three protocol paths. Verify:

- **Injection points:**
  - `packages/gateway/src/pipeline.ts:1024-1051` — protocol dispatch for LTM injection
  - `packages/gateway/src/translate/anthropic.ts:333` — Anthropic 3-block injection
  - OpenAI path — string concatenation
  - Responses path — instructions concatenation

- **Consistency checks:**
  - [ ] All three paths receive the same LTM content (REQ-005 parity)
  - [ ] No path silently skips LTM injection on edge cases (empty system prompt, missing instructions)
  - [ ] Preference vs context-bound distinction is applied identically (or documented as asymmetric)
  - [ ] Token budgets for LTM are computed the same way regardless of protocol

#### 3.3 Protocol Parity (REQ-006 vs REQ-007)

Tool result handling (REQ-006) and streaming error propagation (REQ-007) should be consistent across the same set of three protocols:

- **Check:** If tool_result is broken for OpenAI (CB-006), are there also other message types that are silently dropped by the same translator?
- **Check:** If streaming errors are broken for OpenAI Chat Completions (CB-012), is the non-streaming error path also broken?
- **Check:** Does the Responses API translator have equivalent gaps to the OpenAI translator, or are they independent implementations?

#### 3.4 Requirement Contradiction Check

- [ ] REQ-001 (recoverable fallback) does not conflict with the `.lore.md` gotcha about `resetProvider()` in `resetPipelineState()`
- [ ] REQ-004 (serialized fingerprint lookup) does not conflict with the performance requirement of low-latency session identification
- [ ] REQ-011 (consistent persistence) does not conflict with the 30s periodic flush strategy (some fields are intentionally transient)
- [ ] REQ-012 (pool dedup) does not conflict with the preference fast-path optimization (which already has SQL-level separation)
