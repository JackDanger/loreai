## Project Overview

**Lore** is a three-tier memory architecture that gives AI coding agents persistent, cross-session memory. It sits between an AI agent (Claude Code, Codex, OpenCode, Pi, or others) and the LLM provider (Anthropic, OpenAI, xAI, etc.) as a transparent proxy gateway. Every conversation turn is intercepted, stored in temporal storage (SQLite + FTS5), progressively distilled into compressed summaries, and mined for long-term knowledge entries (decisions, patterns, gotchas, preferences, architecture notes). Those knowledge entries are injected back into future sessions' system prompts and made searchable via a `recall` tool, giving agents a durable memory that survives session boundaries, process restarts, and even project forks.

**Who uses it:** Developers who use AI coding agents in their daily workflow. Lore is deployed as a local CLI (`lore run` / `lore start`), as in-process plugins for OpenCode and Pi, and as a hosted gateway for multi-user environments. The primary consumers of Lore's memory are the AI agents themselves — they receive knowledge context automatically and query it explicitly via recall.

**Why it matters:** Without persistent memory, AI agents lose all context between sessions: they re-discover the same gotchas, forget past decisions, and can't build on prior work. Lore's three-tier architecture (temporal storage → distillation → knowledge extraction) provides a principled solution that balances context freshness against token budget, cost against quality, and local privacy against remote hosting flexibility. Correctness in each tier directly affects the quality of AI assistance developers receive.

**Runtime:** Bun (development/tests) and Node.js ≥ 22.5 (production npm bundles). TypeScript monorepo with four packages: `@loreai/core` (shared engine), `@loreai/gateway` (proxy + CLI), `@loreai/opencode` (OpenCode plugin), `@loreai/pi` (Pi plugin). SQLite with WAL mode and FTS5 full-text search.

## Use Cases

### UC-01: Multi-session gateway with different models

- **Actors:** Two or more AI agents using different LLM models (e.g., Claude Opus via Anthropic protocol, GPT-4 via OpenAI protocol)
- **Preconditions:** Gateway running (possibly in hosted mode), sessions active concurrently
- **Flow:**
  1. Agent A sends a request with `model: claude-opus-4-6` → `setModelLimits()` at `packages/core/src/gradient.ts:425` sets module-level `contextLimit=200000`, `outputReserved=32000`
  2. Agent B sends a request with `model: gpt-4-turbo` → `setModelLimits()` overwrites the same module-level globals at `gradient.ts:37–38` with GPT-4's limits
  3. Agent A's next turn reads the now-incorrect `contextLimit` from the module-level variable — gradient compression decisions are made against the wrong model's budget
- **Postconditions:** Each session MUST use its own model's context limits for gradient layer decisions. Cross-session contamination of token budgets must not occur.
- **Requirements:** REQ-002

### UC-02: Long-running gateway session lifecycle

- **Actors:** Gateway process, multiple AI agent sessions connecting and disconnecting over 24+ hours
- **Preconditions:** Gateway started via `lore start`
- **Flow:**
  1. Sessions connect over time → `sessionStates` Map at `packages/core/src/gradient.ts:304` grows, p-limit instances at `packages/gateway/src/session-limiter.ts:17` accumulate, `globalHeaderValues` at `packages/gateway/src/session.ts:436` expands
  2. Sessions disconnect — but no eviction logic exists for any of these maps
  3. After 24+ hours with hundreds of sessions, memory grows monotonically
- **Postconditions:** Memory usage should stabilize; stale sessions should be evicted or aged out. Persisted state should survive process restarts via the 30s periodic flush in `packages/gateway/src/idle.ts:141–173`.
- **Requirements:** REQ-011

### UC-03: Agentic workflow through OpenAI upstream

- **Actors:** AI agent, gateway, OpenAI Chat Completions upstream
- **Preconditions:** Gateway proxying to an OpenAI-protocol upstream; tool definitions registered in the conversation
- **Flow:**
  1. Agent makes a tool call → model returns `tool_use` block → gateway forwards to tool → tool returns `tool_result`
  2. Next turn: agent sends conversation including `tool_result` blocks in prior messages
  3. `buildOpenAIMessages()` at `packages/gateway/src/translate/openai.ts:492–543` iterates message blocks — `tool_result` blocks hit the empty branch at lines 523–526 and are silently dropped
  4. Upstream model receives the conversation without tool results — it cannot see what tools returned
- **Postconditions:** Tool results from prior turns MUST be preserved as `role: "tool"` messages with `tool_call_id` in the OpenAI Chat Completions format. Agentic workflows must function correctly regardless of upstream protocol.
- **Requirements:** REQ-006

### Embedding and Search

#### REQ-001: Recoverable Embedding Provider Fallback

| Field | Value |
|---|---|
| **ID** | REQ-001 |
| **Title** | Recoverable embedding provider fallback |
| **Tier** | 3 — derived from code exploration |
| **Functional Section** | Embedding and Search |
| **Specificity** | Specific |

**Summary:** When the local ONNX embedding provider fails (e.g., transient OOM), the auto-fallback to a remote provider (Voyage or OpenAI) MUST be recoverable. The system must not permanently replace the local provider for the entire process lifetime after a single failure.

**User Story:** As a developer running Lore with local embeddings, I want transient ONNX failures to fall back to remote embeddings temporarily so that a single OOM doesn't permanently degrade to higher-cost, higher-latency remote embeddings for the rest of the process.

**Implementation Note:** At `packages/core/src/embedding.ts:700`, `cachedProvider = fallback.provider` permanently replaces the local provider. The three-state system at line 502 (`undefined`=not resolved, `null`=permanently off, `EmbeddingProvider`=active) is bypassed — the fallback sets `cachedProvider` directly to the remote provider with no mechanism to retry local. `_shutdownAndDisable()` at line 569 correctly uses `null` for permanent disable, but the auto-fallback doesn't distinguish transient from permanent failure.

**Conditions of Satisfaction:**
1. After a transient local provider failure, the system falls back to remote embeddings for the immediate request.
2. The system retries local provider initialization after a configurable cooldown (e.g., 60s or next `embed()` call after N seconds).
3. If local provider succeeds on retry, subsequent requests use local again.
4. If local provider fails permanently (e.g., ONNX binaries missing), the system converges to remote-only without infinite retries.
5. The fallback event is logged once per transition (already handled by `remoteFallbackLogged` at line 606).

**Alternative Paths:**
- Per-request fallback: try local first on every call, fall back to remote on each failure. Simpler but adds latency on every embed call when local is down.
- Timer-based retry: set a retry timestamp; after it elapses, reset `cachedProvider = undefined` to trigger re-resolution via `getProvider()`.

**Use Cases:** UC-02 (long-running gateway lifecycle)

**References:**
- `packages/core/src/embedding.ts:502` — `cachedProvider` three-state variable
- `packages/core/src/embedding.ts:677–703` — `embed()` with auto-fallback logic
- `packages/core/src/embedding.ts:569–577` — `_shutdownAndDisable()` permanent disable

#### REQ-002: FTS5 Degradation Signal

| Field | Value |
|---|---|
| **ID** | REQ-002 |
| **Title** | FTS5 fallback degradation signal |
| **Tier** | 3 — derived from code exploration |
| **Functional Section** | Embedding and Search |
| **Specificity** | Specific |

**Summary:** When FTS5 full-text search fails and the system falls back to LIKE-based queries, a degradation signal SHOULD be propagated to callers so they can log the event and optionally adjust scoring behavior.

**User Story:** As the LTM scoring pipeline, I want to know when FTS5 failed and I'm receiving LIKE-based results so that I can adjust confidence in relevance scores and log the degradation for diagnostics.

**Implementation Note:** At `packages/core/src/ltm.ts:272–274`, the FTS5 error is caught silently (`catch { // FTS5 error — fall through to no match }`). Similarly at `packages/core/src/search.ts:159–163`, `ftsQuery()` builds FTS5 MATCH expressions that can fail on syntax errors. The LIKE fallback loses BM25 ranking and prefix matching — results are unranked, which degrades `forSession()` relevance scoring silently.

**Conditions of Satisfaction:**
1. When FTS5 throws and the system falls back to LIKE, a structured signal (return value flag or thrown typed error) indicates the degradation.
2. Callers that use FTS5 results for scoring (e.g., `scoreEntriesFTS()` in `ltm.ts`) can detect the signal and apply a confidence penalty or log a warning.
3. The degradation is logged at `info` level for operational visibility.

**Alternative Paths:**
- Return a `{ results, degraded: boolean }` tuple from search functions.
- Use a typed `FTSDegradedError` that wraps the LIKE results.

**Use Cases:** UC-02 (long-running lifecycle — FTS5 corruption over time)

**References:**
- `packages/core/src/search.ts:159–163` — `ftsQuery()` FTS5 MATCH builder
- `packages/core/src/search.ts:194` — `ftsQueryRelaxed()` progressive relaxation
- `packages/core/src/ltm.ts:272–274` — silent FTS5 error catch in `findDuplicate()`

### Gateway Networking

#### REQ-003: Multi-Interface Port Conflict Probe

| Field | Value |
|---|---|
| **ID** | REQ-003 |
| **Title** | Multi-interface port conflict probe |
| **Tier** | 3 — derived from code exploration |
| **Functional Section** | Gateway Networking |
| **Specificity** | Specific |

**Summary:** The gateway port-conflict detection probe MUST check all configured host interfaces, not just the first one, to correctly detect an existing Lore gateway.

**User Story:** As a developer starting `lore run` when a gateway is already running on `0.0.0.0:3207`, I want the startup logic to detect the existing gateway so that it reuses it instead of failing with a confusing "port in use" error or falling back to an alternate port unnecessarily.

**Implementation Note:** At `packages/gateway/src/cli/start.ts:138`, the probe URL is constructed from `config.hosts[0]` only: `const probeUrl = \`http://${config.hosts[0]}:${candidatePort}\``. If `config.hosts = ["127.0.0.1"]` but the existing gateway is bound to `0.0.0.0`, the probe succeeds (127.0.0.1 is a subset of 0.0.0.0). But if `config.hosts = ["192.168.1.100"]` (Tailscale IP) and the existing gateway is on `127.0.0.1`, the probe misses it. The server binding loop at `packages/gateway/src/server.ts:357–368` binds to ALL configured hosts sequentially — if the resolved port from the first bind is taken on a subsequent interface, the second `Bun.serve()` throws uncaught.

**Conditions of Satisfaction:**
1. Port conflict probe checks at least `127.0.0.1` and all entries in `config.hosts`.
2. If any probe succeeds, the existing gateway is reused.
3. If port 0 is resolved from the first bind, subsequent binds on the same resolved port that fail are handled gracefully (not an uncaught throw).

**Alternative Paths:**
- Probe `127.0.0.1` unconditionally (it's always reachable for local gateways), plus each configured host.
- Wrap the sequential `Bun.serve()` loop in try/catch per host.

**Use Cases:** UC-01 (multi-session gateway startup)

**References:**
- `packages/gateway/src/cli/start.ts:137–148` — port conflict probe using `config.hosts[0]`
- `packages/gateway/src/server.ts:356–368` — sequential multi-host bind loop

### Session Management

#### REQ-004: Serialized Fingerprint-Based Session Lookup

| Field | Value |
|---|---|
| **ID** | REQ-004 |
| **Title** | Serialized fingerprint-based session lookup |
| **Tier** | 3 — derived from code exploration |
| **Functional Section** | Session Management |
| **Specificity** | Specific |

**Summary:** Fingerprint-based session identification (Tier 3 fallback) MUST serialize concurrent lookups for the same fingerprint to prevent session splits where two requests from the same conversation create separate sessions.

**User Story:** As a developer whose agent sends two rapid requests at session start (before a header-based session ID is established), I want both requests to be assigned to the same session so that my conversation history isn't fragmented across two independent sessions.

**Implementation Note:** `identifySession()` at `packages/gateway/src/pipeline.ts:909–1000` computes a SHA-256 fingerprint via `fingerprintMessages()` at `packages/gateway/src/session.ts:180`. The fingerprint is derived from the first user message + auth suffix. If two concurrent requests arrive for a new session, both compute the same fingerprint, both find no match in `sessions`, and both call `generateSessionID()` — creating two sessions for the same conversation. Tier 1 (known headers) and Tier 2 (learned headers) don't have this race because they use synchronous Map lookups.

**Conditions of Satisfaction:**
1. Concurrent `identifySession()` calls with the same fingerprint are serialized (e.g., via a per-fingerprint mutex or CAS loop).
2. The first request creates the session; the second request finds and reuses it.
3. Serialization does not block unrelated fingerprints (no global lock).
4. The serialization mechanism does not introduce deadlocks.

**Alternative Paths:**
- Per-fingerprint lock using a Map of Promises (lock-by-key pattern).
- Optimistic creation with a dedup check after insertion.

**Use Cases:** UC-01 (multi-session gateway)

**References:**
- `packages/gateway/src/pipeline.ts:909–1000` — `identifySession()` 3-tier strategy
- `packages/gateway/src/session.ts:180–199` — `fingerprintMessages()` SHA-256

### State Persistence

#### REQ-011: Consistent Session State Persistence Lifecycle

| Field | Value |
|---|---|
| **ID** | REQ-011 |
| **Title** | Consistent session state persistence lifecycle |
| **Tier** | 3 — derived from code exploration |
| **Functional Section** | State Persistence |
| **Specificity** | Specific |

**Summary:** Session state persistence MUST restore all persisted fields on load, or must not persist fields that are intentionally transient. Write-only DB columns waste I/O and create confusing maintenance traps.

**User Story:** As a developer restarting the gateway, I want gradient calibration state to be correctly restored so that sessions don't re-calibrate from scratch, wasting cache budget and causing unnecessary cache busts.

**Implementation Note:** Two related issues:

1. **`saveForceMinLayer` data loss (CB-001):** At `packages/core/src/db.ts:1019–1031`, `saveForceMinLayer(sessionID, 0)` executes `DELETE FROM session_state WHERE session_id = ?`, destroying the entire row. This row also contains cost data (from `saveSessionCosts`), tracking data (from `saveSessionTracking`), gradient calibration EMAs, cache warming state, and LTM cache text — all persisted by other functions to the same row. The function should `UPDATE` the column instead of deleting the row.

2. **Write-only `consecutiveBusts` (CB-002 related):** `saveGradientState()` at `packages/core/src/gradient.ts:673–676` writes `consecutiveBusts` to the `dynamicContextCap` column. But `getSessionState()` at `gradient.ts:329–333` explicitly does NOT restore it: `"Don't restore consecutiveBusts from DB — it's a short-term rolling signal that must rebuild from live API responses."` The column is written every 30s but never read — dead persistence.

**Conditions of Satisfaction:**
1. `saveForceMinLayer(sessionID, 0)` updates the `force_min_layer` column to 0 (or NULL) instead of deleting the entire row.
2. Fields intentionally not restored (like `consecutiveBusts`) are either not persisted at all, or are clearly documented as diagnostic-only columns in the schema.
3. All fields that ARE persisted (cost snapshots, tracking state, LTM cache) survive a `forceMinLayer=0` transition.

**Alternative Paths:**
- Change the DELETE to an UPDATE: `UPDATE session_state SET force_min_layer = 0 WHERE session_id = ?`.
- Remove `consecutiveBusts` persistence entirely (stop writing it to `dynamicContextCap`).

**Use Cases:** UC-02 (long-running lifecycle, process restart)

**References:**
- `packages/core/src/db.ts:1019–1031` — `saveForceMinLayer()` DELETE on layer=0
- `packages/core/src/gradient.ts:665–677` — `saveGradientState()` writing `consecutiveBusts` to `dynamicContextCap`
- `packages/core/src/gradient.ts:325–334` — `getSessionState()` explicitly NOT restoring `consecutiveBusts`

### LTM Scoring Pipeline

#### REQ-012: Pool Deduplication in LTM Scoring

| Field | Value |
|---|---|
| **ID** | REQ-012 |
| **Title** | Pool deduplication in LTM scoring |
| **Tier** | 3 — derived from code exploration |
| **Functional Section** | LTM Scoring Pipeline |
| **Specificity** | Specific |

**Summary:** The LTM scoring pipeline SHOULD deduplicate entries after merging the project and cross-project pools to prevent the same entry from being injected into the system prompt twice.

**User Story:** As an AI agent receiving LTM context, I want each knowledge entry to appear at most once in my system prompt so that I don't waste token budget on duplicates and don't give undue weight to repeated entries.

**Implementation Note:** `forSession()` at `packages/core/src/ltm.ts:415–641` builds two pools:
- Pool 1 (project-specific): `WHERE project_id = ? AND cross_project = 0` at line 440
- Pool 2 (cross-project): `WHERE (project_id IS NULL OR cross_project = 1)` at line 449

For the **preference fast path** (line 462), `allPrefs = [...projectEntries, ...crossEntries]` merges without dedup — but the SQL filters are mutually exclusive (`cross_project = 0` vs `cross_project = 1`), so duplicates are prevented by query structure.

For the **non-preference path** (lines 511–598), both pools go through vector/FTS5 scoring independently, then merge at line 597: `const allScored = [...scoredProject, ...scoredCross]`. Vector search results from `embedding.vectorSearch()` at line 522 search ALL entries — an entry with `project_id = current AND cross_project = 1` matches Pool 2's SQL query AND could appear in vector results for Pool 1's scoring. The `packedIds` Set at line 604 prevents double-packing in the final phase, but duplicates in `allScored` could still affect score-based ordering.

**Conditions of Satisfaction:**
1. After merging `scoredProject` and `scoredCross`, entries are deduplicated by ID before sorting by score.
2. When an entry appears in both pools, the higher score is retained.
3. The `packedIds` guard at line 604 is retained as defense-in-depth.

**Alternative Paths:**
- Deduplicate at the SQL level by using a single query with `UNION` semantics.
- Deduplicate the merged array with a `Map<id, Scored>` before sorting.

**Use Cases:** UC-01, UC-02

**References:**
- `packages/core/src/ltm.ts:436–452` — Pool 1 and Pool 2 SQL queries
- `packages/core/src/ltm.ts:462–478` — preference fast path merge
